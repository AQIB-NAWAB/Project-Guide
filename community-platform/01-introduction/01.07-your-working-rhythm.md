# 1.7 — Your working rhythm

A four-week build is a marathon, not a sprint. A few small habits, kept from day one, will carry you the whole way — and make the difference between a project you finish and one you abandon in week three. You'll see these re-surface lightly throughout the course; here they are up front.

## Commit every day

Make a real commit at the end of each working session — small, focused, with a message that says what changed. Your Git history becomes a diary of the build (great for the final write-up) and a safety net when something breaks. A day's work that isn't committed is a day's work you can lose.

## Keep secrets out of Git, always

From the very first chapter: **no passwords, API keys, tokens, or database URLs in your code or your commits.** They live in environment configuration (you'll set this up in Chapter 2). If a secret ever does slip in, treat it as compromised and rotate it. This habit is non-negotiable in real engineering — start it now.

## Tick the gate before moving on

Every chapter ends in a checklist that **gates** the next one. Don't rush past it. If a box won't tick — the endpoint won't return the right thing, the test won't pass — *that's today's work*, not a thing to skip. The course is built so that skipping a gate quietly breaks something three chapters later.

## Write your learning log honestly

Answer each chapter's questions in your `learning-log/` folder, in your own words. If you find yourself copying phrases from the chapter instead of explaining the idea, that's a signal you don't yet understand it — go back and learn it properly. The log is for *you* first; the viva is just where it pays off.

## Read the error before you ask

When something breaks — and it will, constantly, that's the job — **read the actual error message first.** Then the docs. Then try a smaller version to isolate it. Most problems announce exactly what's wrong if you slow down and read. When you do ask for help, frame it as *"here's what I expected, here's what happened."*

## Build in public (optional, but do it)

Now and then, write up something you just learned — a short post, a thread, a note in your README: *"How I stored a comment tree so it loads in one query."* It cements your own understanding, and it compounds into a reputation. The best engineers are visible ones.
