# 2.4 — Layer vs module

You need a **mandated decision** before creating folders. This sub-chapter compares the two mainstream Node/Express layouts honestly, then tells you which one this course requires — and why changing your mind later breaks the guide.

## Why this matters for absolute beginners

"Layer vs module" sounds like senior-engineer vocabulary. The practical difference is simple: when you fix a **cart bug**, do you open **one folder** or hunt through `controllers/`, `services/`, and `models/` hoping names match? Marketplaces have many product areas — auth, shops, products, cart, checkout. Grouping by **domain** keeps related files neighbors. The course mandates modules so every sub-chapter can say "open `modules/cart/`" and you know exactly where to go.

## Think of it like…

**Organizing a hardware store by project vs by tool type.** Layer-based: all hammers in Aisle 1, all screws in Aisle 2 — fine for three SKUs. Module-based: "Kitchen renovation" kit with hammer, screws, and tape together — fine when customers arrive with a **project** (checkout, vendor dashboard) not a tool name.

## Common beginner questions

**Q: My tutorial used `controllers/` — can I keep it?**  
A: Not if you want chapter paths to match. Translate once now to modules, not twenty times later.

**Q: What goes in `shared/`?**  
A: Code every module needs — config, DB connection, shared error helpers — not business logic for one feature.

**Q: Should route handlers contain all the logic?**  
A: Start thin in Chapter 2; Chapter 6+ grows **service** files inside each module. Routes parse HTTP; services talk to the database.

## Layer-based structure

Everything of the same **technical kind** lives together:

```
server/src/
  controllers/
    authController.js
    productController.js
    orderController.js
  services/
    authService.js
    productService.js
    orderService.js
  models/
    User.js
    Product.js
    Order.js
  routes/
    authRoutes.js
    productRoutes.js
  middleware/
    authMiddleware.js
```

**How you add "vendor store setup":**

1. Add methods to `storeController.js` (or create `storeController.js` in controllers/)  
2. Add logic to `storeService.js`  
3. Add `Store.js` model  
4. Wire routes in `storeRoutes.js`  
5. Import middleware from `middleware/`  

One feature → **four or five folders touched**. Each folder accumulates unrelated domains. `controllers/` becomes a junk drawer.

---

## Module-based structure

Everything about one **domain** lives together:

```
server/src/
  modules/
    auth/
      auth.routes.js
      auth.service.js
      auth.middleware.js
      auth.schema.js       ← validation shapes (later)
    products/
      products.routes.js
      products.service.js
      products.model.js
    orders/
      ...
  shared/
    config.js
    db.js
    errors.js
```

**How you add "vendor store setup":**

1. Create `modules/stores/`  
2. Add routes, service, model **inside that folder**  
3. Mount `stores.routes.js` from `app.js`  

One feature → **one new folder**. Delete the feature → delete the folder.

---

## Comparison table

| | Layer-based | Module-based |
|---|---|---|
| **Pros** | Familiar from tutorials; trivial for 3-route apps; "all controllers" in one grep | High cohesion — cart code lives with cart; scales by adding folders; easy to find/delete whole features |
| **Cons** | Features scattered; folders grow without bound; merge conflicts cluster; hard to navigate at 40+ routes | Slightly more setup upfront; less familiar if you only know layer tutorials |
| **Adding a feature** | Touch many directories | Add one module directory |
| **Best for** | Tiny APIs, homework-scale apps | Multi-domain apps — **marketplaces, SaaS, platforms** |
| **This course** | ❌ Not aligned with chapter paths | ✅ **Required** |

## The "adding discount codes" test

Imagine Week 5 adds discount codes. Compare mental load:

**Layer-based:** Open `controllers/`, find or create discount controller. Open `services/`. Open `models/`. Open `routes/`. Hope naming is consistent. Run tests — which tests? They're organized by layer too.

**Module-based:** Open `modules/discounts/`. Everything is there. Ship.

The marketplace has **auth, stores, products, cart, orders, catalogue, uploads, cache** — at minimum eight domains. Module-based isn't overkill; layer-based is under-structure.

## How modules share code

Modules shouldn't import each other's internals chaotically. Shared concerns live in `shared/`:

| `shared/` file | Holds |
|---|---|
| `config.js` | Validated env vars (Chapter 4) |
| `db.js` | Mongoose connection (Chapter 4) |
| `errors.js` | AppError class, error formatter (later) |

Modules import from `shared/`. Modules call other modules through **service interfaces** or HTTP — not by reaching into each other's private files. You'll refine this as the app grows; for now, empty `shared/` is enough.

## Client-side parallel

React apps face the same fork:

| Layer-ish | Feature-ish |
|---|---|
| `components/` + `services/` + `utils/` only | `pages/` per screen + `api/` + `hooks/` per domain |

This course uses:

```
client/src/
  pages/         ← Login.jsx, Shop.jsx, VendorDashboard.jsx
  components/    ← reusable UI: ProductCard, CartLine
  api/           ← client.js + optional per-domain API functions
  hooks/         ← useAuth, useCart later
```

Pages are **product-shaped** ("Shop the marketplace"), not `DashboardPageComponent.jsx`.

## Your choice for this course (mandatory)

**Use module-based layout on the server.** Required — not a preference.

Reasons specific to **this guide**:

- Chapter 6: *"scaffold `modules/auth/`"*  
- Chapter 11: *"products.routes.js inside modules/products/"*  
- Chapter 18: *"orders service groups cart by storeId"*  

Layer-based repos still work in production — but **not with this course's paths**. Pick module-based now; adapt file extensions (`.js` vs `.ts`) if you later migrate to TypeScript, but keep the **module boundaries**.

## File naming inside a module

Convention for this course (adjust slightly if your style differs — stay consistent):

| File | Responsibility |
|---|---|
| `*.routes.js` | Express router: paths, HTTP verbs, call service |
| `*.service.js` | Business logic; database calls |
| `*.model.js` | Mongoose schema (Chapter 5+) |
| `*.middleware.js` | Route-specific middleware |
| `*.schema.js` | Request body validation shapes (later) |

Routes stay thin — parse request, call service, send response. Fat services, thin routes.

## ✅ Do / ❌ Don't

**✅ Do:** Create empty module folders for `auth`, `stores`, `products`, `cart`, `orders` in Chapter 2 — placeholders remind you where code lands.

**✅ Do:** Mount future routers in `app.js` with one line per module: `app.use('/api/auth', authRoutes)`.

**❌ Don't:** Put business logic directly in route handlers long-term — Chapter 6 will start small, but refactor toward service files as modules grow.

**❌ Don't:** Copy a layer-based tutorial repo and fight the course for twenty chapters.
