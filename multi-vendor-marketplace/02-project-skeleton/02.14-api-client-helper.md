# 2.14 — API client helper

**Why this step matters:** Scattering `fetch('http://localhost:5000/...')` across components breaks the moment you deploy or change ports. One helper reading `VITE_API_URL` is how every professional SPA keeps API calls **configurable and consistent**.

**Where the project is now:** Home page renders; server health works via curl. Client has no configured API base URL yet.

**After this step:** `client/src/api/client.js` exports helpers to call your Express API using `import.meta.env.VITE_API_URL`.

---

## Step 1 — Environment file

Create `client/.env`:

```env
VITE_API_URL=http://localhost:5000
```

**No trailing slash** on the URL — your helper should normalize paths consistently.

Create `client/.env.example` (safe to commit):

```env
VITE_API_URL=http://localhost:5000
```

**✅ Do:** Commit `.env.example`. **❌ Don't:** Commit `.env` if it ever holds secrets — here it's public URL only, but habit matters.

Restart Vite dev server after changing `.env` — Vite reads env at startup.

---

## Step 2 — Create `api/client.js`

**Requirements — implement yourself:**

1. Read base URL from `import.meta.env.VITE_API_URL`
2. If missing, throw clear error in development or log warning — fail fast rather than silent `undefined/health`
3. Export at minimum:
   - **`getHealth()`** — performs GET to `{baseUrl}/health`, parses JSON, returns parsed object
   - Or export generic **`apiGet(path)`** where path is `/health`

**Behaviour contract for `getHealth()`:**

| Case | Result |
|---|---|
| Server returns 200 + JSON | Return parsed object to caller |
| Network failure | Throw or return error object caller can display — your choice, document in comment |
| Non-200 status | Treat as error — Home page should show failure message in 2.15 |

**Do not** hardcode `http://localhost:5000` inside components — only in `.env`.

## Optional generic helper shape

Many projects export:

```javascript
// Pseudocode — YOU implement
async function apiGet(path) {
  // fetch VITE_API_URL + path
  // if !response.ok throw
  // return response.json()
}
```

Centralizing fetch here means Chapter 7 adds auth headers in **one file**, not twenty components.

---

## Step 3 — Verify env loads

Temporarily log in `client.js` or Home (remove after test):

```javascript
console.log('API URL', import.meta.env.VITE_API_URL)
```

Browser console should show `http://localhost:5000`. If `undefined`, check file name `.env`, prefix `VITE_`, and restart Vite.

---

## Step 4 — Commit

```bash
git add client/.env.example client/src/api/
git commit -m "feat(client): add api client helper and env example"
```

Do not commit `client/.env` if your ignore rules exclude it — good.

## ✅ Do / ❌ Don't

**✅ Do:** Use `fetch` API (built into browsers) unless you have reason for axios — either works.

**✅ Do:** Set `Content-Type` on future POST requests in this helper (Chapter 6+).

**❌ Don't:** Put JWT in `.env` on client.

**❌ Don't:** Scatter `fetch('http://localhost:5000/...')` across components — defeats purpose of helper.
