# 2.2 — Why structure matters

Folder structure feels like housekeeping. It isn't. It's **architecture written in paths** — the decision about where auth lives, where product routes live, and where shared database code lives. Get it wrong on day one and you'll either refactor under pressure mid-course or mentally translate every chapter instruction for twenty chapters.

## Why this matters for absolute beginners

When every tutorial puts code in one file, it is easy to believe structure is something seniors worry about later. It is not. Structure is **how you find things** when something breaks at 11 p.m. before a demo. This course names exact paths in every build step — `modules/products/products.routes.js`, `modules/auth/auth.service.js`. If your repo uses a flat `routes.js` instead, you are not "just organizing differently"; you are fighting the guide on every chapter. One afternoon choosing module folders saves weeks of "where did I put cart logic?"

## Think of it like…

**Room labels in a shared kitchen.** A home kitchen can be one drawer. A restaurant kitchen labels stations — prep, grill, pastry — so anyone can grab the right tool without opening every cabinet. Your marketplace has distinct stations: auth, stores, products, cart, orders. Module folders are those labels written on disk.

## Common beginner questions

**Q: Can't I reorganize later when the app grows?**  
A: You can — teams do — but it costs days of moving files, fixing imports, and re-testing. This chapter spends one day so the next eighteen chapters stay aligned.

**Q: Is module-based the only valid architecture?**  
A: No — layer-based works for tiny APIs. Multi-domain marketplaces outgrow layer folders quickly; this course mandates modules for **shared path language**, not because layers are "wrong."

**Q: Does folder structure affect users?**  
A: Not directly — users never see your paths. It affects **you**: speed of fixes, merge conflicts, and whether you can explain your repo in a viva.

## The two tempting shortcuts

### Shortcut 1 — No structure

Everything in `index.js`. Or a flat `src/` with `routes.js`, `db.js`, and `helpers.js` growing without bound.

**Day one:** Fast. One file, one mental model.  
**Day ten:** You have vendor routes, product routes, cart routes, auth middleware, and upload logic in the same 800-line file — or scattered with no rule.

**Day twenty:** Chapter 11 says *"add ownership check in the products module service"* and you don't have a products module. You have a function somewhere near line 400.

### Shortcut 2 — Random starter template

Clone "ultimate-node-boilerplate" from the internet. Fifteen folders you didn't design. `infrastructure/`, `domain/`, `adapters/` — maybe good for that author's job, opaque for you.

**The cost:** You can't explain why `VendorController` lives where it does. When the course says *"create `server/src/modules/stores/stores.routes.js`"*, your repo doesn't have `modules/` and you improvise — two mentees on the same course now have incompatible repos.

## What good structure optimizes for

| Goal | What structure should do |
|---|---|
| **Find code fast** | "Product checkout bug" → open `modules/orders/` |
| **Change one feature safely** | Edit cart without touching auth files |
| **Onboard future you** | Return after a week away; paths tell the story |
| **Match the course** | Every sub-chapter names a path; you create that path |

This marketplace has distinct **domains**: users & auth, stores, products, cart, orders, catalogue reads, cache. Structure should mirror those domains — not mirror technical layers alone.

## Layer-based vs module-based — preview

Two mainstream philosophies:

**Layer-based** — group by technical role:
```
src/
  controllers/    # every HTTP handler
  services/       # every piece of business logic
  models/         # every database schema
```

Adding "discount codes" touches three folders. Each folder grows forever. Related code **drifts apart**.

**Module-based (feature-based)** — group by domain:
```
src/
  modules/
    auth/
    stores/
    products/
    cart/
    orders/
  shared/           # config, db, errors — used by all modules
```

Adding "discount codes" adds one folder. Everything about discounts **stays together**. The app scales by **adding modules**, not by inflating shared layer folders.

The next sub-chapter compares them formally and **mandates one for this course**. For now: a marketplace with many domains is the textbook case for module-based layout.

## Why the course mandates a choice

Foundational choices can't stay open. If Chapter 6 says *"create `modules/auth/auth.routes.js`"* and you chose layer-based, either:

- You translate every path for twenty chapters, or  
- You diverge from the guide and reviewers can't help you.

Same for the client: `client/src/pages/`, `client/src/api/` — consistent paths so Chapter 7's login page lands in a predictable place.

**This course requires module-based server layout and the client folder convention introduced in Chapter 2.** Not because it's the only valid architecture in the world — because **shared path language** is how a guided build stays guided.

---

## What happens if you change structure later

You *can* refactor layer → module mid-project. Teams do it when apps outgrow early choices. Cost:

- Move dozens of files  
- Fix every import path  
- Re-test every route  
- Lose days that could ship features  

Cheaper to spend **one chapter** now and commit.

---

## Client structure matters too

Backend-only for three weeks produces backend developers who've never seen their marketplace in a browser. This course pairs API and UI chapters — starting **here** with both `server/` and `client/` in the same repo.

```
your-marketplace/
  server/          ← Node + Express API
  client/          ← Vite + React UI
  learning-log/
  docs/
  README.md
```

Two `package.json` files — two apps, one product. The monorepo pattern (one Git repo, multiple packages) matches how many startups ship MERN.

## Signs you picked wrong structure (future you)

| Symptom | Likely cause |
|---|---|
| "Where did I put cart logic?" | No `cart` module |
| Merge conflicts every day in `controllers/` | Layer folders are shared choke points |
| Import paths like `../../../` everywhere | Flat or chaotic layout |
| Course path doesn't exist | Diverged from module convention |

Fix early — this chapter — not at Chapter 18 checkout.

## Historical note

Early PHP marketplaces often started as single `index.php` files. Node ecosystems repeated the pattern with one `index.js` until package size forced splits. Module folders are the Node-era equivalent of **bounded contexts** in domain-driven design — you don't need DDD vocabulary for the course, but the parallel explains why senior engineers insist on structure early.
