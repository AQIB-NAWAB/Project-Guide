# 2.11 — The health route

**Why this step matters:** `GET /health` is your first **proof the API works** — independent of React, MongoDB, or auth. Load balancers and Chapter 4's extended health both build on this route; Chapter 2's gate requires curl to return honest JSON.

**Where the project is now:** Express boots and listens. No defined routes yet — or only a catch-all 404.

**After this step:** `GET /health` returns **200** with JSON `{ "status": "ok" }` (minimum). Verified with curl.

---

## The contract

| Field | Value |
|---|---|
| Method | `GET` |
| Path | `/health` |
| Authentication | None — public |
| Success status | `200 OK` |
| Content-Type | `application/json` |
| Body (minimum) | `{ "status": "ok" }` |

**Optional fields** (your choice):

```json
{
  "status": "ok",
  "service": "GreenBasket API",
  "uptime": 123.45
}
```

Chapter 4 extends health with database status — don't add `db: connected` until MongoDB exists.

---

## What you must implement

1. Register a **GET** handler for `/health` on the Express app in `app.js` (or a tiny `shared/health.routes.js` you mount — either is fine at this scale).

2. Handler behaviour:
   - Set status code **200**
   - Respond with JSON object containing **`status: "ok"`** string
   - Do not throw for normal operation

3. **You write the handler body.** The course specifies behaviour, not implementation.

---

## Suggested structure (not the solution)

Many teams mount health at app root:

```
app.get('/health', handler)
```

Others prefix API routes:

```
app.use('/api', apiRouter)
// health at /api/health
```

**This course uses `/health` at root** unless you document a different path consistently everywhere (client env in 2.14 must match).

If you choose `/api/health`, update `VITE_API_URL` usage in 2.15 to fetch `/api/health`.

---

## Wire CORS (optional here, required by 2.15)

If `cors` is installed:

- Import `cors` package
- `app.use(cors({ origin: 'http://localhost:5173' }))` — adjust when client port differs

Browser demo in 2.15 fails without CORS. Installing here saves a debug step later.

## Verify with curl

Server running (`npm run dev` in `server/`):

```bash
curl -i http://localhost:5000/health
```

**Expected:**

```
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
...

{"status":"ok"}
```

Also test wrong path:

```bash
curl -i http://localhost:5000/does-not-exist
```

**Expected:** `404` — proves routing works selectively.

---

## Verify with browser

Open `http://localhost:5000/health` in browser address bar.

**Expected:** JSON visible or download — not HTML error page from connection refused.

## ✅ Do / ❌ Don't

**✅ Do:** Return JSON via `res.json()` or equivalent — not plain text `ok`.

**✅ Do:** Use GET only — health checks should not mutate state.

**❌ Don't:** Put authentication on `/health` — load balancers and Chapter 2 client demo need public access.

**❌ Don't:** Return `200` with `{ "status": "error" }` body when DB is down — Chapter 4 will define DB-aware health properly.

---

## Hint

If curl returns connection refused, server isn't listening — fix 2.10 first. If curl returns 404, route not mounted on correct path or typo in URL.
