# 2.5 — Client vs server

Full-stack beginners often mix **what runs in the browser** with **what runs on your machine in Node**. This sub-chapter draws the line once, using **your** `client/` and `server/` folders — not a generic diagram you'll forget.

## Why this matters for absolute beginners

If you think React "talks to MongoDB," you will put connection strings in the client — a serious security mistake. The browser runs **untrusted code** anyone can inspect in DevTools. The server runs **trusted code** on your machine (later, a host) and holds secrets. Every marketplace feature follows the same loop: click in React → HTTP request → Express → database → JSON response → React updates the screen. Memorize that loop now; every chapter extends it with new routes, not new architecture.

## Think of it like…

**A restaurant dining room vs the kitchen.** Customers (browser) see the menu and place orders — they never walk into the walk-in freezer. The kitchen (server) holds ingredients, recipes, and keys to suppliers (MongoDB, Cloudinary). Waiters carry plates through a pass (HTTP/JSON). You would not put the freezer in the dining room; you do not put `MONGODB_URI` in React.

## Common beginner questions

**Q: Why two ports (5173 and 5000)?**  
A: Vite and Express are separate programs in development. Same machine, different doors — the browser loads UI from one, API calls go to the other.

**Q: What is CORS and why do I need it?**  
A: Browsers block JavaScript on one origin from calling another unless the server **allows** it. Different ports count as different origins — hence `cors` middleware on Express.

**Q: Can I put API logic inside React `useEffect`?**  
A: You can **call** the API from React; you must not **implement** business rules or database access there. All persistence stays on the server.

**Q: Is `VITE_API_URL` a secret?**  
A: No — the browser needs to know where the API lives. Secrets (`JWT_SECRET`, `MONGODB_URI`) stay in `server/.env` only.

## Two processes, two terminals

After Chapter 2 you'll run **two commands in two terminals**:

| Terminal | Directory | Command | Serves |
|---|---|---|---|
| 1 | `server/` | `npm run dev` | Express API on port **5000** (default) |
| 2 | `client/` | `npm run dev` | Vite + React on port **5173** (default) |

Two processes. Two ports. **One product.**

Closing one terminal doesn't stop the other — which is why half the bugs this chapter teaches you to debug are "client works, API down" or the reverse.

## HTTP — the contract between them

The browser **does not** call Mongoose. React **does not** open MongoDB. Every persistence operation flows:

```
React component
  → fetch POST /api/products  (HTTP + JSON)
    → Express route
      → service
        → Mongoose
          → MongoDB
```

Response travels back up the same chain.

**REST** in this course means: resources identified by URLs, verbs mean something (`GET` read, `POST` create, `PATCH` update, `DELETE` remove), status codes tell the story (`200` OK, `201` created, `400` bad input, `401` not logged in, `403` not allowed, `404` not found).

Chapter 2's only API endpoint is **`GET /health`** — no database, no auth. Still HTTP. Still JSON.

## Why secrets stay on the server

Anything bundled into the React app is **visible to every user**. They can open DevTools → Sources and read minified JS. Therefore:

| Secret | Where it lives |
|---|---|
| `MONGODB_URI` | `server/.env` only |
| `JWT_SECRET` | `server/.env` only |
| `CLOUDINARY_API_SECRET` | `server/.env` only |
| Public API base URL | `client/.env` as `VITE_API_URL` — not secret |

Vite only exposes env vars prefixed with `VITE_` to the client bundle — and you still only put **non-secret** values there.

**❌ Never:** paste Mongo connection string into `client/src/api/client.js`.

---

## CORS — why the browser blocks you (until you fix it)

By default, browsers restrict JavaScript on `http://localhost:5173` from calling `http://localhost:5000` — **different ports = different origins**.

You'll install the `cors` package on the server and allow the client origin. That's infrastructure setup (Chapter 2 build step), not business logic.

Symptom if you skip CORS: fetch fails in browser with a CORS error; same URL works fine in curl.

## `app.js` vs `server.js` — split on the server

| File | Job |
|---|---|
| `app.js` | Create Express app; register middleware; mount routes; **export** app |
| `server.js` | Import app; read `PORT`; call `app.listen()` |

**Why split?** Later you'll import `app` in tests and send fake HTTP requests **without** opening a real port. Chapter 30-style isolation tests benefit from this separation. Cost now: one extra file. Benefit later: no refactor.

## What the Home page proves in 2.15

When Home fetches `/health` and displays the result:

- React `useEffect` + fetch works  
- `VITE_API_URL` is correct  
- Server is up and CORS allows the browser  
- JSON parsing in UI works  

That's the **minimum full-stack loop** — everything else is more routes and more pages on the same pattern.

## Common beginner mistakes

| Mistake | Symptom |
|---|---|
| Only running client | API calls fail; or you hardcode fake data |
| Wrong `VITE_API_URL` | Network error, connection refused |
| Forgot CORS | Browser console CORS policy error |
| Put API code in React | Can't enforce isolation; secrets leak |
| One terminal, one port | Can't serve Vite and Express on same port without proxy (we use two ports + CORS for clarity) |

Optional advanced: Vite `proxy` in `vite.config.js` instead of CORS — this course uses explicit CORS to teach cross-origin reality.

## ✅ Do / ❌ Don't

**✅ Do:** Keep all MongoDB and JWT logic in `server/src/modules/` or `shared/`.

**✅ Do:** Put every `fetch` to your API through `api/client.js` (or helpers built on it).

**❌ Don't:** Call Cloudinary upload secret from React — server uploads in Chapter 13.

**❌ Don't:** Store refresh tokens in `localStorage` — Chapter 7 covers safer patterns.

## Key ideas

- **client/** = browser; **server/** = Node API  
- They talk over **HTTP + JSON**  
- **Secrets** server-only; **VITE_** vars client-safe only  
- **CORS** required for localhost cross-port  
- **app.js** builds; **server.js** listens
