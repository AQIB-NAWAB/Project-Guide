# 11.5 — Prime your thinking

Vendor product list may stay unpopulated in Chapter 11 — but **populate** appears in Chapter 14 and cart. Read now so your `storeId` refs stay consistent. This is the last learn step before the product API build cluster — the mental model you set here determines whether Chapter 14's public catalogue leaks drafts or triggers N+1 query storms.

---

## Two audiences, one collection

The same `products` collection serves two different HTTP routes with two different rules:

| Route | Who calls it | What they see |
|---|---|---|
| `GET /vendor/products` | Logged-in vendor | **All** their products — drafts and published |
| `GET /products` (Ch 14) | Anyone | **Published only** — filtered, paginated |

Building Chapter 11 with this split in mind prevents "fix it later" refactors. Your `isPublished` field is not decorative — it is the gate between back office and shop window.

## Ownership mental model

Before coding service:

```
createProduct(vendorId, data):
  store = getStoreByVendorId(vendorId)
  if !store → error
  insert { ...data, storeId: store._id }

listVendorProducts(vendorId):
  store = getStoreByVendorId(vendorId)
  return Product.find({ storeId: store._id })

assertProductOwner(productId, vendorId):
  product = findById(productId)
  store = findStore(product.storeId)
  if store.vendorId !== vendorId → forbidden
```

---

## Think of it like a warehouse with two doors

The **stockroom door** (vendor routes) shows everything on the shelves — including items not yet priced for display. The **shop window** (public catalogue, Ch 14) shows only what's tagged "for sale." Same warehouse, different doors, different filters. Your service layer should know which door the request came through.
