# 11.10 — List vendor products

## Why this step matters

The vendor back office needs inventory visibility — drafts and published together. `GET /vendor/products` filters by the caller's store only. Chapter 12's product table calls this endpoint; Chapter 14's public catalogue is a different route with different filters.

---

**After this step:** `GET /api/vendor/products` returns **200** array of products for authenticated vendor's store only — includes drafts and published.

---

## Handler flow

```
GET /api/vendor/products
  1. requireAuth
  2. requireRole('vendor')
  3. listVendorProducts(userId)
  4. 200 + { products: [...] } or raw array — pick one, document
```

Consistent JSON shape helps Chapter 12 UI — e.g. `{ products: [], total: N }`.

---

## Service: listVendorProducts

| Step | Action |
|---|---|
| 1 | Resolve vendor's store |
| 2 | If !store → 404 or empty array `[]` — recommend **200 + []** if no store vs **404** — pick one; **200 []** avoids error for edge case; **404** matches no store semantics. Course: **200 with empty array** when no store OR **404 no store** — recommend **404** if no store (consistent with POST product) |
| 3 | `Product.find({ storeId })` sort `createdAt` desc |

Document choice in README.

---

## Scoping guarantee

Vendor A list must **never** include Vendor B products — test with seed data from two stores.

Query uses storeId from JWT chain only — never query param `storeId` from client.

---

## Optional query params (stretch)

| Param | Filter |
|---|---|
| `?published=true` | only published |
| `?published=false` | drafts only |

Not required for gate.

---

## Response fields

Return full documents for vendor dashboard — name, price, unit, stockQty, isPublished, `_id`, `storeId`.

No need to populate store name — vendor knows their shop.
