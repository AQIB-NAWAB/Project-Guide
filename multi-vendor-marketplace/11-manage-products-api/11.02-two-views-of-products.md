# 11.2 — Two views of products

The same Product document serves **two audiences** with different routes, filters, and trust boundaries. Confusing them causes draft products on the public shop or vendor-only prices leaking — Chapter 11 builds only the **vendor inventory view**.

---

## Vendor inventory view

| Property | Value |
|---|---|
| **Who** | Authenticated vendor |
| **Purpose** | Manage stock, price, publish state |
| **Sees** | All own products — draft and published |
| **Routes** | `POST /products`, `GET /vendor/products`, `PATCH/DELETE /products/:id` |
| **Scope** | `storeId` = vendor's store from JWT chain |

Vendor needs **full lifecycle** — including unpublished drafts and zero-stock items they are restocking.

---

## Public catalogue view

| Property | Value |
|---|---|
| **Who** | Anyone (customer or anonymous) |
| **Purpose** | Browse and buy |
| **Sees** | Published + in-stock only (Ch 14 rules) |
| **Routes** | `GET /products`, `GET /products/:id` |
| **Scope** | Platform-wide with filters |

Built in **Chapter 14** — not this chapter.

---

## Side-by-side

| Concern | Vendor (Ch 11) | Public (Ch 14) |
|---|---|---|
| Auth | Required | Optional / none for read |
| Draft products | Visible | Hidden |
| Other vendors' products | Never | Visible if published |
| `storeId` in response | Own only | All stores — populate name |
| DELETE | Yes | No public delete |

---

## Shared collection, different queries

```
Vendor list query:
  { storeId: vendorStoreId }

Public list query (Ch 14):
  { isPublished: true, stockQty: { $gt: 0 } }
```

Same `products` collection — **query filter** enforces view, not duplicate tables.

---

## `isPublished` or status field

| State | Vendor list | Public catalogue |
|---|---|---|
| draft / unpublished | ✓ shown | ✗ hidden |
| published + stock | ✓ | ✓ |
| published + zero stock | ✓ | ✗ hide or show out-of-stock — Ch 14 decides |

Chapter 11 **sets** publish field via create/PATCH; Chapter 12 toggles in UI; Chapter 14 filters.

---

## Why separate route prefixes

| Pattern | Example | Benefit |
|---|---|---|
| Vendor-scoped list | `GET /vendor/products` | Obvious auth + scope in AUTHORIZATION.md |
| Public list | `GET /products` | Clean customer API |

**❌ Don't:** Use `GET /products?vendorId=me` — client tampering risk. Vendor list derives scope from JWT.

---

## Grocery fields reminder

Vendor inventory includes domain fields from Ch 5:

- `unit` (kg, each, bunch)  
- `weightOrQuantity`  
- `isPerishable`  
- `stockQty`  
- `price`  

Vendor API returns all — public catalogue may shape response in Ch 14.

---

## ✅ Do / ❌ Don't

**✅ Do:** Name routes clearly — `/vendor/products` vs `/products`.

**✅ Do:** Default new products to **unpublished** if your model supports it — safer MVP.

**❌ Don't:** Implement public `GET /products` in Chapter 11 — scope creep.

**❌ Don't:** Return other vendors' products on vendor list — ever.
