# 4.3 — What you'll build

By the end of Chapter 4, your marketplace API **connects to MongoDB on startup**, reads configuration from environment variables with validation, and reports database status on `/health`. Your React Home page still fetches health — now showing whether the database layer is alive.

This sub-chapter is the **blueprint** for sub-chapters 4.8–4.15. No code solutions here — contracts and file locations only.

## Why this matters for absolute beginners

This blueprint tells you **what "done" looks like** before you write `config.js`. Without it, it is easy to connect MongoDB inline in `server.js` and forget health reporting, fail-fast validation, or `.env.example`. Each row in the deliverables table maps to one build sub-chapter — finish the list, pass the gate.

## Common beginner questions

**Q: Do I create Mongoose models in this chapter?**  
A: No — connection and config only. Models are Chapter 5.

**Q: What should health return when DB is down?**  
A: Pick honest behaviour (503 or `db: disconnected`) and document it — never claim connected when MongoDB is unreachable.

**Q: Why connect before `listen`?**  
A: So the API does not accept traffic pretending to be healthy while the database is unavailable.

**Q: Does the client need new env vars for MongoDB?**  
A: No — browser still only uses `VITE_API_URL`. Home page displays the **db** field from API JSON.

## Boot order contract

Professional APIs connect dependencies **before** accepting traffic:

```
1. Load config (validate env — fail if MONGODB_URI missing)
2. Connect MongoDB (await mongoose connection)
3. Start HTTP server (app.listen)
4. On shutdown signal — disconnect MongoDB gracefully (stretch in 4.11)
```

**Why order matters:**

- If you `listen` first, health checks may report `status: ok` while DB is down — lying to load balancers.  
- If config validation runs after connect attempt, error messages are vague ("connection failed" vs "MONGODB_URI undefined").  

Chapter 4 gate expects **fail-fast on missing config** and **connected DB before demo**.

## Environment variables — server

Minimum required for this chapter:

| Variable | Required | Example placeholder in `.env.example` |
|---|---|---|
| `MONGODB_URI` | yes | `mongodb://127.0.0.1:27017/your-app-db` |
| `PORT` | no (default 5000) | `5000` |
| `NODE_ENV` | no | `development` |

Chapter 6 adds `JWT_SECRET`, etc. — leave them out of `.env.example` until needed, or add commented placeholders:

```env
# JWT_SECRET=generate-in-chapter-6
```

**Config module behaviour (you implement):**

- Read `process.env`  
- If `MONGODB_URI` missing or empty string → **throw** or **process.exit(1)** with clear message  
- Export a plain object, e.g. `{ port, mongoUri, nodeEnv }` — naming your choice, shape stable for imports

## `shared/db.js` contract

Export at minimum:

| Export | Behaviour |
|---|---|
| `connectDB()` | Async; calls Mongoose connect with URI from config; resolves when connected; throws/logs on failure |
| `disconnectDB()` | Async; closes connection — used on graceful shutdown (optional gate) |

**Do not** call `mongoose.connect()` directly from `server.js` scattered inline — centralize in `db.js` so Chapter 5 seed script and tests reuse the same helper.

## Extended health contract

**Minimum from Chapter 2:**

```json
{ "status": "ok" }
```

**After Chapter 4 (when DB connected):**

```json
{
  "status": "ok",
  "db": "connected"
}
```

| Scenario | HTTP status | Body guidance |
|---|---|---|
| API up, MongoDB connected | `200` | `"status": "ok"`, `"db": "connected"` |
| API up, MongoDB not connected | Course choice: **`503`** or `200` with `"db": "disconnected"` — **pick one, document in README** | Must not claim `"db": "connected"` when false |

**Recommended for learning:** If server only starts after successful connect (boot order above), runtime health usually shows connected. Sub-chapter 4.15 tests **stop MongoDB while server runs** — then health or process behaviour must reflect failure honestly.

**You implement** how health reads connection state — `mongoose.connection.readyState`, cached flag set at connect, or ping command. Spec: response must reflect reality.

## Verification story (4.15)

You will demonstrate three states:

1. **Happy path:** MongoDB running → server starts → curl `/health` shows `db: connected` → Home shows same  
2. **Missing config:** Rename or remove `MONGODB_URI` → server refuses to start with **clear error**  
3. **DB down:** Server running → stop MongoDB → health or logs show failure — not silent success  

Screenshot or learning log notes for each — mentor may ask.

## ✅ Do / ❌ Don't

**✅ Do:** One config module — avoid `process.env.MONGODB_URI` in ten files.

**✅ Do:** Commit `.env.example` every time you add a required server env var.

**❌ Don't:** Create Mongoose schemas in this chapter — connection only.

**❌ Don't:** Return connection string in health JSON — ever.
