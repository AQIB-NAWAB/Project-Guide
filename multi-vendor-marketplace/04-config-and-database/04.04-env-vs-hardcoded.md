# 4.4 — Environment variables vs hardcoded

Every deployment target — your laptop, a mentor's machine, Render, Railway, Atlas — needs **different** connection strings and ports. Hardcoding those values in `server.js` works until the first time someone else clones your repo or you deploy to production.

This sub-chapter explains **12-factor config** as it applies to your FreshMarket server — not enterprise theory, but the rules you'll follow through Chapter 22.

## Why this matters for absolute beginners

Hardcoding feels faster on day one — paste the URI, move on. Day thirty, your mentor clones the repo and nothing runs; deploy uses port 8080 while your app listens on 5000; a connection string sits in Git forever. **Configuration is not code** — it is input to code. One `config.js` reader keeps env vars discoverable; scattered `process.env` everywhere does not.

## Think of it like…

**A recipe vs the ingredients in your fridge.** The recipe (code) is the same everywhere; ingredients (env vars) change per kitchen — your laptop, staging, production. You would not bake the shop's address into the recipe; you would not bake MongoDB passwords into `server.js`.

## Common beginner questions

**Q: Why can't `MONGODB_URI` default to localhost in code?**  
A: Silent wrong-database bugs — production might think it connected while pointing nowhere useful. Missing URI should **fail loudly** at startup.

**Q: dotenv vs Node `--env-file` — which wins?**  
A: Either — course accepts both if documented in README. Env must load **before** config validation.

**Q: Why prefix client vars with `VITE_`?**  
A: Vite only exposes those to the browser bundle — a safety rail. Still never use `VITE_` for secrets.

## The hardcoded anti-pattern

```javascript
// ❌ Do not ship this pattern
mongoose.connect('mongodb://127.0.0.1:27017/greenbasket')
app.listen(5000)
```

**Problems:**

| Problem | When it hurts |
|---|---|
| Wrong DB on deploy | Production API writes to localhost string that doesn't exist on host |
| Secret in Git | Connection string in source is committed forever |
| Port conflict | Host sets `PORT=8080`; app still listens on 5000 |
| Teammate clone | Their MongoDB runs on different URI — they edit your source |

Configuration is **environment-specific data**, not program logic.

## 12-factor App — Config (Factor III) in plain language

> *Store config in the environment.*

For your course project that means:

1. **Code** reads `process.env.VAR_NAME` (via your `config.js` wrapper).  
2. **Environment** supplies values — `.env` file locally, host dashboard in production.  
3. **Same codebase** runs everywhere; only env differs.  

Not in scope for Factor III: checking config into JSON in the repo (except non-secret defaults in `.env.example`).

## Layers of configuration in your repo

```
┌─────────────────────────────────────────┐
│  Host / shell / platform                │
│  sets process.env (production)          │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│  server/.env (local dev only)           │
│  loaded by dotenv or --env-file         │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│  shared/config.js                       │
│  validates + exports typed config obj   │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│  db.js, server.js, modules…             │
│  import config — never raw env scatter  │
└─────────────────────────────────────────┘
```

**Single reader:** `config.js` is the only file that should list which env vars exist and which are required. Other files import `{ mongoUri, port }`.

## `.env` vs `.env.example`

| File | Committed? | Contents |
|---|---|---|
| `.env` | **Never** | Real secrets and local overrides |
| `.env.example` | **Always** | Key names + fake/localhost placeholders + comments |

When you add `MONGODB_URI` in 4.9, `.env.example` might contain:

```env
# MongoDB connection string — copy to .env and set your real URI
MONGODB_URI=mongodb://127.0.0.1:27017/freshmarket-dev

# HTTP port (optional — defaults to 5000 in config)
PORT=5000
```

**No real Atlas password** in the example file.

## Loading `.env` in Node — pick an approach

The course accepts any of these **if documented in README**:

| Approach | Notes |
|---|---|
| **`dotenv` package** | `import 'dotenv/config'` at top of `server.js` — widely used |
| **Node `--env-file`** | `node --env-file=.env src/server.js` — Node 20+; update npm scripts |
| **Shell export** | `export MONGODB_URI=...` before `npm run dev` — fine for you, fragile for teammates |

Whatever you pick: env must be loaded **before** `config.js` validation runs.

Update `server/package.json` dev script if using `--env-file`:

```json
"dev": "node --env-file=.env --watch src/server.js"
```

Adjust path if `.env` lives in `server/` and you run from that directory.

## Validation — fail fast

Reading `process.env.MONGODB_URI` without validation gives `undefined` — Mongoose then throws a confusing error deep in the stack.

**Better:**

- At startup, `config.js` checks required keys  
- Missing key → log **"Missing required env: MONGODB_URI"** → exit non-zero  
- Developer fixes `.env` before retrying  

Optional enhancements (not required for gate):

- Validate URI **shape** starts with `mongodb://` or `mongodb+srv://`  
- Warn if `NODE_ENV=production` but URI points to localhost  

You implement validation logic — course specifies **behaviour**, not code.

## Defaults — when they are OK

| Setting | OK to default in code? | Example |
|---|---|---|
| `PORT` | yes | `process.env.PORT || 5000` |
| `NODE_ENV` | yes | default `development` |
| `MONGODB_URI` | **no** | must come from env — no silent default |
| `JWT_SECRET` (Ch 6) | **no** | must come from env |

**Rule:** Secrets and environment-specific endpoints **never** default in source. Convenience defaults for port are standard.

## Client-side env (Vite recap)

Vite only exposes variables prefixed with `VITE_`:

```env
VITE_API_URL=http://localhost:5000
```

In code: `import.meta.env.VITE_API_URL`

**Why different from server:** Client env is embedded in the **browser bundle** at build time. Never put `MONGODB_URI` in a `VITE_` variable — it would ship to every user.

Chapter 2 established this; Chapter 4 **re-verifies** client env while adding server env.

## Config in tests and seed (preview)

Chapter 5's `scripts/seed.js` will need the same `MONGODB_URI`. Import `connectDB` from `shared/db.js` and config from `shared/config.js` — **one connection path** for API and scripts.

If seed hardcodes a different URI than the API, you'll debug ghost data for hours.

| Value | Server `.env` | Client `.env` | Source code | `.env.example` |
|---|---|---|---|---|
| `MONGODB_URI` | ✓ real | ✗ | ✗ | placeholder only |
| `VITE_API_URL` | ✗ | ✓ | ✗ | placeholder |
| Default PORT 5000 | optional override | ✗ | ✓ fallback in config | optional |
| Express middleware order | ✗ | ✗ | ✓ | ✗ |

## ✅ Do / ❌ Don't

**✅ Do:** Document in README: "Copy `server/.env.example` to `server/.env` and set MONGODB_URI."

**✅ Do:** Re-run server after editing `.env` — process doesn't hot-reload env.

**❌ Don't:** Commit `.env` "just this once" because mentor needs it — send them `.env.example` instructions.

**❌ Don't:** Use `VITE_MONGODB_URI` — architectural mistake.
