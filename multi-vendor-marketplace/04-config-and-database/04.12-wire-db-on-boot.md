# 4.12 — Wire DB on boot

**Why this step matters:** Boot order — config, connect, then listen — prevents the API from accepting traffic while the database is still disconnected. Production APIs fail startup rather than lie to health checks; you adopt that discipline on localhost first.

**Where the project is now:** `config.js` and `db.js` exist but `server.js` may still call `listen()` immediately without connecting MongoDB.

**After this step:** Server startup sequence loads env → validates config → **awaits database connection** → starts HTTP listener. Failed connection prevents normal boot.

---

## Target boot sequence

```
server.js (entry)
  1. Load environment (.env)
  2. Import config (validation runs)
  3. await connectDB()
  4. app.listen(config.port)
  5. Log "API listening on port X"
```

**Async entry:** Top-level `await` in ESM is valid in Node when `"type": "module"`. Alternative: wrap in `async function start()` and call it.

**You implement** — course specifies order, not full file contents.

---

## `server.js` responsibilities

| Concern | Belongs in server.js? |
|---|---|
| Load dotenv / env file | yes — first |
| Call `connectDB()` | yes — before listen |
| Create Express app | no — stays in `app.js` |
| Define routes | no — `app.js` |
| Graceful shutdown | optional yes — SIGINT → disconnectDB |

**Minimal shape (pseudocode):**

```javascript
// import env loader
// import app from './app.js'
// import { config } from './shared/config.js'
// import { connectDB } from './shared/db.js'

// async function start() {
//   await connectDB()
//   app.listen(config.port, () => log port)
// }
// start().catch(err => { log err; process.exit(1) })
```

Unhandled rejection in `start()` must **exit non-zero** — not leave zombie process without listener or with half-connected state.

## `app.js` unchanged this step?

**Do not listen in `app.js`.** Only `server.js` listens — unchanged rule from Chapter 2.

You may add health db field in 4.13 in `app.js` — not in this sub-chapter unless you combine steps.

---

## Expected terminal output (happy path)

When `npm run dev` runs with valid `.env` and MongoDB up, you should see **both**:

1. MongoDB connected (or equivalent)  
2. API listening on port XXXX  

Order: connection log **before** listen log.

If you only see listen log, boot order bug — connect not awaited.

---

## Failure path — missing or bad config

**Test A — missing URI:**

Comment out `MONGODB_URI` in `.env`.

**Expected:** Server fails at config validation **before** Mongoose — clear env error.

**Test B — wrong URI / MongoDB down:**

Invalid host or stopped `mongod`.

**Expected:** `connectDB()` fails; process exits; **no** "listening on port" message — or listen never reached.

Restore valid `.env` and running MongoDB after tests.

## Update npm scripts if using `--env-file`

`server/package.json`:

```json
{
  "scripts": {
    "dev": "node --env-file=.env --watch src/server.js",
    "start": "node --env-file=.env src/server.js"
  }
}
```

If using dotenv, dev script may stay `node --watch src/server.js` with `import 'dotenv/config'` at top of `server.js`.

Run from `server/` directory so `.env` path resolves.

## ✅ Do / ❌ Don't

**✅ Do:** Use `config.port` for listen — not hardcoded 5000.

**✅ Do:** Catch startup errors and exit — PM2/deploy platforms restart clean process.

**❌ Don't:** `listen` first and connect in background — race for Chapter 4 health honesty.

**❌ Don't:** Swallow connect errors — silent API without DB is worse than crash.

## Troubleshooting

| Symptom | Fix |
|---|---|
| Server listens but MongoDB down | connect not awaited or errors caught empty |
| `Top-level await` syntax error | Node version; or wrap in async IIFE |
| Env vars undefined | Load dotenv before config import order wrong |
| Watch restarts loop | Fix URI; check Atlas IP |
