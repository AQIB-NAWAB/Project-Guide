# 4.2 — Why secrets matter

You are about to create `server/.env` with a **MongoDB connection string** — a secret that grants read/write access to your database. One mistaken `git add` and that string lives in Git history forever, even if you delete the file later.

This sub-chapter is not fear-mongering. It is the **habit layer** that makes everything in 4.9–4.15 safe to execute.

## Why this matters for absolute beginners

Leaks rarely come from Hollywood hackers — they come from **committed `.env` files**, Discord screenshots, and `console.log(process.env)`. You are about to paste a string that grants full read/write to your database. Learning Git ignore rules and `.env.example` **before** that paste is cheaper than rotating Atlas passwords after a public repo scan. These habits protect you when the project is solo; they save you when the repo becomes portfolio or team work.

## Think of it like…

**The key to the stockroom.** Customers and vendors never get copies — only trusted staff (the server). You would not tape the key under the front door (GitHub). You keep a **template** listing which locks exist (`.env.example`) while the real key stays in your pocket (local `.env`).

## Common beginner questions

**Q: Is localhost MongoDB URI really a secret?**  
A: Less critical than Atlas — still keep in `.env` for consistency and deploy parity. Never put **any** URI in React.

**Q: What's the difference between `.env` and `.env.example`?**  
A: `.env` has real values — never commit. `.env.example` has fake placeholders — always commit so teammates know which keys to set.

**Q: I already committed `.env` once — what now?**  
A: Remove from tracking, rotate credentials, and ensure history is clean before continuing — `git log` on `.env` paths should be empty going forward.

**Q: Can I share `.env` with my mentor?**  
A: Share **instructions** (`cp .env.example .env`), not the file. Say "Atlas configured" in logs, not the password.

## What counts as a secret in this project

| Secret | Example location | If leaked |
|---|---|---|
| `MONGODB_URI` | `server/.env` | Full database read/write; data exfiltration; ransom |
| JWT signing key | `server/.env` (Chapter 6) | Forge tokens; impersonate any user |
| Cloudinary API secret | `server/.env` (Chapter 13) | Upload/delete assets in your account |
| Refresh token pepper | `server/.env` (Chapter 6) | Session hijacking at scale |

**Not secrets on the client:**

| Public config | Location | Why it's OK |
|---|---|---|
| `VITE_API_URL` | `client/.env` | Browser sees this anyway in Network tab |
| Cloudinary cloud name | client env later | Public identifier; uploads still need server secret |

Rule from Chapter 2: **anything that must not appear in browser JavaScript stays on the server.** MongoDB URI never touches React.

## Real-world leak patterns (why teams care)

Engineers learn this from incidents, not slides:

1. **Committed `.env` in a public repo** — bots scan GitHub within minutes for `mongodb+srv://` patterns.  
2. **Screenshot or screen-share** — terminal shows connection string during demo; recording uploaded.  
3. **Copied `.env` into Slack/Discord** — searchable forever.  
4. **Debug logging** — `console.log(process.env)` or logging full config object in production.  
5. **Client bundle mistake** — importing server config into Vite (tree-shaking won't save you from bad imports).

You are building alone today. In six months you may have collaborators, CI logs, and a public portfolio repo. **Habits formed in Chapter 4 prevent résumé embarrassment.**

## `.env` is local truth — Git is shared history

```
server/.env          ← YOUR machine / YOUR deploy host — NEVER commit
server/.env.example  ← COMMIT — keys only, placeholder values
.gitignore           ← MUST list .env
```

**`.env.example` contract:**

- Lists **every required variable name**  
- Uses **obvious placeholders** — not real credentials  
- Includes one-line comment per variable if helpful  

Teammates and mentors run:

```bash
cp server/.env.example server/.env
# then fill in real values locally
```

If a new env var is added in Chapter 6, **update `.env.example` in the same commit** — not "I'll document it later."

## Verify Git is not tracking secrets (do this now)

From your marketplace project root:

```bash
git check-ignore -v server/.env
```

**Expected:** `.gitignore` rule matches — file is ignored.

```bash
git log --all --full-history -- "**/.env"
```

**Expected:** **empty output** — no `.env` ever committed.

If you see history: stop and fix before adding `MONGODB_URI`. Rotating the Atlas password after a leak is painful; preventing commit is cheap.

## Atlas-specific note

MongoDB Atlas connection strings look like:

```
mongodb+srv://<username>:<password>@cluster0.xxxxx.mongodb.net/<dbname>?retryWrites=true&w=majority
```

The **password is in the URL**. Treat the entire string as one secret. Create a **database user with least privilege** (read/write on your app database only — not Atlas admin).

For local MongoDB:

```
mongodb://127.0.0.1:27017/your-db-name
```

Less sensitive on localhost-only dev — still keep in `.env` for consistency and Chapter 22 deploy parity.

## ✅ Do / ❌ Don't

**✅ Do:** Use different Atlas users or databases for dev vs production when you deploy (Chapter 22).

**✅ Do:** Rotate credentials if you suspect leak — treat as incident, not embarrassment.

**❌ Don't:** Put `.env` in `.gitignore` but force-add with `git add -f`.

**❌ Don't:** Paste connection strings into learning log or issue comments — write "configured Atlas cluster" instead.

**❌ Don't:** Email `.env` to yourself — use password manager or host env panel.
