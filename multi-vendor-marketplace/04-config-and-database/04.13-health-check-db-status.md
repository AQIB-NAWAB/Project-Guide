# 4.13 — Health check DB status

**Why this step matters:** Chapter 2's health proved the process was up; this extension proves **memory works**. The client Home page and ops tooling rely on honest `db` status — not a green `ok` when MongoDB is down.

**Where the project is now:** Server connects MongoDB on boot and listens. `/health` still returns only `{ "status": "ok" }` from Chapter 2.

**After this step:** `GET /health` JSON includes a **`db`** field reflecting MongoDB connection state — verified with curl.

---

## Extended contract

| Field | When | Value |
|---|---|---|
| `status` | always | `"ok"` when HTTP handler runs and API process healthy |
| `db` | Chapter 4+ | `"connected"` when Mongoose connection is active |

**Minimum success body (DB up):**

```json
{
  "status": "ok",
  "db": "connected"
}
```

You may add optional fields (`service`, `timestamp`) — do not remove `status`.

---

## Implementation guidance — you write handler logic

Health handler must **reflect actual connection state**, not hardcoded `"connected"`.

**Valid approaches:**

1. Read `mongoose.connection.readyState === 1` inside handler  
2. Use helper exported from `db.js` (e.g. `isDbConnected()`)  
3. If boot guarantees connect before listen, still verify at request time — connection can drop while server runs (tested in 4.15)  

**Invalid approach:**

```javascript
// ❌ Always lying
res.json({ status: 'ok', db: 'connected' })
```

---

## When DB disconnects at runtime (preview for 4.15)

If MongoDB goes down **after** successful boot, choose behaviour and document in README:

| Strategy | HTTP | Body | Notes |
|---|---|---|---|
| A | `503` | `{ "status": "degraded", "db": "disconnected" }` | Load balancers drop instance |
| B | `200` | `{ "status": "ok", "db": "disconnected" }` | Process up but DB bad — explicit field |

**Course accepts A or B** — consistency matters. Chapter 2 said don't return 200 with fake ok for DB — apply that principle to `db` field truthfulness.

If server **exits** when connection lost (advanced), health endpoint unavailable — also valid for 4.15 demo.

---

## Where to implement

Same location as Chapter 2 health route:

- Inline in `app.js`, **or**  
- `shared/health.routes.js` mounted on app  

Do not create a new module folder for one route unless you prefer — `shared/` is fine.

**Do not** expose `MONGODB_URI`, host credentials, or internal error stacks in JSON response — log details server-side only.

## Verify with curl

Server running, MongoDB up:

```bash
curl -s http://localhost:5000/health | jq .
```

**Expected:**

```json
{
  "status": "ok",
  "db": "connected"
}
```

Without `jq`:

```bash
curl -i http://localhost:5000/health
```

Confirm `200` and JSON body includes `"db":"connected"` (or spaced JSON equivalent).

## Connection to deploy (Chapter 22)

Platforms poll `/health` to keep instances alive. `db: connected` proves dependency readiness — not just Node event loop. Load balancers may require 503 when DB down — note your choice in learning log.

---

## Troubleshooting

| Problem | Likely cause |
|---|---|
| `db` missing from JSON | Handler not updated |
| Always `disconnected` while app works | Wrong readyState check; connect uses different mongoose instance |
| Health throws 500 | Handler accesses undefined mongoose import |
