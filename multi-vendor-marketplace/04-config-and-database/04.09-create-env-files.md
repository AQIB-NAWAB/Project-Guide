# 4.9 — Create env files

**Why this step matters:** This is the first moment a real secret enters your project. Creating `.env` locally and `.env.example` for Git establishes the pattern every later chapter (JWT, Cloudinary) repeats — names in repo, values only on your machine.

**Where the project is now:** Server runs without MongoDB configuration. No `server/.env` or validated config module.

**After this step:** `server/.env.example` committed; `server/.env` exists locally with real `MONGODB_URI`; Git ignores secrets.

---

## Step 1 — Verify `.gitignore`

Root `.gitignore` (from Chapter 2) should include:

```
.env
.env.*
!.env.example
```

Or explicit:

```
server/.env
client/.env
```

**Test before creating secrets:**

```bash
git check-ignore -v server/.env
```

If no match, add `.env` rules **now** and commit:

```bash
git add .gitignore
git commit -m "chore: ensure env files are gitignored"
```

---

## Step 2 — Create `server/.env.example`

Create **`server/.env.example`** — safe to commit:

```env
# Copy this file to .env and fill in real values
# cp server/.env.example server/.env

# MongoDB connection string (required)
MONGODB_URI=mongodb://127.0.0.1:27017/your-app-db

# HTTP port (optional — defaults to 5000 in config module)
PORT=5000

# Node environment (optional)
NODE_ENV=development
```

**Adjust placeholder URI** if you use Atlas — use fake host/password pattern:

```env
MONGODB_URI=mongodb+srv://USERNAME:PASSWORD@cluster.example.mongodb.net/your-app-db
```

Replace `your-app-db` with your chosen database name from 4.5.

---

## Step 3 — Create `server/.env` (local only)

```bash
cp server/.env.example server/.env
```

Edit **`server/.env`** — paste the **working** URI from mongosh/Compass testing:

```env
MONGODB_URI=mongodb://127.0.0.1:27017/greenbasket-dev
PORT=5000
NODE_ENV=development
```

Use your real product slug and real Atlas password if applicable.

**Double-check:**

```bash
git status
```

**Expected:** `server/.env` does **not** appear as untracked/staged. If it appears, `.gitignore` is wrong — fix before proceeding.

---

## Step 4 — Re-verify client env (from Chapter 2)

Confirm **`client/.env`** exists:

```env
VITE_API_URL=http://localhost:5000
```

Confirm **`client/.env.example`** matches — committed in Chapter 2.

If server `PORT` in `.env` is not 5000, update **both** client `.env` and `.env.example` to match — e.g. `http://localhost:5001`.

---

## Step 5 — Update root README

Add **Environment setup** section:

```markdown
## Environment setup

### Server

cp server/.env.example server/.env
# Edit server/.env — set MONGODB_URI to your local or Atlas connection string

### Client

cp client/.env.example client/.env
# Default VITE_API_URL=http://localhost:5000
```

One paragraph: server secrets never committed; client only has public API URL.

---

## Step 6 — Document env loading approach

In README or comment in `server.js` (next chapters), note how `.env` loads:

- `dotenv` at entry, **or**  
- `node --env-file=.env` in npm script  

You implement loading in 4.10/4.12 — but teammates reading README after this step should know `.env` file location.

---

## Commit

```bash
git add server/.env.example README.md .gitignore
git commit -m "chore(server): add env example and document setup"
```

**Do not** `git add server/.env`.

---

## ✅ Do / ❌ Don't

**✅ Do:** Run `git log --all -- "**/.env"` after commit — still empty.

**✅ Do:** Keep server and client `.env` separate files — different roots, different vars.

**❌ Don't:** Put `MONGODB_URI` in root `.env` unless config module reads from there — course convention is `server/.env`.

**❌ Don't:** Commit `.env.example` with your real Atlas password "as example" — rotate password if you did.

---

## Troubleshooting

| Problem | Fix |
|---|---|
| `server/.env` shows in `git status` | Fix gitignore; never commit |
| Copied example but forgot to change password placeholder | Server won't connect — expected until fixed |
| Client can't reach API | PORT mismatch between server `.env` and `VITE_API_URL` |
