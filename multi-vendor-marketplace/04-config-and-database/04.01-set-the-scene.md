## In plain English

| Term | Meaning |
|---|---|
| **Environment variable** | A key-value setting outside your code — e.g. `MONGODB_URI` in `.env` |
| **`.env` file** | Local file holding secrets and config — never committed to Git |
| **`.env.example`** | A committed template listing variable **names** with fake placeholders |
| **Config module** | One file (`shared/config.js`) that reads and validates all env vars |
| **Boot order** | The sequence on startup: load config → connect DB → start listening |
| **Fail-fast** | Crash immediately with a clear error if required config is missing |
| **Mongoose** | Node library that connects to MongoDB — models come in Chapter 5 |
| **Connection string (URI)** | The address and credentials MongoDB expects — treat as a secret |

## Common beginner questions

**Q: Why config before schemas — isn't the data model the point?**  
A: Models need an **active connection**. More importantly, teams establish **secret-handling habits** before real user data exists. Leaking a connection string in Git is painful; learning `.env` discipline on an empty database is cheap.

**Q: Local MongoDB or Atlas — which should I pick?**  
A: Either works for the gate. Local is fine offline; Atlas matches many deploy guides. Sub-chapter 4.5 helps you decide; the course does not mandate cloud.

**Q: Will the React app talk to MongoDB directly?**  
A: **Never.** Only the server reads `MONGODB_URI`. The browser talks HTTP to Express; Express talks to MongoDB. Chapter 2 established that boundary — this chapter implements it on the server side.

**Q: What if the server starts but MongoDB is down?**  
A: You will design honest behaviour — fail at boot and/or report `db: disconnected` on health. The app must not pretend everything is fine when the database is unreachable.

**Q: Do I need to re-debate SQL vs MongoDB here?**  
A: No. Chapter 3 closed that decision. If the ERD gate is incomplete, finish the draft first — you can connect MongoDB, but implementing models without a blueprint invites rework.

## The gap between "runs" and "remembers"

Right now your marketplace can answer one question: *Is the Node process alive?*

It cannot answer:

- Is MongoDB reachable from this server?
- What connection string does this environment use?
- Should the API start if the database is down?

Those are **configuration and boot-order** problems — not schema problems. Chapter 5 writes Mongoose models; **Chapter 4 connects the wire**. Order matters: connect → then implement schemas → then seed.

## What "config & database" means in production terms

Professional Node APIs treat configuration as **inputs to the process**, not constants in source code:

| Concern | Where it lives | Why |
|---|---|---|
| MongoDB connection string | `server/.env` → read by `shared/config.js` | Different per machine (local, staging, production) |
| API port | `server/.env` or host default | Deploy platforms set `PORT` dynamically |
| Client API base URL | `client/.env` as `VITE_API_URL` | Already started in Chapter 2 |
| Secrets | Never in Git | `.gitignore` + `.env.example` without values |

**Fail-fast:** If `MONGODB_URI` is missing or invalid, the server should **not** silently run pretending everything is fine. Chapter 4 teaches that discipline before you have user data to lose.
