# 5.10 ‚Äî User and Profile models

**Why this step matters:** Every auth and ownership feature anchors on User ‚Äî email, role, verification state ‚Äî and Profile for display data. These schemas are prerequisites for register in Chapter 6; get field names right now.

## Design decision ‚Äî identity vs profile split

| Concern | User | Profile |
|---|---|---|
| Login email | ‚úì | |

**Why split:** Auth chapter rotates tokens and hashes passwords on User without touching profile display fields. Vendor dashboard shows profile name without exposing hash in same document queries.

Alternative single-document User is valid in industry ‚Äî **this course mandates split**.

## User schema specification

| Field | Type | Required | Notes |
|---|---|---|---|
| `email` | String | yes | Lowercase trim; unique index |
| `passwordHash` | String | yes | Plain password never stored ‚Äî Ch 6 hashes |
| `role` | String enum | yes | `'vendor'` \| `'customer'` |
| `isVerified` | boolean | yes | default `false` ‚Äî OTP in Ch 6 |
| `refreshTokenHash` | String | no | Ch 6 refresh flow |
| timestamps | auto | yes | `createdAt`, `updatedAt` |

**Indexes:**

- `{ email: 1 }` unique

**Seed note:** Seed may store pre-hashed placeholder or plain string documented as "replaced at auth" ‚Äî prefer bcrypt hash in seed if Ch 6 bcrypt cost known; or document re-seed after auth.

## Validation rules (schema-level)

Implement in Mongoose schema ‚Äî not HTTP layer:

- `email` ‚Äî format validator or regex; lowercase on save
- `role` ‚Äî enum only vendor/customer
- `passwordHash` ‚Äî min length on hash string (bcrypt ~60 chars)

**Do not** implement register route here.
