# 5.13 ‚Äî Cart and Order models

**Why this step matters:** Cart references live products; orders embed **snapshot** lines ‚Äî the hardest modelling choice in the marketplace. Getting embed field names right here makes Chapter 18 checkout a wiring task instead of a schema rescue.

**Where the project is now:** User, Profile, Store, and Product models exist. Cart and checkout collections missing.

**After this step:** `cartItem.model.js` and `order.model.js` complete the data model ‚Äî cart lines as refs, order lines as embedded snapshots.

---

## CartItem schema specification

Persistent cart ‚Äî not browser localStorage (Ch 16 API).

| Field | Type | Required | Notes |
|---|---|---|---|
| `userId` | ObjectId ref User | yes | Cart owner ‚Äî customer only |
| `productId` | ObjectId ref Product | yes | Live product ref |
| `storeId` | ObjectId ref Store | yes | Denormalised for checkout split |
| `quantity` | Number | yes | min 1 integer |
| timestamps | auto | yes | |

**Indexes:**

- `{ userId: 1 }` ‚Äî fetch cart
- `{ userId: 1, productId: 1 }` unique optional ‚Äî one line per product per user (merge qty on add ‚Äî Ch 16)

**Why storeId on line:** Checkout groups by store without joining Product at split time.

## Order schema specification

One Order document = **one vendor's portion** of a checkout ‚Äî not the whole multi-vendor cart.

### Top-level fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `customerId` | ObjectId ref User | yes | Buyer |
| `storeId` | ObjectId ref Store | yes | Fulfilling vendor |
| `status` | String enum | yes | See below |
| `paymentStatus` | String enum | yes | MVP: `'pending'` only |
| `orderItems` | [OrderItemSchema] | yes | Embedded array |
| `subtotal` | Number | yes | Sum of line totals at purchase |
| timestamps | auto | yes | |

**status enum (MVP):**

```
pending | packed | dispatched | cancelled
```

**paymentStatus enum:**

```
pending | paid | failed
```

Course scope: no real payment ‚Äî always `paymentStatus: 'pending'` until stretch.

---

### Embedded OrderItem sub-schema (no separate collection)

| Field | Type | Required | Notes |
|---|---|---|---|
| `productId` | ObjectId ref Product | yes | Traceability ‚Äî product may be deleted later |
| `nameAtPurchase` | String | yes | Snapshot |
| `priceAtPurchase` | Number | yes | Snapshot |
| `unitAtPurchase` | String | yes | Snapshot |
| `quantity` | Number | yes | min 1 |
| `lineTotal` | Number | yes | priceAtPurchase √ó quantity |

**Snapshot rule:** Values copied from Product at checkout (Ch 18) ‚Äî never updated when Product.price changes.

## Multi-vendor checkout preview

Customer cart with items from Store A and Store B:

```
Checkout ‚Üí Order #1 (storeId: A, orderItems: [...])
        ‚Üí Order #2 (storeId: B, orderItems: [...])
```

Two documents, same `customerId`, different `storeId`.

## Relationship diagram

```
User (customer) ‚Üê‚Äî‚Äî CartItem.userId
Product ‚Üê‚Äî‚Äî CartItem.productId
Store ‚Üê‚Äî‚Äî CartItem.storeId

User (customer) ‚Üê‚Äî‚Äî Order.customerId
Store ‚Üê‚Äî‚Äî Order.storeId
Product ‚Üê‚Äî‚Äî orderItems[].productId (ref + snapshot fields)
```

Vendor reads: `Order.find({ storeId: myStoreId })`

---

## CartItem vs Order ‚Äî ref vs snapshot

| Aspect | CartItem | Order.orderItems[] |
|---|---|---|
| Price | Read live from Product at display | Frozen at checkout |
| Mutability | PATCH quantity, DELETE line | Immutable after create |
| Lifetime | Until checkout clears cart | Permanent history |
