# 5.2 — The denormalisation trap

Every junior marketplace tutorial tempts you to **copy display names onto child documents** so queries feel simple. *"Just put `vendorName` and `storeName` on every product — one query, no joins."*

That shortcut creates a maintenance nightmare. This sub-chapter explains the trap so you **reference** stores instead of duplicating names — and understand when denormalisation is actually justified (order snapshots in Chapter 18).

## Why this matters for absolute beginners

Denormalisation is a real technique — not always wrong. Beginners apply it in the **wrong place**: copying store names onto products "to avoid populate." That feels like optimization; it is **stale data waiting to happen**. The rule of thumb: **live catalogue references; historical receipts copy.** Learn that distinction now and Chapter 18 checkout will feel obvious instead of contradictory.

## Think of it like…

**A mall directory vs a printed receipt.** The directory (catalogue) points to "Green Valley Grocer — moved to Level 2" in one place. The receipt in your wallet still says what you paid yesterday — it does not auto-update when the shop renames.

## Common beginner questions

**Q: Isn't populate slow — why not copy store name?**  
A: One batched populate per page is fine (Chapter 14 optimizes further). Copying names onto N products is write amplification on every rename.

**Q: CartItem — embed product or reference?**  
A: Reference live product pre-checkout; snapshot at order creation — not permanent embed on cart.

**Q: How do I explain this in the viva?**  
A: "Reference for mutable shared data; copy for immutable purchase history."

## The tempting pattern

Imagine a product document like this:

```json
{
  "name": "Organic Bananas",
  "price": 2.49,
  "vendorName": "Green Valley Grocer",
  "storeName": "Green Valley — Main St",
  "storeAddress": "42 Market Lane"
}
```

**Feels good because:**

- Catalogue page reads one collection — no populate
- Product card shows store name immediately
- Tutorial completes in ten minutes

**Fails because:**

| Event | Consequence |
|---|---|
| Vendor renames store | Every product document stale until bulk update |
| Vendor changes address | Same — N products to patch |
| Typo in copied name | Two spellings of same store in catalogue |
| Vendor deleted | Orphan strings with no link back to owner |

In a multi-vendor marketplace, **store identity is authoritative in one place** — the `stores` collection.

---

## The correct pattern for live catalogue data

```
products.storeId  →  ObjectId ref to stores._id
```

**Read path (Chapter 14):**

- `Product.find({ isPublished: true }).populate('storeId', 'name slug')`
- One query with populate — not N+1 if done correctly
- Store rename updates **one** document; catalogue reflects immediately

**Write path (Chapter 11):**

- Vendor creates product with their own `storeId` only
- Ownership check: product.storeId belongs to vendor's store

## When denormalisation IS correct — order snapshots

Checkout is different from catalogue. At purchase time you **must copy** price, name, and unit into the order — because the product document will change tomorrow.

```json
{
  "orderItems": [
    {
      "productId": "...",
      "nameAtPurchase": "Organic Bananas",
      "priceAtPurchase": 2.49,
      "unitAtPurchase": "bunch",
      "quantity": 2
    }
  ]
}
```

| Context | Reference or copy? | Why |
|---|---|---|
| Live product catalogue | **Reference** storeId | Source of truth stays current |
| Order history | **Copy (embed snapshot)** | Customer paid 2.49 forever — not today's price |
| Cart (pre-checkout) | **Reference** productId | Price may update before checkout — re-fetch or warn |

Chapter 5 embeds snapshot **field names** on Order schema. Checkout logic fills them in Chapter 18.

## Real-world cost story

A food marketplace shipped with denormalised vendor names on 40,000 products. Rebrand campaign required a **batch migration script** and two days of stale search results. Referenced design: one `stores` update, populate handles the rest.

You won't hit 40,000 products in this course — but the **pattern** is the same at 40 products.

## Red flags in schema review

If your ERD or model has any of these on **Product**, push back:

- `vendorName` string (use storeId ref + populate)
- `vendorEmail` on product (PII leak + stale data)
- `storeName` duplicated (use populate unless order snapshot context)

If your **Order** lacks snapshot fields, push back:

- Order line only stores `productId` with no price copy
- "We'll look up price later" — breaks immutability

## Decision for FreshMarket (this course)

| Document | Relationship to Store | Denormalise? |
|---|---|---|
| Product | `storeId` ref | **No** — populate for display |
| CartItem | `productId` + `storeId` refs | **No** — live prices until checkout |
| Order | `storeId` ref + embedded items | **Yes** — snapshot fields on embed |

Document this table in your learning log when models are done.

---

## ✅ Do / ❌ Don't

**✅ Do:** Use `storeId: { type: ObjectId, ref: 'Store' }` on Product.

**✅ Do:** Plan `nameAtPurchase`, `priceAtPurchase`, `unitAtPurchase` on embedded order lines.

**❌ Don't:** Add `vendorName` to Product to "save a populate" — you'll unpick it in code review.

**❌ Don't:** Embed full product documents inside CartItem — cart is mutable pre-checkout.
