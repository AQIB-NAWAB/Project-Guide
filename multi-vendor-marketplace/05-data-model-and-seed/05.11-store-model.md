# 5.11 — Store model

**Why this step matters:** The store is the hub of vendor ownership — one shop per vendor, linked by `userId`. Product and order isolation in later chapters walk this link; a weak Store schema breaks the whole chain.

**Where the project is now:** User and Profile models exist. No Store schema yet.

**After this step:** `store.model.js` defines the vendor shop document with `userId` ref to User — foundation for product ownership and Chapter 9 store API.

---

## Store in the marketplace narrative

A **vendor** account is not a shop. After registration and OTP (Ch 6–7), a vendor **creates** a Store document. Until then, vendor has User + Profile but no `stores` row.

One vendor → one store (enforced in Ch 9 API; unique index on `userId` helps here).

---

## Store schema specification

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | String | yes | Public shop name — "Green Valley Grocer" |
| `slug` | String | yes | URL-safe unique — `green-valley-grocer` |
| `description` | String | no | Editorial intro for shop page |
| `address` | String | no | Pickup/delivery area hint |
| `logoUrl` | String | no | Cloudinary — Ch 13 |
| `userId` | ObjectId ref User | yes | Owning vendor |
| `isActive` | Boolean | yes | default `true` — soft hide shop |
| timestamps | auto | yes | |

**Indexes:**

- `{ userId: 1 }` **unique** — one store per vendor
- `{ slug: 1 }` unique — public URLs later

---

## Ownership chain link

```
User (_id) ←—— Store.userId
Store (_id) ←—— Product.storeId  (5.12)
```

Chapter 8 `assertStoreOwner(storeId, userId)`:

1. Load Store by `_id`
2. Compare `store.userId.toString()` with `userId` from JWT
3. 403 if mismatch

Schema must keep `userId` required and indexed.

## Slug generation (spec only)

Chapter 9 create-store route will slugify name. For seed script:

- Store A: slug `vendor-a-market` or derived from name
- Store B: slug `vendor-b-market`

Document slugs in seed — used in isolation tests and optional public URLs.
