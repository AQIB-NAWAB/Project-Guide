# 5.3 — What you'll build

By the end of Chapter 5, your repo contains **six Mongoose models**, a **runnable seed script**, and a **final ERD** committed to `docs/erd.md`. No API routes yet — only schemas and test data.

## Why this matters for absolute beginners

Chapter 5 is where FreshMarket becomes **real in the database** — not health checks, not empty folders, but vendors, stores, and products you can click in Compass. The deliverables table is your gate checklist. Weak models here mean every auth, cart, and checkout chapter fights the same schema mistakes. Strong models mean you mostly **wire routes** to shapes that already exist.

## Common beginner questions

**Q: Why six models before any routes?**  
A: Routes without schemas guess field names. Register needs User+Profile; checkout needs Order embed shape — define once, reuse everywhere.

**Q: Must seed create exactly `vendor-a@test.local`?**  
A: Use deterministic test emails documented in README — Chapter 8 isolation tests assume two distinct vendors.

**Q: What's idempotent seed?**  
A: Running `npm run seed` twice does not duplicate stores — clear-then-insert or upsert; document your approach.

**Q: Keep `erd-draft.md`?**  
A: Yes — add `erd.md` as final; diff them in learning log to show design evolution.

## Seed data specification

The seed script must create **deterministic, named test entities** for isolation tests in Chapter 8:

| Entity | Count | Notes |
|---|---|---|
| Vendor users | 2 | `vendor-a@test.local`, `vendor-b@test.local` — passwords hashed in Ch 6; plain for seed OK if documented |
| Customer users | 2 | `customer-a@test.local`, `customer-b@test.local` |
| Profiles | 4 | One per user |
| Stores | 2 | Store A → Vendor A; Store B → Vendor B |
| Products | 5–10 per store | Mix of perishable/non-perishable; varied units |
| CartItems | 0 | Created in later chapters |
| Orders | 0 | Created at checkout demo |

**Idempotent seed:** Running seed twice should not duplicate stores — use upsert or clear-then-insert pattern. Document your choice in README.

## Model design principles (course mandate)

1. **Module colocation** — each model lives in its feature module folder
2. **Refs over embed** — except order line snapshots
3. **Grocery fields** — not generic `{ title, price }` only
4. **Indexes noted** — unique email on User; compound indexes where queries demand (document in ERD)
5. **Timestamps** — `createdAt` / `updatedAt` on all collections
6. **No route handlers** — models and seed only
