# 5.4 — Ownership chain

Multi-vendor marketplaces fail in production when **object-level ownership** is unclear. Authentication answers *who are you?* Ownership answers *may you touch this document?*

This sub-chapter maps the **ownership chain** — the reference path every isolation check in Chapter 8 will walk.

## Why this matters for absolute beginners

Roles alone are not enough — "vendor" does not tell you **which** store. The chain User → Store → Product tells every handler which documents to load and compare against `req.user.id`. If Product stored `vendorId` directly, you could skip the store — but this course enforces **one store per vendor** through Store as the hub. Learn the chain now; Chapter 8 walks it with middleware.

## Think of it like…

**Nested keys in a hotel.** Guest key opens their room (customer cart). Staff key opens their floor's supply closet (vendor products in their store). Security checks not just "are you staff?" but **which floor you belong to** — same role, different scope.

## Common beginner questions

**Q: Why `customerId` on Order but `userId` on Store?**  
A: Same User collection, different semantics — buyer vs vendor owner. Consistent ref naming helps grep and viva clarity.

**Q: Why `storeId` on CartItem?**  
A: Multi-vendor checkout splits by store without extra product lookups at checkout time.

**Q: What is IDOR?**  
A: Insecure Direct Object Reference — changing `:id` in URL to access someone else's row. Schemas must support ownership lookups to prevent it in Chapter 8.

## The chain in one diagram

```
User (role: vendor)
  └── owns → Store (userId)
              └── owns → Product (storeId)
                          └── referenced by → CartItem (productId, storeId)
                                              └── checkout → Order (storeId, customerId)
                                                              └── embeds → orderItems[]
```

**Customer chain:**

```
User (role: customer)
  └── owns → CartItem (userId)
  └── owns → Order (customerId)
```

Vendor never owns CartItem or customer Order directly — they see orders **scoped by storeId**.

## Each link — what it enforces

### User → Store

| Field | On Store | Rule |
|---|---|---|
| `userId` | ObjectId ref User | Exactly one store per vendor (enforced in Ch 9 API) |
| Lookup | `Store.findOne({ userId })` | Vendor dashboard entry point |

**Isolation:** Vendor B cannot create a product with Vendor A's `storeId` if API checks store ownership first.

### Store → Product

| Field | On Product | Rule |
|---|---|---|
| `storeId` | ObjectId ref Store | Every product belongs to exactly one store |
| Query | `{ storeId: vendorStore._id }` | Vendor product list — never global find |

**Isolation:** PATCH `/products/:id` must verify `product.storeId` matches caller's store.

### Product → CartItem

| Field | On CartItem | Rule |
|---|---|---|
| `productId` | ObjectId ref Product | What the customer wants |
| `storeId` | ObjectId ref Store | **Duplicated on line** — enables checkout split without re-query |
| `userId` | ObjectId ref User | Cart is per-customer |

**Why storeId on CartItem when product has storeId?** Checkout groups by store in one pass. Cart line carries store context even if product is deleted later.

### CartItem → Order

| Transition | Behaviour |
|---|---|
| Checkout | Group cartitems by `storeId` |
| Per group | One Order document with `customerId`, `storeId`, embedded `orderItems` |
| Snapshots | Copy price, name, unit at purchase into embed |

**Vendor view:** `Order.find({ storeId: myStore._id })` — not by customerId.

## Reference table — collection ownership

| Collection | Owner field | Who can write |
|---|---|---|
| users | self | self (register); admin out of scope |
| profiles | userId | owning user |
| stores | userId | owning vendor |
| products | storeId → userId | owning vendor via store |
| cartitems | userId | owning customer |
| orders | customerId + storeId | customer creates; vendor updates status |

## IDOR preview (Chapter 8)

**Insecure direct object reference** — attacker changes `:id` in URL to another user's resource.

| Attack | Missing check | Fix |
|---|---|---|
| PATCH `/stores/other-vendor-store-id` | No userId match | `assertStoreOwner(storeId, req.user.id)` |
| PATCH `/products/other-product-id` | No storeId chain | Load product → verify store.userId |
| GET `/cart` as User A with User B token | No userId filter | Always `{ userId: req.user.id }` |

Your schemas must **support** these lookups — fields named consistently (`userId`, `storeId`, `customerId`).

---

## Mongoose ref naming convention

Use **consistent ref strings** matching model names:

| Model file | `mongoose.model()` name | ref in other schemas |
|---|---|---|
| user.model.js | `'User'` | `ref: 'User'` |
| store.model.js | `'Store'` | `ref: 'Store'` |
| product.model.js | `'Product'` | `ref: 'Product'` |

Mismatch (`ref: 'store'` vs model `'Store'`) breaks populate silently.

## Query patterns you'll implement later

**Vendor products (Ch 11):**

```
1. req.user.id from JWT
2. store = Store.findOne({ userId: req.user.id })
3. Product.find({ storeId: store._id })
```

**Never:** `Product.findById(id)` then return without store ownership check.

**Customer cart (Ch 16):**

```
CartItem.find({ userId: req.user.id }).populate('productId')
```

## ✅ Do / ❌ Don't

**✅ Do:** Use `userId` on Store (vendor owner), `customerId` on Order (buyer) — distinct semantics.

**✅ Do:** Index foreign keys you query often (`storeId` on Product, `userId` on CartItem).

**❌ Don't:** Store vendor id directly on Product — always go through Store for one-store-per-vendor rule.

**❌ Don't:** Use ambiguous field names like `owner` without role context.
