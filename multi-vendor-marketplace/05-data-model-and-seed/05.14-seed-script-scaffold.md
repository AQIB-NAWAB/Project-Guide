# 5.14 — Seed script scaffold

**Why this step matters:** Empty schemas do not prove your design — two vendors, two stores, and sample products in Compass do. The seed script is how you (and Chapter 8 tests) get predictable data without manual Compass clicking every reset.

**Where the project is now:** All six models exist but MongoDB collections empty (or absent).

**After this step:** `server/scripts/seed.js` connects via shared `db.js`, inserts deterministic test data, exits cleanly. `npm run seed` documented in README.

---

## Seed script contract

| Requirement | Detail |
|---|---|
| Entry | `server/scripts/seed.js` |
| Connection | Import `connectDB` from `../src/shared/db.js` |
| Config | Same `MONGODB_URI` as API — via config module |
| Idempotency | Clear collections or upsert — document strategy |
| Exit code | 0 on success; non-zero on failure |
| Disconnect | Call disconnect after seed — don't hang process |

---

## Target npm script

`server/package.json`:

```json
{
  "scripts": {
    "dev": "...",
    "start": "...",
    "seed": "node --env-file=.env scripts/seed.js"
  }
}
```

Adjust if using dotenv import inside seed file instead of `--env-file`.

Run from `server/` directory:

```bash
npm run seed
```

## Insert order (foreign key dependencies)

```
1. Users (4) — vendors A/B, customers A/B
2. Profiles (4) — one per user
3. Stores (2) — linked to vendor users
4. Products (10–20) — linked to stores
5. CartItems — skip (0)
6. Orders — skip (0)
```

Load User `_id` after insert before creating Store `userId`.

## Idempotency strategies (pick one)

| Strategy | Behaviour |
|---|---|
| **Clear and insert** | `deleteMany({})` on seeded collections in reverse dependency order — simple for dev |
| **Upsert by email/slug** | Safer for shared Atlas — match on unique keys |

Document choice in script header comment and README.

**Warning:** Clear strategy wipes data — fine for local dev; caution on shared DB.
