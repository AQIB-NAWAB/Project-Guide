## In plain English

| Term | Meaning |
|---|---|
| **Mongoose schema** | Blueprint for one collection — field types, required flags, enums |
| **Model** | The compiled schema you call to create and query documents |
| **`ObjectId`** | MongoDB's default id type — used for references between documents |
| **`ref`** | Field pointing to another collection's document — enables `populate()` |
| **Seed script** | A one-off program that inserts test data — two vendors, sample products |
| **Ownership chain** | User → Store → Product — who may touch which document |
| **Snapshot fields** | Copied price/name/unit on order lines — frozen at purchase time |
| **Denormalisation** | Copying data for speed — correct on orders, wrong on live products |

## Common beginner questions

**Q: Why models before auth — shouldn't login come first?**  
A: Login **creates and reads User documents**. Register needs fields (`email`, `passwordHash`, `role`, `isVerified`) defined in a schema. Building auth against imaginary users leads to rework. Data shape first; HTTP auth layer second.

**Q: Why separate User and Profile?**  
A: Login identity (email, password, role) changes rarely; display info (name, avatar) updates often. Splitting collections keeps auth lean and matches how Chapter 6 register flow creates both records.

**Q: Can I skip seed and add data manually in Compass?**  
A: Seed is **required for the gate** — two vendors, two stores, products with grocery fields. Chapter 8 isolation tests assume predictable test emails like `vendor-a@test.local`.

**Q: What's wrong with putting `vendorName` on every product?**  
A: When the vendor renames their shop, every product copy goes stale. Reference `storeId` and look up the name when displaying — except on **orders**, where copied names are intentional receipts.

**Q: Do I build API routes this chapter?**  
A: No — models and seed only. Routes arrive from Chapter 6 onward. If you add routes now, you blur the course's step-by-step arc.

## The gap between "connected" and "useful"

Right now MongoDB answers: *Can the server reach the database?*

It cannot answer:

- What does a vendor look like in a document?
- How does a product know which store it belongs to?
- What gets copied into an order at checkout time?

Those are **schema decisions** — the most consequential choices in a marketplace. Wrong refs break isolation in Chapter 8. Missing snapshot fields break order history in Chapter 18. Chapter 5 is where design from Chapter 3 becomes **enforceable structure**.
