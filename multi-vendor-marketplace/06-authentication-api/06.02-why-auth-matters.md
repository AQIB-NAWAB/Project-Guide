# 6.2 — Why auth matters

Authentication failures make headlines. For a marketplace handling vendor inventory and customer orders, **identity integrity** is not optional — it's the foundation every isolation check in Chapter 8 assumes.

This sub-chapter motivates bcrypt, JWT discipline, and verification gates — not to scare, but to make lazy shortcuts feel unacceptable.

## Why this matters for absolute beginners

Auth is where beginners most often cut corners — plain text passwords in tutorials, tokens in localStorage, skipping email verification "for now." Each shortcut becomes a **real vulnerability** at scale. You are building habits: hash passwords, never log secrets, verify before trusting. Chapter 8 assumes every request can carry a trusted user id — that assumption starts here.

## Think of it like…

**Bank account opening vs walking into a shop.** Anyone can browse produce (public catalogue later). To run a stall or place orders tied to your name, you prove identity — ID check (register + verify), secret PIN (password), daily wristband (JWT). No wristband, no access to someone else's cart.

## Common beginner questions

**Q: Is bcrypt overkill for a learning project?**  
A: No — the pattern is identical in production. Use real bcrypt with reasonable cost factor.

**Q: Why verify email with OTP if email does not really send?**  
A: The **state machine** matters — `isVerified` gates login and vendor onboarding. Delivery is simplified; flow is not.

**Q: Does auth replace ownership checks?**  
A: No — authn is who; authz is what you may touch (Chapter 8).

## The LinkedIn lesson (pattern, not year-specific)

Large breaches often share a root cause: **passwords stored reversibly or hashed weakly**, plus **tokens that never expire**.

When database dumps leak:

| Stored as | Attacker capability |
|---|---|
| Plain text | Immediate account takeover |
| MD5/SHA1 without salt | Rainbow table crack at scale |
| bcrypt/scrypt/argon2 | Slow brute force — buys time to force password reset |

Your course mandate: **bcrypt** on User.passwordHash — never store or log plain passwords.

## Marketplace-specific risks

| Actor | If auth fails |
|---|---|
| Customer A | Sees Customer B's cart and orders |
| Vendor B | Edits Vendor A's products (IDOR — Ch 8 without auth is worse) |
| Anonymous | Registers admin role — if role not validated |

Auth does not alone prevent IDOR — but without auth there is no user id to check.

## Password rules for this course

| Rule | Implementation |
|---|---|
| Never log password | Not in console, not in error messages |
| Never return passwordHash in API JSON | Strip on User.toJSON or select false |
| Hash on register | bcrypt in auth.service |
| Compare on login | bcrypt.compare — timing-safe |
| Min length | Enforce in auth.schema — e.g. 8 chars |

Document test password in README for local dev only — not production pattern.

## Verification gate — why OTP exists

Marketplaces need **reachable email** before vendor can list products — reduces spam shops.

Course simplification:

- Generate OTP on register
- Store hash or expiry on User (field your choice — document in ERD update)
- `POST /auth/verify-otp` sets isVerified true
- Login rejects if isVerified false

**Dev delivery:** Log OTP to server console or include in register response when `NODE_ENV=development` — document clearly.

## ✅ Do / ❌ Don't

**❌ Don't:** Skip verification because "it's dev" — flow must work for Ch 7 UI.

**❌ Don't:** Put JWT_SECRET in client env.
