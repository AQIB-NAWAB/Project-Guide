# 6.6 — What a JWT is

A JSON Web Token is three Base64 segments separated by dots: **header.payload.signature**. Understanding each part prevents you from treating JWT as magic or putting secrets in the payload.

## Why this matters for absolute beginners

JWT looks like random gibberish — beginners treat it as an opaque session id. It is not. Anyone can decode the payload; only the signature proves the server issued it. That is why `requireAuth` must **verify**, not decode — and why role comes from verified token, never from client body fields.

## Think of it like…

**A tamper-evident food delivery seal.** You can read the label (payload) through the bag; you cannot change "order #42" without breaking the seal (signature). Broken seal — reject at the door.

## Common beginner questions

**Q: Where does `req.user.role` come from?**  
A: Verified JWT claims after `jwt.verify` — not from request body on PATCH.

**Q: Can I skip expiry?**  
A: No — short-lived access is core to the refresh pattern.

**Q: Refresh token — JWT or random string?**  
A: Course allows either if hashed in DB; document your choice.

## Anatomy

```
eyJhbGciOiJIUzI1NiIs...   ← header (alg, typ)
.
eyJzdWIiOiI2Nj...         ← payload (claims)
.
SflKxwRJSMeKKF2QT4fwpM... ← signature
```

### Header

```json
{ "alg": "HS256", "typ": "JWT" }
```

Algorithm must match verify — HS256 with JWT_SECRET for this course.

### Payload (claims)

```json
{
  "sub": "664a1b2c3d4e5f6789012345",
  "role": "vendor",
  "iat": 1710000000,
  "exp": 1710000900
}
```

**Claims are base64-decodable by anyone** — not encrypted. No passwords, no PII you wouldn't show publicly.

### Signature

```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  JWT_SECRET
)
```

Server recreates signature on verify — tampered payload fails verification.

---

## Access token payload for FreshMarket

| Claim | Value | Used by |
|---|---|---|
| sub | User._id string | requireAuth → req.user.id |
| role | vendor \| customer | requireRole Ch 8 |
| iat | timestamp | debugging |
| exp | timestamp | auto 401 when expired |

Use `jsonwebtoken.sign(payload, JWT_SECRET, { expiresIn: config.accessExpiry })`.

---

## Verification in requireAuth (spec)

```
1. Extract Bearer token from Authorization header
2. jwt.verify(token, JWT_SECRET)
3. On success → req.user = { id: payload.sub, role: payload.role }
4. On JsonWebTokenError / TokenExpiredError → 401
```

Never trust client-sent userId without verify.

---

## Expiry behaviour

Expired access → client gets 401 → Chapter 7 refresh loop → new access.

Short expiry limits window if token stolen from memory.

---

## Common mistakes

| Mistake | Consequence |
|---|---|
| `jwt.decode` without verify | Attacker forges payload |
| JWT_SECRET in repo | Anyone mints tokens |
| role from req.body not token | Privilege escalation |
| No exp claim | Token valid forever |

## Refresh token difference

Refresh token may be opaque random string stored hashed — **not required to be JWT**. Course allows JWT refresh with separate expiry claim — document choice.

If JWT refresh: include `jti` (unique id) for rotation tracking.

---

## ✅ Do / ❌ Don't

**✅ Do:** Read exp from verify — don't skip expiry check.

**✅ Do:** Use same JWT_SECRET for access; could use separate REFRESH_SECRET — optional stretch.

**❌ Don't:** Store JWT in URL query params — logs and referrer leaks.

**❌ Don't:** Put email in JWT if avoidable — use /me for profile.
