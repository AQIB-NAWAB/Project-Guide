# 6.14 — Access token

**Why this step matters:** Short-lived access JWT is what `requireAuth` verifies on every protected call — payload must carry user id and role without leaking password hash or email you should not expose.

**Where the project is now:** Login returns tokens; access JWT structure may be ad hoc.

**After this step:** Access token is a **short-lived signed JWT** with `sub` (userId) and `role` claims. Issued by shared `signAccessToken(user)` helper used by login and refresh.

---

## Token specification

| Property | Value |
|---|---|
| Algorithm | HS256 |
| Secret | `config.jwtSecret` |
| Expiry | `config.jwtAccessExpiry` — default `15m` |
| Transport | `Authorization: Bearer <token>` |

---

## Payload contract

```json
{
  "sub": "664a1b2c3d4e5f6789012345",
  "role": "vendor",
  "iat": 1710000000,
  "exp": 1710000900
}
```

| Claim | Source |
|---|---|
| sub | User._id.toString() |
| role | User.role |
| iat, exp | jwt.sign automatic |

No email, no passwordHash, no refresh data in access token.

---

## Service helper (implement)

```
signAccessToken(user) → string JWT
```

Used by:

- loginUser (6.13)  
- refreshAccessToken (6.15)  

---

## Verification in requireAuth (6.17 preview)

```
jwt.verify(token, config.jwtSecret)
→ payload.sub → req.user.id
→ payload.role → req.user.role
```

Expired token → 401 `{ message: "Token expired" }` or generic Unauthorized.

## ✅ Do / ❌ Don't

**✅ Do:** Read expiry from config — not hardcoded in service.

**✅ Do:** Use `sub` standard claim for user id.

**❌ Don't:** Accept `alg: none` — use library defaults that reject it.

**❌ Don't:** Put refresh token in access payload.

---

## Verify checklist

- [ ] Access token verifies with JWT_SECRET
- [ ] Payload contains sub and role
- [ ] Expiry matches config (15m default)
- [ ] login response accessToken present
- [ ] Tampered token fails verify
