# 6.1 — Set the scene

**Where the project is now:** Week 1 complete — models and seed data in MongoDB. Every API request is **anonymous**. `GET /health` works; no `/api/auth` routes. User documents exist in Compass with password hashes from seed.

**Where this chapter goes:** You build the **Authentication API** — register, login, refresh, OTP verification — bcrypt for passwords, JWT for access tokens, hashed refresh tokens in the database, and `requireAuth` middleware that returns 401 when credentials are missing or invalid.

This is a **backend-only chapter**. No React screens until Chapter 7.

## Bridge from last chapter

Chapter 5 completed Week 1 foundations. Compass shows users, stores, products — seeded with two vendors and grocery fields. Your ownership chain is real: User → Store → Product, with CartItem and Order schemas ready for later chapters. Every HTTP request is still **anonymous**; anyone who could reach your API could theoretically mutate any document because nothing checks *who* is calling.

What carries forward: User and Profile models (with `passwordHash`, `role`, `isVerified`), `shared/config.js` patterns, and module layout — auth **HTTP layer** lands in `modules/auth/` while models stay in `modules/users/`. What you add now is **identity on the wire**: register, login, bcrypt hashing, JWT access tokens, hashed refresh tokens, OTP verification, and `requireAuth` middleware returning 401 when credentials are missing or invalid.

**Why auth comes after schemas, not before:** Register creates a User document and a Profile document. Login reads `passwordHash` and `isVerified` from fields you defined in Chapter 5. Refresh tokens store a hash on User. Building auth first would mean guessing field names and refactoring when models arrive. Professional order is **data shape → auth API → auth UI** — same climb this course follows. Chapter 8 adds authorization (what you may do); this chapter adds authentication (who you are).

The hat you wear now is **security-minded API builder**. You test every step with curl before any React form exists. Chapter 7 pairs to this chapter — same contracts, browser storage strategy — but the server must be trustworthy on its own first.

## In plain English

| Term | Meaning |
|---|---|
| **Authentication (authn)** | Proving who you are — login, tokens, passwords |
| **Authorization (authz)** | Proving what you may do — roles, ownership (Chapter 8) |
| **bcrypt** | One-way password hashing — never store plain passwords |
| **JWT** | A signed token the client sends on each request to prove identity |
| **Access token** | Short-lived JWT — sent as `Authorization: Bearer …` |
| **Refresh token** | Long-lived token used to get a new access token without re-login |
| **Middleware** | Code that runs before a route handler — e.g. reject requests without valid JWT |
| **OTP** | One-time code verifying email before account is fully active |

## Think of it like…

**Issuing market ID badges after the tenant roster exists.** Chapter 5 wrote the roster (User documents with roles). This chapter builds the security desk: register new vendors and customers, verify their email, issue a day pass (access token) and a longer membership card (refresh token), and check badges at the door (`requireAuth`). No badge, no entry — but the roster must exist before the desk can stamp anyone in.

## Common beginner questions

**Q: Why curl before React login pages?**  
A: If the API is wrong, a pretty form hides nothing. curl proves register, login, and protected routes work **in isolation** — then Chapter 7 wires the same contracts to buttons and inputs.

**Q: Why JWT instead of sessions?**  
A: This SPA sends Bearer tokens — a common MERN pattern. Sub-chapter 6.4 compares sessions honestly; the course picks JWT + hashed refresh for teaching and React fit.

**Q: Will OTP emails really send?**  
A: Not in MVP — log OTP to the server console or return in dev responses. The **flow** must work; production email is out of scope.

**Q: Can I skip OTP because it's annoying in dev?**  
A: No — Chapter 7's Verify OTP page and vendor onboarding depend on `isVerified`. Build the gate; simplify delivery, not the steps.

**Q: Does auth alone stop Vendor B editing Vendor A's products?**  
A: Not fully — that is **authorization** (Chapter 8). Auth gives you `req.user.id` and `role`; ownership checks use the chain from Chapter 5.

## The gap between "data exists" and "data is protected"

Right now anyone who can reach your API could theoretically write routes that mutate any document — there is no **caller identity** on requests.

Chapter 6 answers:

- Who is registering — vendor or customer?
- Are passwords stored safely?
- How does the client prove identity on each request?
- How do unverified users get blocked?

Chapter 8 adds **authorization** (what you're allowed to do). Chapter 6 adds **authentication** (who you are).
