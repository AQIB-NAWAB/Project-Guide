# 6.13 — Login handler

**Why this step matters:** Login proves bcrypt compare works and gates unverified users — the daily path vendors and customers repeat. Tokens issued here feed every protected route and Chapter 7's browser session.

**Where the project is now:** Register works; login endpoint stub or incomplete.

**After this step:** `POST /api/auth/login` validates credentials with bcrypt, blocks unverified users, returns access and refresh tokens on success.

---

## Request contract

**POST** `/api/auth/login`

**Body:**

| Field | Required |
|---|---|
| email | yes |
| password | yes |

---

## Success response — 200

```json
{
  "accessToken": "eyJhbG...",
  "refreshToken": "eyJhbG...",
  "user": {
    "id": "664a...",
    "email": "vendor-a@test.local",
    "role": "vendor",
    "displayName": "Vendor A"
  }
}
```

**user object:** No passwordHash. Include profile displayName from Profile lookup.

Token structure refined in 6.14–6.15 — login calls `issueTokens(user)` service helper.

---

## Error responses

| Status | Condition |
|---|---|
| 400 | missing email/password |
| 401 | wrong email or password |
| 403 | isVerified false |
| 500 | server error |

Use generic "Invalid credentials" for wrong email/password if preventing enumeration.

---

## Service behaviour

1. Find User by email — include passwordHash (select +passwordHash if schema hides)  
2. If !user → 401  
3. `bcrypt.compare(password, user.passwordHash)` — if false → 401  
4. If !user.isVerified → 403 `{ message: "Email not verified" }`  
5. Load Profile for displayName  
6. Call token issuance (6.14 + 6.15)  
7. Return tokens + safe user object
