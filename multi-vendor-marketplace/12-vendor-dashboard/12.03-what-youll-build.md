# 12.3 — What you'll build

By Chapter 12 gate, vendor manages products **end-to-end in browser** — list, create, edit, publish — with API errors surfaced clearly.

This chapter does not add server routes. Every screen **consumes** Chapter 11's API through new client helpers and pages. If curl cannot create a product, the browser form will not either — verify the API gate before building UI.

---

## The user journey you are implementing

```
Login as vendor
    → store gate passes (Ch 10)
    → /vendor/products (table — may be empty)
    → "Add product" → form → POST → redirect back to table
    → click row → edit form → PATCH price/stock
    → toggle Publish → PATCH isPublished
    → row shows "Published" in table
```

That journey is the Week 2 demo minus photos. Chapter 13 adds the image step on the edit page.

## API helpers (spec)

| Function | HTTP |
|---|---|
| `listVendorProducts()` | GET /vendor/products |
| `createProduct(data)` | POST /products |
| `updateProduct(id, data)` | PATCH /products/:id |
| `deleteProduct(id)` | DELETE /products/:id — optional UI |

Match auth pattern from `stores.js`.

---

## Chapter gate

Vendor manages products entirely in browser — create draft, edit, publish visible in table.

---

## UI expectations (not pixel-perfect)

| Screen | Minimum viable |
|---|---|
| ProductList | Table with name, price, unit, stock, published badge; link to edit |
| CreateProduct | Controlled form; submit disabled while loading; show API errors |
| EditProduct | Pre-filled from GET or list cache; publish toggle; save button |
| Navigation | "Products" link in vendor layout nav |

Editorial style from your design language is welcome — but **wiring** beats polish. A plain table that works beats a styled grid that doesn't fetch.
