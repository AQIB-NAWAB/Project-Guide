# 13.12 — Verify image flow

## Why this step matters

Upload bugs hide in three places — multer, Cloudinary, and React file state. End-to-end proof: pick image → see thumbnail → confirm MongoDB has URL only → replace photo → old asset gone. This is the technical gate before recap and the Week 2→3 bridge.

---

**Where the project is now:** Upload API and UI exist; individual pieces tested in isolation.

**After this step:** Full vendor photo flow demonstrated — upload, MongoDB URL only, thumbnail in dashboard, replace works. Ready for Week 2 demo statement in 13.13.

Run with two terminals (server + client). Use **your** product name in notes.

## MongoDB document audit

In Compass, inspect one product with image:

**Must have:**

- `imageUrl`: `https://res.cloudinary.com/...`  
- `imagePublicId`: non-empty string (if you implemented delete)

**Must not have:**

- `imageData`, `base64`, `Buffer`, large string fields  
- Document size >> few KB for metadata fields

Paste redacted doc shape into learning log — not API secrets.

---

## curl regression

Re-run upload curl from 13.9 after UI works — proves client is not doing something special.

Verify `Content-Type` of response is `application/json` — not multipart.

---

## Performance sanity

Vendor product list with 10 products with images:

- API response should be **JSON kilobytes** — not megabytes  
- Network tab: list request does not download image bytes — separate img requests to Cloudinary  

If list response is huge, you embedded images wrong — fix before gate.
