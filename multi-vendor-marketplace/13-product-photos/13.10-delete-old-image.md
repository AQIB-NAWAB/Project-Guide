# 13.10 — Delete old image

## Why this step matters

Replacing a photo without deleting the old Cloudinary asset leaks storage cost and clutters your account. `imagePublicId` exists for lifecycle management — remove the previous asset when a vendor uploads a new one.

---

**Where the project is now:** Upload route saves new URL; replacing a photo leaves previous Cloudinary asset orphaned.

**After this step:** When product already has `imagePublicId`, successful new upload deletes the old asset before or after DB update — document your order.

---

## Replace flow contract

| Step | Action |
|---|---|
| 1 | Load product; verify ownership |
| 2 | Capture `previousPublicId = product.imagePublicId` (if any) |
| 3 | Upload new image to Cloudinary |
| 4 | Update product with new `imageUrl` + `imagePublicId` |
| 5 | If `previousPublicId` exists → `deleteProductImage(previousPublicId)` |
| 6 | Return updated product |

**Order choice:**

| Order | Risk |
|---|---|
| Upload → DB update → delete old | Old URL dead if delete fails — acceptable; log orphan |
| Delete old → upload new | Product briefly has no image if upload fails — worse UX |

**Recommended:** upload → save new URL → delete old. Gate expects old asset removed when replace succeeds.

---

## Delete helper behaviour

| Case | Behaviour |
|---|---|
| `publicId` null/undefined | no-op |
| Cloudinary destroy 404 | log; continue — already gone |
| Cloudinary destroy network error | log; new URL still saved — note in learning log |
| Delete product (Ch 11) | **Stretch:** destroy image in delete handler |

---

## Service layer location

Logic belongs in **`products.service.js`** e.g. `replaceProductImage(productId, userId, file)`:

- Keeps route thin  
- Reuses ownership assertion  
- Single place for upload + delete + DB update  

Route handler: parse file → call service → map errors to HTTP.

---

## MongoDB consistency

After replace, document must have:

- `imageUrl` pointing to **new** Cloudinary URL  
- `imagePublicId` matching new asset  
- No field containing old URL  

Compass verification: upload twice on same product — only latest URL on doc.

---

## Manual verification

1. Upload image A — note URL in list  
2. Upload image B on same product  
3. URL in API response changed  
4. Image A URL may 404 in browser (destroyed) — or absent from Cloudinary dashboard  
5. Image B displays in vendor list  

---

## Product delete stretch

When vendor deletes product (`DELETE /products/:id`):

- Optional gate+: call `deleteProductImage` if `imagePublicId` set  
- Document in README if implemented  

Not required for Chapter 13 minimum gate — replace flow is required.

## ✅ Do / ❌ Don't

**✅ Do:** Store `imagePublicId` on first upload — delete impossible from URL alone reliably.

**✅ Do:** Log delete failures with public_id for ops cleanup.

**❌ Don't:** Delete old asset before new upload succeeds.

**❌ Don't:** Leave multer temp files on disk if you switch to disk storage — use memory + stream.
