You entered Chapter 13 with text-only products. You leave with **Cloudinary-backed photos**, **URL-only MongoDB documents**, and **vendor dashboard thumbnails** — the last piece of the Week 2 vendor experience.

---

## What you built — narrative recap

**Object storage discipline:** Images in Cloudinary; MongoDB stores `imageUrl` (+ `imagePublicId` for lifecycle) — no base64 trap.

**Server config:** Cloudinary credentials in `server/.env`; validated at boot; never in client.

**Upload pipeline:** multer parses multipart → ownership check → Cloudinary upload → product update.

**Replace flow:** New upload removes previous Cloudinary asset — storage does not grow unbounded on edits.

**Vendor UI:** File input on edit (or chosen flow); thumbnails in product list.

## Decisions you can defend in a viva

1. **Why Cloudinary instead of MongoDB for bytes?** — Document size, query speed, CDN.  
2. **Why server upload vs client direct?** — Secret protection; ownership enforcement.  
3. **Why separate POST for image vs PATCH JSON?** — multer + JSON don't mix cleanly.  
4. **What is `imagePublicId` for?** — Delete/replace without parsing URLs.  
5. **What if Cloudinary is down?** — Upload fails 502; existing URLs still serve from CDN until expiry.
