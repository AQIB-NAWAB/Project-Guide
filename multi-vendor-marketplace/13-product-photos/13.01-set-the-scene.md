# 13.1 — Set the scene

**Where the project is now:** Chapters 11–12 gave vendors a product CRUD API and dashboard. Products have names, prices, units, stock — but no photos. A grocery marketplace without images feels like a spreadsheet. Customers need to see what they're buying; vendors need their shelves to look real.

**Where this chapter goes:** Product images live in **Cloudinary** (object storage). MongoDB stores only the **URL string** on the product document — never binary data, never base64 blobs.

This is a **build chapter**. Every sub-chapter from 13.8 onward adds upload infrastructure to your repo. The gate is **runnable proof**: vendor uploads a photo, URL persists in MongoDB, image renders in the dashboard list.

## In plain English

| Term | Simple meaning |
|---|---|
| **Object storage** | Service that holds files (images) — Cloudinary, S3, etc. |
| **`imageUrl`** | Short string on the product document pointing to the hosted image |
| **Multipart upload** | HTTP request carrying a file — not JSON |
| **multer** | Express middleware that parses file uploads |
| **`imagePublicId`** | Cloudinary's id for delete/replace — not the same as MongoDB `_id` |
| **CDN** | Content delivery network — serves images fast worldwide |
| **Base64 trap** | Storing image data inside MongoDB documents — avoid this |

## Common beginner questions

**Q: Why not store images in MongoDB?**  
Documents balloon, queries slow, backups hurt. Store a URL string; let Cloudinary hold bytes. Sub-chapter 13.2 explains the cost in detail.

**Q: Why upload through the server instead of directly from the browser?**  
Your Cloudinary **secret** must never ship in the React bundle. Server upload also reuses ownership checks from Chapter 8.

**Q: Can I skip photos and still pass Week 2?**  
Photos are part of the Week 2 demo — *"vendor adds a product with a photo"*. Plan for Cloudinary setup in 13.8.

**Q: One image enough?**  
Yes for MVP. Multiple images per product is out of scope.

**Q: What comes after the checklist?**  
**13.14** bridges to Week 3 (customer experience). **13.15** is the gate. Chapter 14 builds the public catalogue API — buyer side, not vendor.

## The gap between "text catalogue" and "shop window"

Right now your vendor dashboard can answer:

- What products does this store sell?
- What is the price per kilogram?

It cannot answer:

- What does the produce look like?
- Where do image bytes live without bloating MongoDB?
- What happens when a vendor replaces a photo?

Those are **object storage and upload pipeline** problems — not CRUD problems. Chapter 12 built forms; **Chapter 13 wires the camera**.

## What "product photos" means in production terms

Professional marketplaces treat images as **assets in object storage**, referenced by URL in the database:

| Concern | Where it lives | Why |
|---|---|---|
| Image bytes | Cloudinary (or S3, etc.) | CDN delivery; resize/transform; not your DB |
| Public URL | `product.imageUrl` in MongoDB | Small string; queryable; cacheable |
| Upload credentials | `server/.env` — Cloudinary API secret | Server-only; never in React bundle |
| Multipart file | Request body → multer → stream to Cloudinary | Browser sends file; server proxies upload |

**Never store images in MongoDB documents** — next sub-chapter explains the cost.
