# 13.2 — The base64 trap

A tempting shortcut: read the file in the browser, base64-encode it, save the string in `product.imageData`. It works in a demo. It fails in production — sometimes within days when document size limits and query latency bite.

This sub-chapter explains the **cost** of storing images in MongoDB. You will **not** build this pattern — you will recognize it in code review and reject it.

---

## What base64 in MongoDB looks like

```javascript
// ❌ Do not ship this pattern
{
  name: "Organic Tomatoes",
  price: 3.50,
  imageData: "data:image/jpeg;base64,/9j/4AAQSkZJRgABAQEAYABgAAD..." // megabytes
}
```

**Problems:**

| Problem | When it hurts |
|---|---|
| Document bloat | 16 MB BSON limit per document; one high-res photo can approach it |
| Slow queries | Every `GET /vendor/products` transfers image bytes — even for a table that needs thumbnails |
| Index waste | MongoDB indexes metadata, not the blob — but working set memory still loads full docs |
| Backup/replication cost | Every replica stores every byte; backups grow with catalogue |
| No CDN | Browser cannot cache a URL; must re-fetch entire product doc |

A product list of 50 items with embedded 500 KB images = **25 MB per API response**. Your vendor dashboard would crawl.

---

## The right mental model

```
Browser file  →  Server (multer)  →  Cloudinary  →  HTTPS URL
                                                      ↓
                                              product.imageUrl (string ~200 chars)
```

MongoDB stores **pointers**, not **pixels**.

| Layer | Holds |
|---|---|
| Cloudinary | Bytes, transforms, CDN |
| MongoDB | `imageUrl: "https://res.cloudinary.com/..."` |
| React | `<img src={product.imageUrl} />` |

---

## "But Cloudinary costs money"

Free tier covers course scope. Local disk storage (`/uploads/foo.jpg`) works for learning but breaks on deploy — ephemeral filesystems on PaaS lose files on restart. Object storage is the industry default; Cloudinary is mandated for this course.

If you ever self-host files:

- Still store **URL path** in MongoDB  
- Still **never** embed base64 in documents  
- Serve via static middleware or nginx — not BSON

---

## GridFS — when blobs belong in MongoDB

MongoDB **GridFS** splits large files across chunks — valid for some domains (PDF archives, medical imaging metadata). For **product thumbnails in a marketplace**:

- Cloudinary/S3 is simpler  
- CDN and resize are built-in  
- Course gate expects Cloudinary

Mention GridFS in viva as "exists but wrong tool for grocery photos."

---

## ✅ Do / ❌ Don't

**✅ Do:** Store a single HTTPS URL string on the product document.

**✅ Do:** Validate URL format optionally — must be your Cloudinary domain or empty.

**❌ Don't:** Accept base64 in JSON body for product create — reject or ignore.

**❌ Don't:** Store multer temp file path permanently — upload to Cloudinary, delete temp file.

**❌ Don't:** Put Cloudinary API secret in the client — uploads go through your API.

---

## Mentor interview angle

*"Where do product images live in your marketplace?"*

Strong answer: object storage (Cloudinary), URL in MongoDB, server-side upload with secret, CDN for delivery, document stays small.

Weak answer: "In the database" or "we use multer to save locally" with no deploy story.
