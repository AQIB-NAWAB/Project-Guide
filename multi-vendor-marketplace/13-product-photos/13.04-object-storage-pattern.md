# 13.4 — Object storage pattern

Every marketplace that serves images at scale follows the same pattern: **bytes in object storage, pointer in the database, CDN in front**. MongoDB, PostgreSQL, and Redis are wrong places for JPEG data — they are right places for URLs.

This sub-chapter names the pattern you implement with Cloudinary in Chapter 13.

---

## The three-layer model

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Client    │     │   API       │     │  Storage    │
│  (browser)  │────▶│  (Express)  │────▶│ (Cloudinary)│
└─────────────┘     └──────┬──────┘     └──────┬──────┘
                           │                    │
                           ▼                    │
                    ┌─────────────┐             │
                    │  MongoDB    │◀── URL only │
                    │  Product    │             │
                    └─────────────┘             │
                           ▲                    │
                           └──── img src ───────┘
                                 (CDN URL)
```

| Layer | Responsibility |
|---|---|
| Client | Display `<img src={url}>`; send file to API on upload |
| API | Authenticate; validate file; stream to storage; save URL |
| Storage | Persist bytes; serve via HTTPS; optional transforms |
| MongoDB | Store URL + optional public_id for lifecycle |

---

## Upload flow (server-mediated)

**Why not upload directly from browser to Cloudinary?**

Unsigned uploads expose upload presets. Signed uploads still need secrets generated server-side. **Server-mediated upload** keeps API secret on server and lets you enforce ownership before bytes leave your control.

| Step | Actor | Action |
|---|---|---|
| 1 | Vendor | Selects file in dashboard |
| 2 | Client | POST multipart to `/products/:id/image` with JWT |
| 3 | Server | multer parses file; checks vendor owns product |
| 4 | Server | Cloudinary SDK upload from buffer/stream |
| 5 | Server | PATCH product with returned `secure_url` |
| 6 | Client | Renders new thumbnail |

Direct-to-S3 with presigned URLs is a valid production pattern — out of scope for this course; same DB contract (URL only).

---

## Read flow (public CDN)

After upload, **every** consumer — vendor dashboard, customer shop (Chapter 15) — reads the same URL. No API round-trip for image bytes on browse; browser fetches from Cloudinary CDN.

| Concern | Where handled |
|---|---|
| Bandwidth on your API | Avoided — images not proxied through Express in normal browse |
| Resize/thumbnail | Cloudinary URL transforms (optional stretch) |
| HTTPS | Cloudinary default |

---

## Lifecycle: replace and delete

| Event | Storage | MongoDB |
|---|---|---|
| First upload | New object created | `imageUrl` set |
| Replace | New object; **delete old** by public_id | URL updated |
| Delete product (Ch 11) | **Stretch:** delete Cloudinary asset | Product doc removed |

Orphan files in Cloudinary cost money over time — storing `imagePublicId` enables cleanup.

## Comparison table

| Approach | MongoDB size | CDN | Course gate |
|---|---|---|---|
| Base64 in document | Huge | No | ❌ |
| Local `/uploads` folder | Small URL | No | ❌ for deploy story |
| Cloudinary / S3 | Small URL | Yes | ✅ |
| GridFS | Chunked in DB | No | ❌ for this domain |

## Folder convention in Cloudinary

Use a predictable `folder` or `public_id` prefix:

```
freshmarket/products/{productId}-{timestamp}
```

**Benefits:** easier debugging in Cloudinary dashboard; bulk cleanup by prefix; no collision across vendors.

**You choose** exact naming — document in README.

---

## ✅ Do / ❌ Don't

**✅ Do:** Use `secure_url` from Cloudinary response (HTTPS).

**✅ Do:** Fail upload if ownership check fails **before** calling Cloudinary — saves quota and audit noise.

**❌ Don't:** Store both URL and base64 "for backup."

**❌ Don't:** Proxy image bytes through `GET /products/:id/image/file` unless you have a specific reason — CDN exists for reads.

## Connection to product schema (Chapter 5)

Your Product model should already allow:

| Field | Type | Notes |
|---|---|---|
| `imageUrl` | String | optional; empty until upload |
| `imagePublicId` | String | optional; for delete API |

If missing, add in 13.8 before upload route — migration is one field, no data backfill required.
