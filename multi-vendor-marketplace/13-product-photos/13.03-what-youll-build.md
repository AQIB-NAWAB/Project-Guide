# 13.3 — What you'll build

By the end of Chapter 13, vendors upload product photos through your API. Images live in **Cloudinary**; MongoDB stores **`imageUrl`** only. The vendor dashboard shows thumbnails in the product list and on create/edit forms.

This sub-chapter is the **blueprint** for sub-chapters 13.8–13.12. No code solutions here — contracts and file locations only.

**Public cloud name** may appear in image URLs — that is expected. Secret stays server-only.

## Upload route contract

| Field | Value |
|---|---|
| Method | POST |
| Path | `/api/products/:id/image` (or `/products/:id/image` — match your mount) |
| Auth | Required — vendor JWT |
| Authorization | Product must belong to vendor's store — same ownership as PATCH |
| Content-Type | `multipart/form-data` |
| Body field | `image` — single file (field name your choice; document it) |

**Success — 200 or 201:**

```json
{
  "id": "...",
  "name": "Organic Tomatoes",
  "imageUrl": "https://res.cloudinary.com/your-cloud/image/upload/v123/freshmarket/products/abc.jpg",
  "imagePublicId": "freshmarket/products/abc"
}
```

| Error | Status | When |
|---|---|---|
| No file | 400 | Missing `image` field |
| Wrong type | 400 | Not image/jpeg, image/png, image/webp — define allowed set |
| Too large | 400 | e.g. max 5 MB — document limit |
| Not owner | 403 | Vendor B uploads to Vendor A's product |
| Not found | 404 | Invalid product id |
| Cloudinary failure | 502 or 500 | Upload API error — log, don't leak secret |

**You implement** status codes and error shape — match existing API error pattern from Chapter 11.

## Replace / delete old image contract

When vendor uploads a **new** image for a product that already has `imagePublicId`:

1. Upload new file to Cloudinary  
2. Update product `imageUrl` + `imagePublicId`  
3. **Delete** previous asset from Cloudinary by public_id  
4. If delete fails, log warning — new URL still saved (orphan cleanup is ops, not gate)

If first upload (no prior image): skip delete step.

## Multer spec

| Setting | Guidance |
|---|---|
| Storage | `memoryStorage()` for small images — buffer passed to Cloudinary SDK |
| Limits | `fileSize` max 5 * 1024 * 1024 (5 MB) |
| File filter | Reject non-image MIME types |
| Single file | `upload.single('image')` |

Temp files on disk require cleanup — memory storage avoids leftover files if you stream buffer to Cloudinary.

---

## Client upload contract

| Step | Behaviour |
|---|---|
| User selects file | Validate size/type client-side (UX only — server enforces) |
| Submit | `FormData` with `image` field; **do not** set Content-Type header manually (browser sets boundary) |
| Auth | Attach Bearer token same as other API calls |
| Success | Update local product state with returned `imageUrl`; show preview |
| Error | Map 400/403 to inline message |

Upload can be **separate step** after product create (POST product first, then POST image) or on edit page only — either satisfies gate if flow is documented in README.

## Verification story (13.12)

1. Vendor logs in → edits product → selects JPEG → upload succeeds  
2. MongoDB product doc shows `imageUrl` string only — no binary field  
3. Product list shows thumbnail from Cloudinary URL  
4. Replace image → old URL no longer on product; old asset removed from Cloudinary (verify in dashboard or log)  
5. Vendor B cannot upload to Vendor A's product — 403

## ✅ Do / ❌ Don't

**✅ Do:** Update `.env.example` when adding Cloudinary vars — same commit as config change.

**✅ Do:** Store `imagePublicId` if you need reliable delete — URL alone is harder to parse.

**❌ Don't:** Save file to `server/uploads/` as permanent storage for gate — use Cloudinary.

**❌ Don't:** Return Cloudinary API secret in any API response.
