# 99.2 — Document it

Hiring managers and mentors cannot read 22 chapters. They read **three docs and watch one video**. This sub-chapter specifies what to commit under `docs/`.

Good documentation is not busywork — it is how you prove you **understood** what you built, not just followed steps.

---

## Why docs matter for beginners

| Without docs | With docs |
|---|---|
| "I built a MERN app" | "Split checkout with price snapshots — here's the ERD" |
| Mentor guesses your architecture | `architecture.md` shows API, worker, Redis roles |
| Interview vague on security | `AUTHORIZATION.md` + CHALLENGES entry on IDOR test |
| Live URL down — nothing left | README + video still show intent |

You already wrote learning logs per chapter. `docs/` is the **public, polished** version for portfolio and viva.

## Required documentation

### `docs/erd.md`

Final ERD aligned with Mongoose models — not draft from Ch 3.

- User, Store, Product, CartItem, Order
- Ownership arrows (vendor → store → product)
- Embedded `orderItems` with snapshot fields **highlighted**

Tool: draw.io, Mermaid in markdown, or exported image.

**Beginner tip:** Label snapshot fields on Order — `nameAtPurchase`, `priceAtPurchase` — and write one sentence: *"Receipt lines freeze at purchase."*

---

### `docs/architecture.md`

One diagram + prose — **include all runtime processes**:

```
Browser (React/Vite, HTTPS)
  → Express API (HTTPS)
    → MongoDB Atlas
    → Redis
        ├── catalogue cache (GET /products)     ← Chapter 20
        └── BullMQ job queues                   ← Chapter 21
    → Worker process (npm run worker)
        ├── order.created → vendor + customer notify
        ├── order.statusUpdated → customer notify
        └── product.updated → cache bust (optional)
    → Cloudinary (images — URL only in MongoDB)
```

Sections to write (plain language, 1–2 paragraphs each):

| Section | Cover |
|---|---|
| **Folder map** | `server/src/modules/` — what each module owns |
| **Auth** | Access token in memory + refresh httpOnly cookie (Ch 7) |
| **Checkout** | One button → many orders grouped by `storeId` + snapshots |
| **Cache** | Cache-aside on catalogue; invalidate on product write |
| **Async** | Why checkout enqueues — sync trap; API vs worker |
| **Deploy** | Three processes: API, worker, static client |

### `docs/CHALLENGES.md`

Honest engineering diary — **3–5 entries minimum**:

| Entry | Example topic |
|---|---|
| 1 | IDOR test — vendor B blocked on vendor A's store |
| 2 | Snapshot immutability — price change did not affect old order |
| 3 | Redis down fallback — catalogue still loaded |
| 4 | CORS on production — fixed `CLIENT_ORIGIN` |
| 5 | Worker not running — 201 but no `[NOTIFY]` logs |
| 6 | Stale cache after PATCH — forgot invalidation |

Format per entry:

- **Problem** — what broke (specific symptom)
- **Cause** — root cause in one paragraph
- **Fix** — what you changed (file or config name)
- **Lesson** — what you'd do earlier next time

Not resume fluff — specific to **your** repo.

## Common beginner questions

**Q: Can I use AI to write architecture.md?**  
You may draft with AI; you must **verify every box** matches your repo. Viva will expose diagrams that lie.

**Q: My ERD changed since Ch 5. Is that bad?**  
Normal. Update `docs/erd.md` to match reality and note drift in CHALLENGES if significant.

**Q: How long should CHALLENGES entries be?**  
Half page each is plenty. Specific beats long.

---

## ✅ Do / ❌ Don't

**✅ Do:** Keep docs in repo — not only in Notion.

**✅ Do:** Update ERD if schema drifted during build.

**✅ Do:** Include worker in architecture diagram — omitting it is a common beginner mistake.

**❌ Don't:** Copy another student's architecture diagram.

**❌ Don't:** Empty CHALLENGES.md — mentors read this first for signal.
