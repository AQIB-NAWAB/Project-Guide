# 21.12 — Order created processor

**Why this step matters:** Enqueue without a processor is a message nobody reads. `[NOTIFY]` logs here are the gate proof that multi-vendor checkout triggers one vendor line per store plus one customer summary.

**After this step:** Worker handles `order.created` jobs — logs mock vendor notifications (per order/store) and one mock customer confirmation.

---

## Processor location — spec

```
server/src/modules/jobs/orderCreated.job.js
  export async function handleOrderCreated(data) { ... }
  export async function enqueueOrderCreated(data) { ... }  // if not only in orders.service
```

Import `handleOrderCreated` in `worker.js` router.

---

## handleOrderCreated(data) — behaviour contract

**Input:** Payload from 21.11 — at minimum `orderIds`, `customerId`.

**Steps (you implement):**

1. Load orders from MongoDB by `orderIds` — populate `storeId` name optional.
2. For **each order** (one per vendor):
   - Log mock vendor email:
     ```
     [NOTIFY][vendor] store=<storeName> orderId=<id> lines=<count> total=<total>
     ```
3. Log mock customer email **once** for batch:
   ```
   [NOTIFY][customer] customerId=<id> orders=<count> orderIds=[...]
   ```

**No SendGrid** — structured `console.log` or `logger.info` satisfies gate.

---

## Multi-vendor proof

Two-store checkout → processor logs:

| Log line | Count |
|---|---|
| `[NOTIFY][vendor]` | 2 — one per store |
| `[NOTIFY][customer]` | 1 — summary of both orders |

Demo script: point mentor at worker terminal **after** browser shows confirmation.

---

## Data loading options

| Approach | Tradeoff |
|---|---|
| Log ids only from payload | Fast; less demo-friendly |
| Fetch Order + Store from DB | Realistic; worker needs MongoDB |
| Fetch User email for customer line | Shows email in log — redact in learning log |

**Gate:** At least store name or storeId per vendor line.

---

## Processor return value

Return `undefined` or `{ notified: orderIds.length }` on success.

**Throw** only for unexpected errors (DB down) — triggers retry per 21.14.

**Do not throw** for "email would fail" — there is no real email.

---

## Idempotency note

If same job runs twice (retry), logs appear twice — acceptable for mock.

With `jobId` dedup on enqueue (21.14), duplicate enqueue suppressed — processor may still retry on crash mid-run. Document in README.

3. Customer checkout with two-store cart
4. Watch worker terminal within 5 seconds

**Expected order:**

- API returns 201 **first**
- Worker logs appear **after**

## ✅ Do / ❌ Don't

**✅ Do:** Use `[NOTIFY]` prefix — grep-friendly for mentors.

**✅ Do:** One customer log summarizing all orders from batch.

**❌ Don't:** Call HTTP endpoints from worker to "simulate" email.

**❌ Don't:** Block processor with `setTimeout` simulating SMTP — instant log OK.
