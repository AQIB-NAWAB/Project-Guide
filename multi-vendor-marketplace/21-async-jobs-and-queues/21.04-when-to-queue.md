# 21.4 — When to queue

Not every server action belongs in a worker. Use this table when deciding sync vs async for FreshMarket features.

---

## Decision matrix

| Work | Sync in HTTP handler? | Queue instead? | FreshMarket example |
|---|---|---|---|
| Validate input | ✓ Always | ✗ | Cart empty → 400 |
| Authorize user | ✓ Always | ✗ | Vendor owns order's store |
| DB writes user must see immediately | ✓ | ✗ | Create orders, decrement stock |
| Third-party API (email, SMS) | ✗ | ✓ | Order confirmation |
| Retryable side effects | ✗ | ✓ | Notify vendor after timeout |
| Heavy CPU (image resize) | ✗ | ✓ | Already sync in Ch 13 — could queue later |
| Cache invalidation | ✓ or queue | Either | Ch 20 sync OK; 21.16 async optional |
| Read-only aggregation | ✓ | ✗ | GET /products — use cache not queue |

**Rule of thumb:** If the user needs the result **now** to continue the flow → sync. If failure can be **retried later** without blocking UX → queue.

---

## Checkout path — annotated

```
POST /checkout
├── SYNC: load cart, validate stock
├── SYNC: insert orders (transaction)
├── SYNC: decrement stock
├── SYNC: clear cart
├── ASYNC: order.created job(s)
└── SYNC: return 201 { orders }
```

Stock and orders are **consistency-critical**. Email is **eventual**.

---

## Status update path — annotated

```
PATCH /vendor/orders/:id
├── SYNC: ownership check
├── SYNC: validate transition
├── SYNC: save new status
├── ASYNC: order.statusUpdated job
└── SYNC: return 200 { order }
```

Customer sees new status on next GET — notification is supplementary.

---

## When sync side effects are acceptable

| Case | Why sync is OK |
|---|---|
| Cache `DEL` one key pattern | Milliseconds; same process has Redis client |
| Audit log to MongoDB | Same DB; transactional if needed |
| In-process event emitter | No separate worker — **not** this chapter's pattern |

Course uses **BullMQ worker process** for gate — not `EventEmitter` in `server.js`.

---

## Anti-patterns

| Pattern | Problem |
|---|---|
| Queue the stock decrement | User sees 201 before stock confirmed — oversell risk |
| Queue order creation | Cart cleared before order exists — data loss |
| Sync email + async order | Backwards — email blocks, order async |
| One giant job for entire checkout | Can't retry notify without re-running stock logic |

Split jobs by **bounded context**: commerce commit in API; notifications in worker.

---

## Idempotency preview

Client double-clicks checkout → two orders (Ch 18 noted). Queue does not fix that.

For notifications: use **deterministic `jobId`** per event — e.g. `order-created:<checkoutBatchId>` — so duplicate enqueue does not double-email. Details in 21.14.

---

## ✅ Do / ❌ Don't

**✅ Do:** Queue work that talks to unreliable external services.

**✅ Do:** Keep financial/inventory mutations in the request transaction.

**❌ Don't:** Queue anything required for the 201 response body.

**❌ Don't:** Assume "async = fire and forget" — monitor worker logs and failed jobs.
