# 21.2 — The sync trap

You add `sendOrderConfirmationEmail(customer)` inside the checkout handler. Demo day: Mailgun times out after 30 seconds. Checkout returns **500** even though orders were created and stock decremented. Customer sees error; database shows success. Support nightmare.

This sub-chapter names the trap **before** you wire BullMQ in 21.9–21.12.

---

## The bug in one sentence

**Side effects that can fail or slow down should not block the HTTP response for work already committed.**

---

## How teams accidentally build this

| Step | What goes wrong |
|---|---|
| Checkout succeeds in MongoDB | Orders exist |
| Handler calls email API inline | Network latency adds 200–2000ms per recipient |
| Email provider 503 | Entire request fails — client retries → duplicate risk |
| Vendor notification throws | Customer never gets 201 despite valid order |
| Logging PII in catch block | GDPR leak in server logs |

MongoDB transaction does not roll back email attempts. **Request handler owns too much.**

---

## Multi-vendor makes it worse

One checkout → **N orders** → **N vendor notifications** + **1 customer notification**.

| Cart shape | Inline notifications |
|---|---|
| 1 store | 2 sequential calls |
| 3 stores | 4 sequential calls |
| Peak hour | Checkout p99 spikes |

Chapter 18 already did the hard part synchronously — stock and order creation **must** be consistent. Notifications **do not** need same atomicity.

---

## What belongs sync vs async (preview)

| Operation | Sync in request? | Why |
|---|---|---|
| Validate cart | ✓ | Client needs immediate 400 |
| Create orders + stock | ✓ | Commerce correctness |
| Clear cart | ✓ | Prevent double checkout |
| Email / push notify | ✗ | Best-effort; retryable |
| Cache invalidation | Either | Sync OK for gate; async cleaner at scale |

Full decision table in 21.4.

---

## Failure modes without a queue

| Symptom | Root cause |
|---|---|
| Checkout slow but DB fast | SMTP in hot path |
| 500 after order created | Uncaught error in notify helper |
| Duplicate emails on retry | Client resubmits; no idempotency |
| No email but 201 | Swallowed errors — silent failure |

Queue + worker + retries address the last three in 21.14.

---

## UI consequence (Week 4 demo)

Checkout confirmation page shows **immediately** after 201 — "We're notifying vendors" is honest. Customer does not wait for worker logs.

Browser never calls notification endpoints directly — server enqueues after commit.

---

## ✅ Do / ❌ Don't

**✅ Do:** Return 201 as soon as orders + cart clear succeed.

**✅ Do:** Enqueue jobs **after** DB writes succeed — not before.

**❌ Don't:** Put `await sendEmail()` in checkout handler for gate.

**❌ Don't:** Fail checkout if mock notification throws — that's the trap.

---

## Mentor interview angle

*"Why not send order emails synchronously?"*

Strong answer: decouple user-facing latency from third-party SLA; retries in worker; API stays fast and predictable.

Weak answer: "Email is fast" — describes demo-day timeout.
