# 21.1 — Set the scene

**Where the project is now:** Week 4 polish underway — catalogue cached in Redis (Chapter 20), commerce flow complete in browser (Chapters 18–19). Checkout creates orders, decrements stock, clears cart. **No notifications** run after checkout or status changes.

**Where this chapter goes:** **Async job queue** with **BullMQ + Redis** — checkout and status PATCH return fast; **workers** send vendor and customer notifications in the background (mock email via `console.log` for gate).

Same Redis instance from Chapter 20 — different keys than cache. Separate **worker process** (`npm run worker`).

---

## Bridge from last chapter

Chapter 20 made browsing **fast**. Chapter 21 makes checkout **reliable under side effects** — the emails, SMS, and push notifications every real marketplace sends **after** an order exists.

Right now your checkout handler does the right **synchronous** work:

1. Validate cart  
2. Create orders + decrement stock  
3. Clear cart  
4. Return 201  

That is correct. The trap is step 5 that teams add later: *"Oh, let's also send confirmation emails before we return."* Chapter 21 teaches you to **stop at step 4** and hand step 5 to a **worker** that runs in a separate process.

| Chapter 20 solved | Chapter 21 solves |
|---|---|
| Repeated catalogue reads are slow | Checkout blocked by slow/unreliable notifications |
| Stale prices from cache | 500 errors when email provider times out **after** order already created |
| Search without index | Customer waits 2 seconds for SMTP while staring at a spinner |

**Why after checkout works, not during:** Order creation and stock must be **immediate and consistent** — the customer needs a 201 with order IDs **now**. Email can arrive three seconds later. If email fails, you **retry in the worker** without undoing a valid order.

## In plain English

| Term | Simple meaning |
|---|---|
| **Async (asynchronous)** | "Do this later" — the API responds first, background work follows |
| **Job queue** | A to-do list stored in Redis — jobs wait in line until a worker picks them up |
| **Enqueue** | Add a task to the to-do list — fast, milliseconds |
| **Worker** | A separate Node process that reads the to-do list and does the slow work |
| **BullMQ** | A library that manages the to-do list — retries, scheduling, job metadata |
| **Processor** | The function the worker runs for each job — e.g. "send mock vendor email" |
| **Side effect** | Something that happens because of an action but is not the main result — email after order is a side effect; the order itself is the main result |
| **Sync trap** | Putting slow side effects **inside** the checkout handler so the customer waits or the whole request fails |

---

## Think of it like a restaurant kitchen

**Synchronous (must happen before the waiter returns):** Take the order, charge the card, print the ticket, deduct ingredients from inventory. The customer cannot leave without knowing whether the order was accepted.

**Asynchronous (can happen after):** Send a text when the food is ready, email a receipt, notify the supplier. If the text fails, the meal still happened — you retry the text later.

Your checkout API is the **waiter**. The worker is the **back office** that sends texts. Chapter 21 wires the ticket to the back office **after** the waiter tells the customer "Order confirmed."

---

## The sync side-effect problem

```
POST /checkout
  → validate cart
  → create orders
  → decrement stock
  → clear cart
  → send email to customer        ← blocks response
  → send email to vendor A        ← blocks response
  → send email to vendor B        ← blocks response
  → return 201 (800ms later)
```

Customer waits for SMTP. One slow vendor notification fails the whole checkout. Chapter 21 moves notifications **off the request path**.

## Common beginner questions

**Q: What is a queue? I thought Redis was for cache.**  
Redis can hold many data types. Chapter 20 used it for **cached strings**. Chapter 21 uses it for **BullMQ job lists** — same Redis server, different key prefixes. Think of cache as a photograph; the queue is a **to-do inbox**.

**Q: Why a separate worker process? Why not just `setTimeout` in the API?**  
`setTimeout` dies when you restart the API, cannot retry reliably, and shares memory with HTTP traffic. A worker process **survives independently**, can retry failed jobs, and scales separately in production (Ch 22).

**Q: What if the worker is not running?**  
Checkout still returns 201 — orders exist. Jobs sit in Redis until you start the worker. That is **eventual** notification delivery — acceptable for email, unacceptable for stock decrement (which stays synchronous).

**Q: Do I need real SendGrid/Mailgun?**  
No. `console.log` with a `[NOTIFY]` prefix satisfies the gate. Swap for a real provider in portfolio — same queue shape.

**Q: Should checkout wait until the worker finishes?**  
**No.** Gate proof: 201 returns **before** worker logs appear. If you await the worker in the handler, you rebuilt the sync trap.

**Q: What jobs will I build?**  
`order.created` after checkout (notify each vendor + customer), `order.statusUpdated` after vendor PATCH (notify customer of new status). Optional: `product.updated` for async cache bust (21.16).

**Q: Two terminals again?**  
Yes. Terminal 1: `npm run dev` (API). Terminal 2: `npm run worker`. Same habit as client + server from Chapter 2 — now three processes if you count the React dev server.
