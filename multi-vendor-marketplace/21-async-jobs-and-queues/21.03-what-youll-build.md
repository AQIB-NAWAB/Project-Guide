# 21.3 — What you'll build

Blueprint for **BullMQ jobs** on the same Redis from Chapter 20. Read once; verify against repo after 21.15.

If queues are new to you: you are **not** changing what checkout returns — you are adding a **second process** that picks up notification tasks after the API says 201.

---

## In plain English

| Piece | Simple role |
|---|---|
| `queue.js` | API writes a to-do item to Redis |
| `worker.js` | Separate program that reads to-dos and runs them |
| `order.created` job | To-do: "tell vendors and customer about new orders" |
| `order.statusUpdated` job | To-do: "tell customer their order status changed" |

---

## Common beginner questions

**Q: How many new terminals?**  
Two for server-side async: API (`npm run dev`) + worker (`npm run worker`). Client dev server is a third if you are testing in browser simultaneously.

**Q: Will I change the React app?**  
No client changes required — async is invisible to the browser. Gate proof is server timing + worker logs.

**Q: What if I already added `sendEmail` in checkout?**  
Remove it before 21.11 — you are undoing the sync trap, not adding on top of it.

```
                                                │ poll

                                       │ email logs      │
                                       └─────────────────┘
```

Two terminals in dev: `npm run dev` + `npm run worker`.

## Job contracts

### `order.created`

**Enqueue when:** `POST /checkout` returns success — all orders inserted, cart cleared, stock decremented.

**Payload (illustrative — you define exact shape):**

```javascript
{
  orderIds: ['...', '...'],       // all orders from this checkout
  customerId: '...',
  customerEmail: '...',           // or fetch in worker from User
}
```

**Worker behaviour:**

- For each order: log mock vendor notification (`storeId`, `orderId`, line summary)
- Once: log mock customer confirmation (all `orderIds`, total count)
- **No** real SMTP required — structured `console.log` with clear prefix e.g. `[NOTIFY]`

**Enqueue from API:** non-blocking — `add()` without awaiting processor completion.

---

### `order.statusUpdated`

**Enqueue when:** `PATCH /vendor/orders/:orderId` succeeds — status persisted.

**Payload:**

```javascript
{
  orderId: '...',
  customerId: '...',
  previousStatus: 'pending',
  newStatus: 'packed',
  storeId: '...'
}
```

**Worker:** mock customer email — "Your order is now packed".

---

### `product.updated` (optional)

**Enqueue when:** Product create/update/delete succeeds (instead of or in addition to sync `invalidateCatalogueCache` from 20.11).

**Payload:** `{ productId, action: 'update' | 'create' | 'delete' }`

**Worker:** call same invalidation logic as Ch 20 — keeps PATCH response fast if invalidation was slow.

## HTTP behaviour contract (unchanged externally)

| Endpoint | Response | Change internally |
|---|---|---|
| POST `/checkout` | 201 `{ orders }` | + enqueue `order.created` after commit |
| PATCH `/vendor/orders/:id` | 200 order JSON | + enqueue `order.statusUpdated` after save |

**Clients see no new fields** — async is server-internal.
