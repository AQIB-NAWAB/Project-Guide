# 21.11 — Order created job

**Why this step matters:** This is where you escape the sync trap — checkout commits commerce state, then hands notifications to the queue. The enqueue call site order (after DB writes) is the most common bug in async checkout implementations.

**After this step:** `POST /checkout` enqueues one `order.created` job **after** orders persist, stock decrements, and cart clears — **before** returning 201.

---

## Enqueue call site — spec

**File:** `orders.service.js` — inside `checkout()` success path, after all synchronous commits.

**Order of operations (mandatory):**

```
1. Validate cart
2. Create orders + decrement stock + clear cart  (sync)
3. enqueueOrderCreated({ orderIds, customerId, ... })  (async)
4. return { orders }
```

**Never** enqueue before step 2 completes — worker must not notify for orders that failed to save.

---

## enqueueOrderCreated() — contract

| Parameter | Source |
|---|---|
| `orderIds` | `_id` of each created order from checkout batch |
| `customerId` | Auth user id |
| `customerEmail` | Optional — from User doc or omit and fetch in worker |

**Job name:** `JOB_NAMES.ORDER_CREATED` (`order.created`)

**Queue:** `getNotificationsQueue().add(...)`

**Await:** `await queue.add(...)` — enqueue is fast (Redis round-trip); still **do not** await worker completion.

---

## jobId deduplication — preview

Pass deterministic `jobId` in options — full spec in 21.14:

```javascript
// illustrative
jobId: `order-created:${orderIds.sort().join('-')}`
```

Prevents duplicate notify if handler accidentally called twice with same batch.

---

## Payload shape — illustrative

```javascript
{
  orderIds: ['674a...', '674b...'],
  customerId: '674c...',
  createdAt: '2026-06-09T12:00:00.000Z'  // optional metadata
}
```

Worker loads order details from MongoDB or logs ids only — **you implement** richness of mock email.

---

## HTTP response unchanged

**Success 201** — same body as Chapter 18:

```json
{
  "orders": [ { "_id": "...", "storeId": "...", "total": 305, ... } ]
}
```

No `notificationSent` flag — async is invisible to client.

---

## Error handling — spec

| Enqueue fails (Redis down) | Behaviour |
|---|---|
| Option A | Log error; still return 201 — orders valid, notify missed |
| Option B | Return 503 — strict notify requirement |

**Course gate:** Option A acceptable if documented — commerce beats email. Log clearly: `[queue] enqueue failed, order created`.

**Do not** roll back orders if enqueue fails.

## curl verification — enqueue only

With worker **stopped**:

```bash
curl -s -w "\nTIME:%{time_total}\n" -X POST http://localhost:5000/api/checkout \
  -H "Authorization: Bearer <customer-token>" \
  -H "Content-Type: application/json" \
  -d '{}'
```

**Expected:** 201 quickly. Redis has pending job — verify in 21.15 with `redis-cli` or start worker and see backlog process.

---

## ✅ Do / ❌ Don't

**✅ Do:** Enqueue once per checkout batch — not one job per order (unless you document why).

**✅ Do:** Include all `orderIds` from split checkout in one payload.

**❌ Don't:** `await sendEmail()` in checkout — sync trap from 21.2.

**❌ Don't:** Enqueue inside MongoDB transaction before commit — race with worker.
