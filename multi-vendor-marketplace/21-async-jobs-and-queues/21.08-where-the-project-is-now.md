# 21.8 — Where the project is now

**Why this step matters:** BullMQ wiring on top of a broken checkout or dead Redis wastes a day. This checkpoint confirms commerce and cache baselines before you add a second process to your dev workflow.

**Where the project is now:** Chapter 20 added Redis cache on catalogue reads. Checkout and order status PATCH work synchronously — any notification logic would block HTTP if added inline.

**After this chapter:** BullMQ queues on the same Redis; worker process handles `order.created` and `order.statusUpdated`; checkout returns before `[NOTIFY]` logs.

Continuity checkpoint before wiring BullMQ. Confirm Chapter 20 Redis and Chapter 18–19 commerce paths work.

## Verify Redis cache baseline (Ch 20)

```bash
curl -s "http://localhost:5000/api/products?page=1&limit=5"
```

Boot log should show Redis connected (or fallback warning). `REDIS_URL` in `server/.env`.

```bash
redis-cli -u "$REDIS_URL" PING
# expect PONG
```

## Verify checkout baseline (Ch 18)

```bash
curl -s -X POST http://localhost:5000/api/checkout \
  -H "Authorization: Bearer <customer-token>" \
  -H "Content-Type: application/json" \
  -d '{}'
```

**Expected:** 201 with `orders` array. Note response time — baseline for 21.15 async comparison.

---

## Verify status PATCH baseline (Ch 19)

```bash
curl -s -X PATCH "http://localhost:5000/api/vendor/orders/<orderId>" \
  -H "Authorization: Bearer <vendor-token>" \
  -H "Content-Type: application/json" \
  -d '{"status":"packed"}'
```

**Expected:** 200 with updated order. No notification logs yet — correct.
