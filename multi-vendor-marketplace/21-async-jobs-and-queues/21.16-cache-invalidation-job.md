# 21.16 — Cache invalidation job

**Why this step matters:** Optional but instructive — it connects Chapters 20 and 21 by moving slow cache bust off the vendor PATCH response. Even if you skip it, reading the tradeoff prepares you for "sync vs async invalidation" questions in viva.

**Where the project is now:** Async notify jobs verified (21.15). Catalogue cache invalidation may still run **inline** in product save from Ch 20.11.

**After this step (optional):** `product.updated` job busts cache in worker — PATCH returns before `[CACHE]` log.

**Optional stretch:** Move catalogue cache bust from synchronous product save (Chapter 20.11) to async `product.updated` job — ties queue chapter to cache chapter.

**Gate:** Completing 21.11–21.15 is sufficient to proceed. This sub-chapter is **recommended** if invalidation added measurable latency to PATCH.

---

## When async invalidation helps

| Symptom in Ch 20 | Async fix |
|---|---|
| `PATCH /products/:id` slow after adding `SCAN`+`DEL` | Return 200 after MongoDB; bust cache in worker |
| Large key sets | Worker can take longer without blocking vendor UI |

If invalidation is sub-10ms locally, sync invalidation from 20.11 remains fine — enqueue is architectural practice.

---

## Enqueue call site — spec

**File:** `products.service.js` — after successful create/update/delete.

**Replace or supplement** direct `invalidateCatalogueCache()` call:

```javascript
// Option A: async only
await enqueueProductUpdated({ productId, action: 'update' });

// Option B: sync + async redundant — don't do both
```

Pick **one** strategy — document in README.

---

## product.updated job — contract

| Field | Values |
|---|---|
| `productId` | MongoDB id |
| `action` | `create` \| `update` \| `delete` |

**Job name:** `JOB_NAMES.PRODUCT_UPDATED`

**jobId:** `product-updated:${productId}:${action}:${timestamp}` — or omit timestamp if one bust per save is enough.

---

## Processor — behaviour contract

**File:** `cacheInvalidation.job.js`

**Steps:**

1. Import same `invalidateCatalogueCache()` from Ch 20 — **reuse logic**, don't duplicate SCAN pattern.
2. Run invalidation.
3. Log: `[CACHE] invalidated catalogue after product ${productId} ${action}`

Worker needs Redis client — not MongoDB unless you fetch product first.

---

## Verify invalidation still correct

Repeat 20.11 test with worker running:

1. `GET /products` — warm cache (HIT).
2. Vendor `PATCH` product price.
3. `GET /products` — shows **new** price.

**Timing difference:** PATCH 200 may return **before** `[CACHE]` log — cache may be stale for milliseconds until worker runs.

| Acceptable? | Mitigation |
|---|---|
| Brief stale window | Short TTL from Ch 20 still applies |
| Not acceptable for gate | Keep sync invalidation; skip this sub-chapter |

**Course position:** Async invalidation OK if TTL + invalidation event eventually consistent — document tradeoff in learning log.

---

## Process architecture update

```
PATCH /products/:id
  → MongoDB save (sync)
  → enqueue product.updated (async)
  → 200 response

worker
  → invalidate catalogue keys
```

## ✅ Do / ❌ Don't

**✅ Do:** Reuse `invalidateCatalogueCache()` — single implementation.

**✅ Do:** Register processor in `worker.js` router.

**❌ Don't:** Run sync and async invalidation without reason — double SCAN.

**❌ Don't:** Break 20.11 gate test — stale price still fails course.
