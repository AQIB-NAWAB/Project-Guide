## BullMQ vs raw Redis commands

| Concept | BullMQ handles |
|---|---|
| Job persistence | Serialized payload in Redis |
| At-least-once delivery | Worker ack/nack |
| Retry scheduling | Delayed zset |
| Concurrency | Worker `concurrency` option |

You implement **processor functions** — not LPUSH/BRPOP manually.

## Common beginner mistakes (read before coding)

1. **Worker not started** — jobs enqueue; nothing logs; mentee thinks checkout broken.  
2. **Queue name mismatch** — `notifications` vs `notification`.  
3. **Enqueue before DB commit** — worker runs before order exists.  
4. **Throwing in processor without retry config** — job lost or stuck.  
5. **Same Redis `FLUSHALL` during dev** — wipes cache from Ch 20 mid-demo.  

Build order in this chapter: queue module → worker boots → checkout enqueue → processor logs.
