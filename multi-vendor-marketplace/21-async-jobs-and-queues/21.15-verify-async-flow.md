# 21.15 — Verify async flow

**Why this step matters:** Wiring jobs is not the gate — **timing proof** is. Mentors must see 201 before worker logs and two-store notify counts; this sub-chapter is the evidence chapter for async architecture.

**Where the project is now:** Queue module, worker, and job processors wired. Checkout and status PATCH enqueue jobs — not yet proven end-to-end.

**After this step:** Documented evidence that API returns before worker logs; multi-vendor notify count correct; retries/dedup tested.

End-to-end proof that checkout and status PATCH are **fast** and notifications run **asynchronously** in the worker.

## Test 1 — Checkout beats notifications

**Steps:**

1. Note time — run checkout from browser or curl.
2. Observe API response — 201 with two orders.
3. Observe worker terminal — `[NOTIFY]` lines appear **after** step 2.

**curl with timing:**

```bash
curl -s -w "\nHTTP:%{http_code} TIME:%{time_total}s\n" \
  -X POST http://localhost:5000/api/checkout \
  -H "Authorization: Bearer <customer-token>" \
  -H "Content-Type: application/json" \
  -d '{}'
```

**Pass criteria:**

- [ ] HTTP 201
- [ ] Response time roughly same as pre-Chapter 21 (no artificial SMTP delay)
- [ ] Worker shows 2× vendor + 1× customer notify logs

## Test 3 — Status update notification

## Failure diagnosis

| Symptom | Likely cause |
|---|---|
| 201 but no logs | Worker not running or wrong queue name |
| Logs before 201 | Notify still inline in handler — sync trap |
| Jobs stuck | Processor throws every attempt — check MongoDB in worker |
| Duplicate logs | Missing `jobId` dedup |
