# 21.13 — Order status job

**Why this step matters:** Status changes are the second hot path for side effects — vendors PATCH fast, customers learn later. Queuing here keeps fulfilment UI snappy and mirrors how real marketplaces notify on dispatch.

**After this step:** `PATCH /vendor/orders/:orderId` enqueues `order.statusUpdated` after status saves — customer notification runs in worker.

---

## Enqueue call site — spec

**File:** `orders.service.js` — inside `updateOrderStatus()` (or equivalent from Ch 19) **after** successful `save()`.

**Order of operations:**

```
1. Ownership + transition validation  (sync)
2. Persist new status  (sync)
3. enqueueOrderStatusUpdated({ orderId, customerId, previousStatus, newStatus, storeId })
4. return updated order JSON
```

---

## API contract (unchanged externally)

```
PATCH /api/vendor/orders/:orderId
Authorization: Bearer <vendor-token>
{ "status": "packed" }
```

**Response 200:** Order document with new `status` — same as Chapter 19.

---

## enqueueOrderStatusUpdated() — contract

| Field | Source |
|---|---|
| `orderId` | Route param |
| `customerId` | From order document |
| `previousStatus` | Value before update |
| `newStatus` | Request body status |
| `storeId` | Order's store — for log context |

**Job name:** `JOB_NAMES.ORDER_STATUS_UPDATED`

---

## Processor — behaviour contract

**File:** `orderStatusUpdated.job.js`

**Log mock customer notification:**

```
[NOTIFY][customer][status] orderId=<id> pending→packed store=<storeName>
```

**You implement** — fetch store name optional; ids sufficient for gate.

---

## Valid transitions reminder

Only enqueue **after** valid transition accepted — not on 400/403 responses.

| Response | Enqueue? |
|---|---|
| 200 status saved | ✓ |
| 400 invalid transition | ✗ |
| 403 wrong vendor | ✗ |
| 404 order missing | ✗ |

---

## jobId — spec

```javascript
// illustrative — dedupe rapid double PATCH
jobId: `order-status:${orderId}:${newStatus}`
```

Same status enqueued twice → BullMQ ignores duplicate `jobId` — see 21.14.

---

## curl verification

```bash
# Terminal 1: worker running
# Terminal 2:
curl -s -X PATCH "http://localhost:5000/api/vendor/orders/<orderId>" \
  -H "Authorization: Bearer <vendor-token>" \
  -H "Content-Type: application/json" \
  -d '{"status":"packed"}'
```

**Expected:** 200 JSON immediately. Worker logs `[NOTIFY][customer][status]` shortly after.

## Customer UI note

Chapter 19 UI refetches orders — status visible without waiting for notification job. Queue is parallel concern.

---

## ✅ Do / ❌ Don't

**✅ Do:** Pass `previousStatus` for meaningful log message.

**✅ Do:** Keep PATCH handler fast — no inline customer email.

**❌ Don't:** Enqueue on rejected transitions.

**❌ Don't:** Notify vendor on status change — customer only for this job.
