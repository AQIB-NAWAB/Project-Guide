# 21.14 — Retries and idempotency

**Why this step matters:** Networks blip and processors crash mid-job — retries turn "best-effort notify" into resilient notify. `jobId` dedup stops double-click enqueue from spamming duplicate mock emails.

**After this step:** Jobs use **retry policy** for transient failures and **jobId** deduplication to avoid duplicate notifications on re-enqueue.

---

## Retry options — spec

Apply on `queue.add()` and/or Worker default job options:

| Option | Suggested value | Purpose |
|---|---|---|
| `attempts` | `3` | Retry transient MongoDB/Redis blips |
| `backoff` | `{ type: 'exponential', delay: 2000 }` | Spread retries |
| `removeOnComplete` | `100` or `true` | Dev Redis hygiene — document choice |
| `removeOnFail` | `false` | Keep failed jobs visible for debugging |

**Processor throw** → BullMQ schedules retry. **Processor return** → job complete.

---

## What should retry

| Failure | Retry? |
|---|---|
| MongoDB timeout loading order | ✓ |
| Redis connection blip on enqueue | Handler logs; job never created — separate concern |
| Unknown job name | ✗ — fix code, don't retry forever |
| Mock log `console.log` throws | Unlikely — don't test artificially |

---

## jobId deduplication — spec

BullMQ rejects second `add()` with same `jobId` if first still active or completed per your `removeOnComplete` setting.

### order.created

```javascript
await queue.add(JOB_NAMES.ORDER_CREATED, payload, {
  jobId: `order-created:${orderIds.slice().sort().join('-')}`,
  attempts: 3,
  backoff: { type: 'exponential', delay: 2000 },
});
```

Same checkout batch → one notification job even if enqueue called twice.

### order.statusUpdated

```javascript
jobId: `order-status:${orderId}:${newStatus}`
```

Vendor clicks "Mark packed" twice quickly → one customer notify job.

---

## Idempotency limits (honest scope)

| Scenario | Queue fixes? |
|---|---|
| Duplicate checkout (double-click) | ✗ — two order batches, two jobIds |
| Retry after partial processor run | Logs may duplicate — acceptable for mock |
| Worker crash after log, before ack | At-least-once delivery — real email needs idempotent provider |

Document in README: **notification idempotency is best-effort** at course level.

## Observing failed jobs

| Method | Command / action |
|---|---|
| Worker log | `Job failed` + attempt count |
| Redis CLI | `LRANGE bull:notifications:failed 0 5` — key names vary by BullMQ version |
| Throw test (dev only) | Temporary `throw new Error('test')` in processor — remove after verify |

**Verify once:** Force failure → see retry attempts in worker log → fix processor → job completes.

---

## Dead letter behaviour

After `attempts` exhausted, job moves to **failed** set. Gate checklist:

- [ ] You know where to find failed jobs in dev
- [ ] README mentions restarting worker after fix

No Bull Board required for gate.

---

## ✅ Do / ❌ Don't

**✅ Do:** Set `attempts` on notification jobs — demo resilience story.

**✅ Do:** Use deterministic `jobId` from domain ids — not random UUID per enqueue.

**❌ Don't:** Retry forever — cap attempts.

**❌ Don't:** Use `jobId` based only on `customerId` — collapses distinct checkouts.
