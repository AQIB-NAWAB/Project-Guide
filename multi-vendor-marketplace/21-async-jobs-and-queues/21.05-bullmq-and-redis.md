# 21.5 — BullMQ and Redis

Chapter 20 used Redis for **cache keys** (`catalogue:*`). Chapter 21 uses the **same Redis server** for **BullMQ queues** — different data structures, different key prefixes. One `REDIS_URL`, two purposes.

---

## Same Redis, different usage

| Use | Chapter | Redis data | Example keys |
|---|---|---|---|
| Cache-aside | 20 | String values (JSON) | `catalogue:page1:categoryDairy` |
| Job queue | 21 | Lists, sorted sets, hashes (BullMQ internals) | `bull:notifications:*` |

BullMQ manages its own key namespace — typically prefixed `bull:` + queue name. **Do not** manually `DEL bull:*` unless debugging — use BullMQ APIs.

---

## BullMQ components

| Piece | Runs in | Role |
|---|---|---|
| `Queue` | API process | `queue.add('order.created', payload)` |
| `Worker` | `worker.js` process | Polls Redis, runs processor function |
| `Job` | Redis + worker | Payload, attempts, timestamps |
| Connection | Both | `REDIS_URL` — `ioredis` recommended by BullMQ docs |

API **produces** jobs. Worker **consumes** jobs. They share Redis; they do **not** share in-memory state.

---

## Why BullMQ for this course

| Alternative | Why not for gate |
|---|---|
| Raw Redis LPUSH/BRPOP | No retries, no job metadata, reinventing wheel |
| Kafka | Ops overhead; no inventory/event-stream scope |
| In-process `setImmediate` | Lost on API restart; no persistence |
| Cron polling MongoDB | Latency; misses real-time notify |

BullMQ fits **Node monolith + Redis already installed**.

---

## Queue naming convention — spec

Document in `jobs.constants.js`:

```javascript
// illustrative
export const QUEUE_NAMES = {
  NOTIFICATIONS: 'notifications',
};

export const JOB_NAMES = {
  ORDER_CREATED: 'order.created',
  ORDER_STATUS_UPDATED: 'order.statusUpdated',
  PRODUCT_UPDATED: 'product.updated',
};
```

Single queue with multiple job names is fine for course scope. Multiple queues also valid — pick one, document it.

## Connection sharing with Ch 20

| Approach | Valid? |
|---|---|
| Separate `ioredis` connections — cache client + BullMQ connection | ✓ Recommended |
| Reuse same `ioredis` instance for cache and BullMQ | Possible — read BullMQ docs on `connection` option |
| Two different `REDIS_URL` values | ✗ Unnecessary for course |

**You implement** — simplest path: new connection in `queue.js` dedicated to BullMQ.

## Dev vs production

| Environment | Processes |
|---|---|
| Local | Terminal 1: API. Terminal 2: `npm run worker` |
| Production (Ch 22) | API service + **separate worker service** or second PM2 process |

Chapter 21 establishes worker locally. Chapter 22 deploys both.

## Observability (gate-level)

| Tool | Purpose |
|---|---|
| Worker `console.log` | Mock email output — primary gate proof |
| Redis CLI `KEYS bull:*` | Confirm jobs landed — dev only |
| Bull Board (optional stretch) | Web UI for queues — not required |

## ✅ Do / ❌ Don't

**✅ Do:** Use same `REDIS_URL` env var as Chapter 20.

**✅ Do:** Install `bullmq` — document version in README.

**❌ Don't:** `FLUSHALL` on shared Redis — destroys cache **and** queued jobs.

**❌ Don't:** Run worker in same file as `server.js` listen — separate entry for gate.
