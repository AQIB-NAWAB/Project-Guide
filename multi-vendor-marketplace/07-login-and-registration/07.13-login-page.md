# 7.13 — Login page

**Why this step matters:** Login stores access in memory and refresh via cookie — the security pattern from 7.2–7.4 becomes real here. Successful login should redirect by role and trigger boot-friendly session restore on return visits.

**Where the project is now:** useAuth.login exists; Login form not fully wired.

**After this step:** Login form calls API, stores access in memory, refresh in cookie (if server sets), sets user state, redirects by role.

---

## Form contract

| Field | |
|---|---|
| email | required |
| password | required |

---

## Submit flow

```
1. await login(email, password)
2. useAuth sets access token + user
3. Redirect:
   - vendor → /vendor (or /vendor/dashboard stub)
   - customer → / (Home) or /shop stub
4. On 401: "Invalid credentials"
5. On 403: "Please verify your email" + link to verify-otp
```

---

## Cookie handling

Login fetch uses `credentials: 'include'` if server sets refresh cookie.

If server still returns refresh in body temporarily, coordinate with 7.14 to set cookie server-side — preferred.

---

## Display success from verify

Show `location.state.message` banner if present.
