# 7.5 — The 401 retry loop

When access token expires mid-session, user should not see login screen if refresh is valid. The API client implements **refresh once, retry once, else logout**.

## Why this matters for absolute beginners

Without retry logic, users get kicked to login every 15 minutes while grocery shopping — unacceptable UX. With naive retry, infinite loops spam `/refresh`. The `_retried` flag is small but critical — distinguish expired access (refresh) from forbidden action (403, no refresh).

## Think of it like…

**Expired day pass at a museum.** Guard checks wristband (401) — not ejecting you immediately if membership desk (refresh) can print a new day pass once. Still forbidden rooms (403) are not fixed by a new wristband.

## Common beginner questions

**Q: Should login 401 trigger refresh?**  
A: No — wrong password is not expired token. Interceptor skips public auth routes.

**Q: Two API calls 401 at once?**  
A: Optional mutex shares one refresh promise — stretch acceptable to document duplicates.

**Q: 403 after retry — logout?**  
A: No — user is authenticated but not allowed; show error, keep session.

---

## Algorithm

```
apiRequest(url, options):
  response = fetch with Bearer access

  if response.status !== 401:
    return response

  if options._retried:
    logout(); redirect login
    throw Unauthorized

  newAccess = await refreshAccessToken()  // cookie POST
  if !newAccess:
    logout(); redirect login
    throw Unauthorized

  setAccessToken(newAccess)
  return apiRequest(url, { ...options, _retried: true })
```

**Critical:** `_retried` flag prevents infinite loop if refresh succeeds but original route still 401 (permission issue).

---

## Concurrent 401s

Multiple requests expiring together may fire parallel refreshes. **Optional mutex:** first 401 starts refresh; others await same promise.

Stretch acceptable: sequential refresh calls — note in learning log if duplicate refresh seen.

---

## 401 vs 403 after retry

| Status after retry | Meaning | Action |
|---|---|---|
| 401 | Still unauthenticated | logout |
| 403 | Authenticated, forbidden | show message — **no** refresh loop (Ch 8) |

Only 401 triggers refresh — not 403.

---

## Public routes

Login, register, verify, health — **no** Bearer header, **no** refresh on 401 from wrong password.

Interceptor applies to authenticated api client helper — not raw fetch on login form.

---

## Expired refresh

Refresh returns 401 → clear auth state → redirect `/login?session=expired` optional query.

---

## Testing

1. Login — set JWT_ACCESS_EXPIRY=30s on server temporarily  
2. Wait — click protected action  
3. Network tab: original 401 → refresh 200 → retry 200  
4. User stays on page — no login flash  

Document in learning log.

---

## ✅ Do / ❌ Don't

**✅ Do:** Retry original request with new access only once.

**✅ Do:** Skip interceptor for /auth/login and /auth/register.

**❌ Don't:** Refresh on 403 — wrong semantics.

**❌ Don't:** Infinite retry loops.
