# 7.16 — Verify in browser

**Why this step matters:** curl proved the API; this step proves **humans** can register, verify, login, and hit a protected page — Week 2's auth demo for portfolio and mentors. Screenshot or short recording belongs in your learning log.

**Where the project is now:** All auth pieces wired individually.

**After this step:** End-to-end browser demo completed — register, verify, login vendor + customer, protected route, logout, optional refresh test.

---

## Demo script — record results

### Test 1 — New vendor registration

1. `/register` — new email, vendor role  
2. `/verify-otp` — enter OTP  
3. `/login` — login success  
4. Redirect to vendor protected page  
5. Logout  

### Test 2 — Seed customer

1. `/login` — customer-a@test.local  
2. Customer home loads — not vendor route  
3. `/vendor` blocked if role prop set  

### Test 3 — Unverified block

1. Register user — skip verify  
2. Login attempt — 403 message shown  

### Test 4 — Session refresh (optional)

1. Login  
2. Short JWT_ACCESS_EXPIRY on server  
3. Wait — trigger apiFetch call  
4. Network: 401 → refresh → retry 200  
5. Still authenticated without re-login  

### Test 5 — Page reload

1. Login as vendor  
2. Hard refresh browser  
3. Boot refresh restores session — vendor page still reachable after load  

Document if reload logs user out — fix boot refresh before gate.

## Common failures

| Symptom | Fix |
|---|---|
| CORS error on refresh | Allow credentials + specific origin |
| Reload logs out | Boot refresh in AuthProvider |
| Infinite login redirect | isLoading false too early |
| Cookie not set | Secure on localhost — disable in dev |
