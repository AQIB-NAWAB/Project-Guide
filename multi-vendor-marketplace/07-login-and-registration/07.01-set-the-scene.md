## Bridge from last chapter

Chapter 6 proved authentication **works on the server** — register, verify OTP, login, refresh, and `GET /api/auth/me` with Bearer token all pass curl tests. User documents in MongoDB have real bcrypt hashes. Yet your React app still only shows the health check; humans cannot register or sign in through the browser.

What carries forward: exact API request/response shapes from Chapter 6, CORS already enabled for Vite, and `client/src/pages/` + `api/client.js` from Chapter 2. What you add now is **session continuity in the browser** — forms, error messages, access token in memory (not localStorage), refresh token in httpOnly cookie, automatic retry when access expires, and protected routes that redirect strangers to login.

The hat you wear now is **frontend auth integrator**. You are not redesigning the API; you are making it usable by vendors and customers. Week 2's demo becomes "register in the browser, verify, login, land on the right page" — not just curl output in a terminal.

## In plain English

| Term | Meaning |
|---|---|
| **Bearer token** | Access JWT sent in the `Authorization` header on API calls |
| **Auth context / hook** | Shared React state (`useAuth`) holding user and login methods |
| **httpOnly cookie** | Cookie JavaScript cannot read — used for refresh token |
| **Interceptor** | API client wrapper that attaches tokens and handles 401 retry |
| **Protected route** | Page that redirects to login if user is not authenticated |
| **Silent refresh** | On app load, call `/auth/refresh` to restore session without login form |
| **XSS** | Attack injecting script into your page — steels tokens if stored carelessly |
| **credentials: 'include'** | fetch option sending cookies on cross-origin requests |

## Common beginner questions

**Q: Why not store the JWT in localStorage — it's easier?**  
A: Any script on your page can read localStorage. Access in **memory** plus refresh in **httpOnly cookie** is the course pattern — harder to steal and industry-aligned for SPAs.

**Q: Do I need new server endpoints?**  
A: Usually no — same Chapter 6 routes. You may add `Set-Cookie` on login/refresh for httpOnly refresh and enable `credentials: 'include'` on the client.

**Q: What happens on page reload if access token is in memory?**  
A: Memory clears — app boot calls refresh with the cookie to get a new access token silently (sub-chapters 7.4–7.5).

**Q: Why separate chapters for API auth and UI auth?**  
A: So you can debug server vs client independently. Broken curl means fix Chapter 6; broken form with working curl means fix Chapter 7.

**Q: Role-based UI (vendor vs customer) — this chapter?**  
A: Basic protected routes only. Role-specific guards and IDOR prevention deepen in Chapter 8.
