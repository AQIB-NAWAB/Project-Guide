# 7.11 — Register page

**Why this step matters:** Register is most users' first UI touchpoint — it must call Chapter 6's API shape, surface validation errors, and route to Verify OTP with email context. Sloppy register UX loses vendors before they open a shop.

**Where the project is now:** Register page UI scaffold; useAuth has register method stub or missing wiring.

**After this step:** Register form submits to `POST /api/auth/register`, shows errors, navigates to VerifyOtp with email in router state on success.

---

## Form contract

| Field | Validation (client) |
|---|---|
| email | required, email shape |
| password | min 8 chars |
| confirmPassword | must match password |
| role | vendor or customer required |
| displayName | optional |

Submit disabled while loading.

---

## Submit flow

```
1. Client validation
2. await register({ email, password, role, displayName })
3. On 201: navigate('/verify-otp', { state: { email } })
4. On 409: show "Email already registered"
5. On 400: show server message
```

---

## Error display

Single `error` string state or field-level — match editorial style.

Show API `message` from JSON body.

---

## Dev OTP hint

If API returns `otp` in dev response, show dismissible hint: "Dev OTP: 123456" — optional UX nicety.

---

## Verify in browser

2. Register new vendor email
