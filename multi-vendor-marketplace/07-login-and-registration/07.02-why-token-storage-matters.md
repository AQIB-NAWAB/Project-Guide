# 7.2 — Why token storage matters

Where you store JWT in the browser determines **XSS resilience** and **session persistence**. Wrong choice — localStorage for access token — is the most common MERN auth mistake.

This sub-chapter motivates Chapter 7's pattern: **access in memory**, **refresh in httpOnly cookie**.

## Why this matters for absolute beginners

localStorage survives refresh and is one line for any XSS script to read. Memory-only access hurts convenience slightly — page reload needs silent refresh — but theft window shrinks. This is a **deliberate trade** you explain in README and viva, not a framework default you accept blindly.

## Think of it like…

**Cash in pocket vs safe deposit box.** Access token in pocket — handy, gone when you leave (tab close/reload). Refresh in bank safe (httpOnly cookie) — JS pickpockets cannot reach it; bank clerk (browser) sends it only to your API.

## Common beginner questions

**Q: If XSS can steal memory token anyway, why bother?**  
A: XSS is catastrophic either way — goal is not making theft **trivial and persistent** via localStorage.

**Q: What is CSRF vs XSS here?**  
A: XSS steals; CSRF tricks browser into unwanted requests — refresh cookie needs SameSite; Bearer access is less CSRF-prone.

---

## Attack surface — XSS

If attacker injects JavaScript on your page (bad npm package, unsanitized HTML):

| Storage | XSS can steal? |
|---|---|
| localStorage | yes — `localStorage.getItem('token')` |
| sessionStorage | yes |
| JavaScript variable (memory) | yes — but only while tab open, not persistent read |
| httpOnly cookie | **no** — JS cannot read |

No storage is XSS-proof for access if XSS exists — but **don't make theft trivial and persistent**.

---

## localStorage anti-pattern

```javascript
// ❌ Do not use for access token
localStorage.setItem('accessToken', token)
```

Survives refresh — convenient but **any script on page reads it**.

---

## Course pattern

| Token | Storage | Lifetime |
|---|---|---|
| Access | React state / useAuth ref | Until refresh or logout |
| Refresh | httpOnly, Secure (prod), SameSite cookie | 7 days |

Page reload loses access token from memory → **silent refresh** on app boot calls POST /auth/refresh with cookie → new access in memory.

---

## CSRF note

Bearer access in header — not auto-sent like cookies — low CSRF on API.

Refresh cookie **is** auto-sent — use SameSite=Lax or Strict; CSRF token optional stretch for cookie-only auth.

When server sets refresh cookie in Ch 7, enable `credentials: 'include'` on fetch.

---

## Real breach pattern

Stolen access token from localStorage → attacker calls API until expiry (15m). Stolen refresh from non-httpOnly cookie worse — longer window.

httpOnly refresh + short access limits damage.

---

## ✅ Do / ❌ Don't

**✅ Do:** Document token storage in README security section.

**✅ Do:** Clear memory on logout.

**❌ Don't:** Log tokens to console in production builds.

**❌ Don't:** Put access token in URL hash.
