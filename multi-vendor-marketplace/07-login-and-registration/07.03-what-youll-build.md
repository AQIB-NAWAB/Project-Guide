# 7.3 — What you'll build

Chapter 7 delivers **browser auth UX** wired to Chapter 6 API — register, verify, login, silent refresh, protected navigation.

## Why this matters for absolute beginners

This checklist is the **browser-side gate** — three pages, shared auth state, interceptor, protected routes. Each maps to build sub-chapters 7.9–7.16. Without the list, it is easy to build Login but forget boot-time refresh or OTP route — users appear logged out on every reload.

## Common beginner questions

**Q: Do I need Redux for auth?**  
A: No — Context + `useAuth` hook is enough for this scope.

**Q: Can I change API shapes from Chapter 6?**  
A: No — fix server if mismatch; client adapts to existing contract.

**Q: Where do vendors land after login?**  
A: Your router choice — document stub path (`/vendor` or similar); full dashboard is Chapter 10+.

## Pages

| Page | Route | API calls |
|---|---|---|
| Register | `/register` | POST /api/auth/register |
| Verify OTP | `/verify-otp` | POST /api/auth/verify-otp |
| Login | `/login` | POST /api/auth/login |

Post-login redirect by role:

- vendor → `/vendor` or dashboard stub  
- customer → `/` or shop stub  

Exact paths your choice — document in router.

---

## useAuth / AuthContext

Provides:

- `user`, `isAuthenticated`, `isLoading`  
- `login(email, password)`  
- `register(...)`  
- `verifyOtp(email, otp)`  
- `logout()`  
- `accessToken` — internal to api client, not exposed to components unnecessarily  

Boot: attempt refresh on mount to restore session.

---

## API client extensions

- Attach `Authorization: Bearer` when access token set  
- On 401: call refresh once → retry original request → if fail, logout + redirect login  
- `credentials: 'include'` for refresh cookie  

---

## ProtectedRoute

Wrapper component:

- If loading → spinner or null  
- If !authenticated → `<Navigate to="/login" />`  
- Else → render children  

Apply to vendor routes preview — full role gates in Ch 8.

---

## Server adjustment (optional)

Set refresh token as httpOnly cookie on login/refresh response:

```
Set-Cookie: refreshToken=...; HttpOnly; Path=/api/auth; SameSite=Lax
```

May require `cookie-parser` on server. Document if login returns cookie + empty refresh in body.

## Out of scope

- Fancy auth UI design — clean forms enough  
- Remember me checkbox — refresh duration handles persistence  
- Social login  

---

## ✅ Do / ❌ Don't

**✅ Do:** Reuse API error messages in UI.

**✅ Do:** Pass email to VerifyOtp via router state from Register.

**❌ Don't:** Change API request/response shapes from Ch 6.

**❌ Don't:** Store access in localStorage for gate.
