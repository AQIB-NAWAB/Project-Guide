# 20.9 — Cache catalogue middleware

**Why this step matters:** This is the core cache-aside pattern — the shop window stops hitting MongoDB on every repeat glance. HIT/MISS logs here are your first proof the performance layer works.

**After this step:** `GET /products` checks Redis before MongoDB; stores JSON response on miss.

---

## Cache-aside flow

```
GET /products?...
  1. Build cache key from path + query
  2. GET key from Redis
  3. HIT → return cached JSON (same status 200)
  4. MISS → run existing catalogue query
  5. SET key with TTL (seconds)
  6. Return response
```

Apply only to **GET** public products list — not single product route unless you extend (optional).

---

## Key generation contract

Deterministic from sorted query params:

```
catalogue:page=1&limit=20&category=Dairy
```

Include `search` param when present — different keys for different searches.

---

## Response body

Cache **final JSON string** sent to client — includes populated store names from Ch 14.

Set `Content-Type: application/json` as today.

---

## Dev observability

Log `CACHE HIT` / `CACHE MISS` with key — remove or reduce in production.

---

## Middleware placement

Either:

- Express middleware before products router for `GET /products` only, or
- Inside `products.service.listPublic()` — service-layer cache

Pick one — document in README.

---

## ✅ Do / ❌ Don't

**✅ Do:** TTL 300–900s — state value in README.

**✅ Do:** Only cache 200 responses.

**❌ Don't:** Cache vendor-only routes.

**❌ Don't:** Cache authenticated personalised responses on public route.
