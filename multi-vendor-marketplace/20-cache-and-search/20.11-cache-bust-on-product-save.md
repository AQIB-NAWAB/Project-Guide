# 20.11 — Cache bust on product save

**Why this step matters:** Without invalidation, your fast cache serves wrong prices — the stale trap from 20.2 becomes a real bug. This step is what makes caching **safe** for a marketplace, not just fast.

**After this step:** Product POST/PATCH/DELETE invalidates catalogue cache keys — next GET reflects changes.

---

## invalidateCatalogueCache() — spec

| Trigger | Action |
|---|---|
| `createProduct` success | Delete keys matching `catalogue:*` |
| `updateProduct` success | Same |
| `deleteProduct` success | Same |

Implement in `products.service.js` after DB write succeeds — before return.

---

## Key deletion strategies

| Approach | Dev OK | Production note |
|---|---|---|
| `SCAN` + `DEL` matching prefix | ✓ | Preferred |
| Track keys in a Set on write | ✓ | Explicit |
| `KEYS catalogue:*` | Dev only | Blocks Redis — document avoid |

---

## Test script

1. `GET /products` — warm cache (HIT).
2. Vendor `PATCH` product name or price.
3. `GET /products` — must show **new** value (MISS then fresh data).
4. Optional: HIT again with **updated** body.

---

## Stale cache failure

If step 3 shows old price — invalidation not wired or wrong key pattern.

---

## ✅ Do / ❌ Don't

**✅ Do:** Test price change specifically — 20.2 trap.

**✅ Do:** Invalidate even if only stock changed — list may filter in-stock.

**❌ Don't:** Rely on TTL alone for gate.
