# 20.2 — Stale cache trap

Vendor changes milk price from ₹65 to ₹70. Customer still sees ₹65 for an hour because catalogue response was **cached without invalidation**. Trust erodes faster than slow pages.

This sub-chapter names the trap **before** you wire Redis in 20.8–20.11. Speed without correctness is worse than no cache at all.

---

## The bug

| Step | Problem |
|---|---|
| Cache `GET /products` 60 min TTL | Fast reads |
| Vendor PATCHes price | MongoDB updated |
| Customer GET | **Stale JSON from Redis** |

TTL alone eventually fixes — **eventually** is not acceptable for price.

---

## Why beginners fall into this trap

Caching feels like a win the first time you see `CACHE HIT` in logs. It is easy to ship TTL and forget **writes**. The bug only appears when a vendor edits a product — often days later, blamed on "the cache being weird."

**Rule:** Every cache needs an **invalidation story** for every write path that affects cached data.

---

## Fix — two mechanisms (both in this chapter)

| Mechanism | When |
|---|---|
| **TTL** | Safety net — e.g. 5–15 min — catches missed invalidation bugs |
| **Event invalidation** | On product POST/PATCH/DELETE — delete matching cache keys immediately |

Chapter 20 implements **both in sequence** (20.4).

```
On GET:  try cache → on miss, query DB → set with TTL
On POST/PATCH/DELETE product:  delete cache keys for catalogue
```

---

## What to invalidate

Minimum: bust catalogue cache keys when **any** product write affects public list:

- Create product
- Update price, stock, publish status
- Delete product

Key pattern example: `products:list:*` or hash of query params — document your scheme in README.

---

## What not to cache

| Route | Why |
|---|---|
| POST /checkout | Write — never cache |
| GET /cart | User-specific |
| GET /orders/my | User-specific |
| GET /health | Cheap; db status must be live |

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Stale cache** | Cached answer is **out of date** — still fast, but wrong |
| **Invalidation** | Throwing away cached copies when the real data changed |
| **TTL** | Automatic expiry — "this sticky note is valid for 5 minutes" |

---

## Think of it like a sale sign in the window

You put a "Milk ₹65" sign in the window (cache). Vendor changes the shelf price to ₹70 (MongoDB update). If you do not replace the sign (invalidation), customers see the wrong price until the sign falls off by itself (TTL expiry).

---

## Common beginner questions

**Q: If TTL is 5 minutes, isn't that good enough?**  
For a demo with no edits, maybe. For a marketplace where vendors change prices and stock, **5 wrong minutes** is a bug. Event invalidation fixes it immediately.

**Q: Should I cache search results too?**  
Yes, with keys that include the `search` param — and invalidate on the same product writes. Or skip caching search if simpler; document your choice.

**Q: What about cart and orders?**  
Never cache user-specific routes this chapter. Catalogue only.

---

## ✅ Do / ❌ Don't

**✅ Do:** Invalidate on write paths in products module.

**✅ Do:** Log cache HIT/MISS in dev.

**❌ Don't:** Cache per-user catalogue without varying key by auth — public only.

**❌ Don't:** Forget invalidation because TTL exists.
