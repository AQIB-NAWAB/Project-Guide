# 20.4 — TTL vs event invalidation

Production caches use **both**. TTL is a safety net; invalidation is correctness on writes.

Beginners often pick one and wonder why production still breaks. This sub-chapter gives you the **decision frame** before you implement in 20.9 and 20.11.

---

## TTL only

| Pro | Con |
|---|---|
| Simple — set expiry, forget | Stale until expiry |
| No invalidation code paths | Wrong price visible for entire TTL window |
| Good for rarely changing data | Bad for vendor-edited catalogue |

**Example failure:** TTL = 15 minutes. Vendor fixes a typo price at 10:00. Customers see wrong price until 10:15.

---

## Event invalidation only

| Pro | Con |
|---|---|
| Fresh immediately after writes | Miss one code path → stale **forever** |
| Correct for price changes | Orphan keys if key pattern wrong |
| Professional default for catalogues | More code to maintain |

**Example failure:** You invalidate on `PATCH` but forget `DELETE` — deleted products appear in cache until manual flush.

---

## Both (course mandate)

```
On GET:  try cache → on miss, query DB → set with TTL
On POST/PATCH/DELETE product:  delete cache keys for catalogue
```

| Layer | Role |
|---|---|
| **Event invalidation** | Correctness — user sees new price immediately |
| **TTL** | Safety net — orphaned keys expire; catches bugs in invalidation |

TTL catches bugs in invalidation — keys expire eventually. Invalidation catches user-visible errors — prices fix now.

---

## Key design

Document pattern in README:

```
catalogue:{hash(sortedQueryParams)}
```

Or single key `catalogue:all` if only unfiltered list cached — **prefer per-query** if filters exist from Ch 14.

Different `page`, `category`, and `search` params → different keys → invalidation must clear **all catalogue keys** (prefix delete), not one key.

---

## Invalidation call sites

`products.service.js` — after successful:

- `createProduct`
- `updateProduct`
- `deleteProduct`

Call `invalidateCatalogueCache()` — not scattered in routes. One function, one place to audit.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **TTL** | "Throw this cached answer away after N seconds" — automatic |
| **Event invalidation** | "Throw cached answers away **when something specific happens**" — e.g. vendor saves product |
| **Prefix delete** | Delete all keys starting with `catalogue:` — clears every cached page/filter variant |

---

## Think of it like milk expiry dates

**TTL** = printed expiry date on the carton — good enough if nobody changes the recipe.

**Event invalidation** = pulling all cartons from the shelf the moment the recipe changes.

**Both** = pull immediately on recipe change *and* expiry date as backup for any cartons you missed.

---

## Common beginner questions

**Q: What TTL should I pick?**  
300–900 seconds (5–15 min) is fine for the course. Document your choice. Shorter TTL reduces stale window but increases MongoDB load.

**Q: Can I use `FLUSHALL` to invalidate?**  
**No** in shared Redis — destroys BullMQ jobs from Ch 21 and all cache keys. Use prefix `SCAN` + `DEL` on `catalogue:*` only.

**Q: Implement TTL or invalidation first?**  
Course order: cache with TTL in 20.9, then invalidation in 20.11 — so you can **feel** the stale trap before fixing it.

---

## ✅ Do / ❌ Don't

**✅ Do:** Implement TTL in 20.9, invalidation in 20.11 — sequence matters for testing.

**✅ Do:** Use Redis `DEL` or `SCAN`+`DEL` for pattern — avoid `KEYS` in production docs note.

**❌ Don't:** Invalidate entire Redis DB — `FLUSHALL` nukes unrelated keys.

**❌ Don't:** Assume one cache key — filters and pagination multiply keys.
