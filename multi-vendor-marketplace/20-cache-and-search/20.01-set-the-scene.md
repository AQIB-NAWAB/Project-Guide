## In plain English

| Term | Simple meaning |
|---|---|
| **Redis** | A very fast in-memory database that sits beside MongoDB — like a sticky note the server checks before opening the filing cabinet |
| **Cache** | A saved copy of a recent answer — so the server does not repeat the same expensive work |
| **Cache hit** | The answer was already on the sticky note — instant return |
| **Cache miss** | Sticky note empty — server fetches from MongoDB, then saves a copy for next time |
| **TTL (time to live)** | How long a cached answer stays valid before it expires automatically — e.g. 5 minutes |
| **Cache invalidation** | Deliberately throwing away stale sticky notes when data changes — e.g. vendor updates a price |
| **Text index** | MongoDB's built-in search index — finds products by words in name and category without scanning every row |
| **Fallback** | When Redis is down, the app still works — just slower — by reading MongoDB directly |

## Think of it like a shop window display

The **catalogue** (`GET /products`) is your shop window — the most looked-at part of the store. Every customer who opens `/shop` asks the same question: *"What products are on page 1?"*

Without cache, your server walks to the warehouse (MongoDB) for **every single glance** — even when nothing changed since the last customer.

With cache, the first customer triggers a warehouse trip; the next hundred read from a **photograph of the window display** (Redis) until a vendor rearranges the shelf (product write) or the photograph expires (TTL).

**Important:** You cache the **shop window**, not the **checkout desk**. Cart, orders, and auth stay fresh every time — they are personal and change constantly.

## Common beginner questions

**Q: What is Redis? Do I need another database?**  
Redis is an **in-memory data store** — extremely fast reads and writes. For this course it holds cached catalogue JSON and (in Chapter 21) job queues. MongoDB remains your **source of truth** for products, orders, and users. Redis is a helper, not a replacement.

**Q: Why not just make MongoDB faster?**  
MongoDB is already correct. The win is **avoiding repeat work** — the same `GET /products?page=1` asked hundreds of times per minute should not hit MongoDB hundreds of times if the product list has not changed.

**Q: What if cache shows an old price after a vendor update?**  
That is the **stale cache trap** (20.2). Fix: **invalidate** cache keys when products change — do not rely on TTL alone for price correctness.

**Q: Will my app break if Redis is not installed?**  
Not if you implement fallback (20.14). The gate requires: Redis stopped → catalogue still loads from MongoDB. Slower, but alive.

**Q: Is this the same Redis as Chapter 21?**  
Yes — **one Redis server**, two jobs: cache keys (`catalogue:*`) and BullMQ queue keys (`bull:*`). Different prefixes, same `REDIS_URL`.

**Q: Do customers see "cache" in the UI?**  
No. Cache is invisible server-side plumbing. Customers just notice the shop feels snappier on repeat visits.
