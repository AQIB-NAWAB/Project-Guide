# 20.3 — What you'll build

Blueprint for the **performance layer** on catalogue reads. Read once; verify against your repo after 20.14.

You are adding **speed and search** — not new customer-facing features. The shop should look the same; repeat loads should feel faster and `?search=` should use an index.

## Behaviour contract

| Feature | Spec |
|---|---|
| Cache | `GET /products` and query variants — key from path + sorted query string |
| TTL | 300–900 seconds — document chosen value in README |
| Invalidation | Product create/update/delete clears catalogue keys |
| Search | `?search=` uses MongoDB `$text` on name + category |
| Text index | Created on Product collection in 20.12 |
| Fallback | Redis connection fail → skip cache, query MongoDB |
| Metrics | Log before/after response time once (20.10) |

## What you will not build

| Out of scope | Why |
|---|---|

| Elasticsearch | MongoDB text index is enough for course |
| CDN for images | Cloudinary already serves images |

**Q: Will caching break my Week 3 demo?**  
Not if invalidation works. Re-run the stale-price test in 20.11 after every cache change.

README — mentors grep for your key pattern during viva.
