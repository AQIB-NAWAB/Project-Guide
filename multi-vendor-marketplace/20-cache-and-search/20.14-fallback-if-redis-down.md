# 20.14 — Fallback if Redis down

**Why this step matters:** Production Redis restarts, misconfigured URLs, and free-tier limits happen. An app that crashes when cache is unavailable is not shippable — degraded speed is acceptable; downtime is not.

**After this step:** API serves catalogue from MongoDB when Redis unavailable — **no crash**, degraded performance only.

---

## Fallback contract

```
getFromCache(key):
  if redis client null or error → return null (miss)

setCache(key, value):
  if redis unavailable → no-op, log debug

catalogue handler:
  always can fall through to MongoDB query
```

`GET /products` returns 200 with correct data — slower without cache.

---

## Test script

1. Stop Redis: `sudo systemctl stop redis` or stop Docker container.
2. Restart API — boot succeeds with warning log.
3. `curl GET /products` → 200 with products.
4. Start Redis — HIT/MISS logging resumes.

---

## Boot behaviour

| Redis down at boot | Acceptable |
|---|---|
| API starts, warns | ✓ Required for gate |
| API crash loop | ✗ Fix |

---

## Health endpoint — optional

Do not mark `redis: connected` in `/health` unless you add it — not required for gate. MongoDB `db` status remains primary.

---

## ✅ Do / ❌ Don't

**✅ Do:** Try/catch around Redis GET/SET — never throw to client.

**✅ Do:** Document fallback in README ops section.

**❌ Don't:** Require Redis for checkout or orders.
