# 17.2 — Grouped by store

A cart with ten lines from three vendors is not one list — it is **three fulfilment promises**. Grouping by store is not decoration. It is the same boundary Chapter 18 uses when one checkout becomes multiple orders.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Flat list** | All cart lines in one block — confusing when items come from different vendors |
| **Store section** | UI block with vendor name as heading and that vendor's lines underneath |
| **Store subtotal** | Sum of (price × quantity) for lines **in that section only** |
| **Grand total** | Sum of all store subtotals — label as estimate until checkout snapshots prices |

**Beginner mistake:** Showing one combined list because "it's all one cart." Technically one cart — but **fulfillment is per vendor**. Grouping teaches the customer (and you) how checkout will split.

---

## One cart, many vendors — the customer mental model

```
┌─────────────────────────────────────┐
│  Your cart                          │
├─────────────────────────────────────┤
│  Green Valley Grocers               │
│    Basmati Rice  2 kg    ₹240       │
│    Milk  1 litre         ₹65        │
│  Subtotal: ₹305                       │
├─────────────────────────────────────┤
│  Spice Corner                       │
│    Turmeric  200 g       ₹40        │
│  Subtotal: ₹40                        │
├─────────────────────────────────────┤
│  Estimated total: ₹345                │
│  [Proceed to checkout]  (Ch 19)     │
└─────────────────────────────────────┘
```

The customer understands: *Green Valley packs one bag; Spice Corner packs another.* That matches how vendors fulfil in the real world.

---

## Why flat lists fail

| Flat list problem | Grouped fix |
|---|---|
| "Who delivers this?" unclear | Store header on each section |
| Subtotal meaningless | Per-store subtotal before checkout |
| Checkout surprise | Customer already saw the split |
| Vendor isolation invisible | Reinforces marketplace model |

---

## Grouping contract — client responsibility

**Input:** `GET /cart` response with items each carrying `storeId` (and ideally populated `store.name`).

**Output:** UI sections keyed by `storeId`:

1. Sort store groups alphabetically or by first item added — pick one, document in README.
2. Within each group, list line items with quantity controls.
3. Compute **store subtotal** = sum of `(price × quantity)` per line — use **current product price** from cart response (snapshot happens at checkout, not in cart view).

**You implement** — course specifies grouping rules, not component library choice.

---

## Edge cases to handle

| Case | Expected behaviour |
|---|---|
| Same store, multiple lines | One section, multiple rows |
| Single-vendor cart | One section — still grouped (not flat) |
| Empty cart | Empty state message + link to shop |
| Store name missing on populate | Fallback to "Store" + short storeId — fix populate in API if this happens often |

---

## Connection to Chapter 18

Checkout will `groupBy(storeId)` server-side. If the cart UI already groups by store, the **checkout review screen** (Chapter 19) is a natural extension — same shape, different CTA.

Mismatch — flat cart UI then split checkout — confuses customers and mentors reviewing your demo.

---

## ✅ Do / ❌ Don't

**✅ Do:** Group using `storeId` from API data — not product name heuristics.

**✅ Do:** Show unit alongside price (`₹120/kg`) — grocery domain from Chapter 3.

**❌ Don't:** Merge lines from different stores into one row.

**❌ Don't:** Compute totals from stale client cache — refetch cart after PATCH/DELETE.
