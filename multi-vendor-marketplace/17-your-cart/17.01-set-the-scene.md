# 17.1 — Set the scene

**Where the project is now:** Chapter 16 gave you a **persisted cart API** — `POST /cart`, `GET /cart`, `PATCH` and `DELETE` on items. Items carry `productId`, `storeId`, and `quantity`. The API is user-scoped and accepts multi-vendor lines. Your customer can browse the marketplace (Chapter 15) but has **no cart screen** and no header badge showing what's waiting.

**Where this chapter goes:** The cart gets a **face** — a dedicated page grouped by store, quantity controls wired to the API, add-to-cart from product detail, and a header count that survives reload.

This is a **frontend build chapter**. It pairs with Chapter 16 the way Chapter 7 paired with Chapter 6. The gate is **browser proof**: items from two stores visible, quantities update, cart persists after refresh.

## Common beginner questions

**Q: Why build the cart page after the API?**  
Same reason as login UI after auth API: the **contract exists** before the screen. You test endpoints with curl; then you connect buttons.

**Q: Should I group cart items in the API or in React?**  
**React groups for display** using `storeId` from each line. The API returns a flat `items` array; the UI builds sections. Checkout (Ch 18) groups server-side for order creation.

**Q: Grand total vs store subtotal — which price?**  
Cart shows **current product prices** from `GET /cart` populate. Orders freeze prices at checkout (Chapter 18 snapshots) — cart totals are estimates until then.

**Q: Badge count: lines or total quantity?**  
Pick one, document in README. Either "3 items" (sum of quantities) or "2 lines" (rows) — just be consistent.

**Q: Add to cart fails with 401 — what now?**  
Customer must log in first. Redirect to login, then return to product detail — optional stretch; minimum gate: login before add works.

**Q: Do I change the server this chapter?**  
No if Chapter 16 gate is complete. Chapter 17 is **client-only**.

## The gap between "cart in the database" and "cart in the customer's head"

Right now your marketplace can answer:

- What items does User X have in their cart? (API)
- Is stock sufficient to add? (API)

It cannot answer in the browser:

- How many items am I carrying?
- Which store is each line from?
- What is my subtotal per vendor before checkout?

Those are **presentation and interaction** problems — not persistence problems. Chapter 18 splits checkout by `storeId`; **Chapter 17 teaches the customer to see that split before they pay.**

## What "your cart" means in product terms

A grocery customer shops across **multiple local vendors** in one session. The cart page must:

| Concern | UI behaviour | API backing |
|---|---|---|
| Grouping | Sections per store name | `storeId` on each cart line |
| Line display | Name, unit, price, quantity | Populated product from `GET /cart` |
| Subtotal | Per-store and optional grand total | Client-side math from line prices |
| Quantity | Stepper or input → PATCH | `PATCH /cart/:itemId` |
| Remove | Button → DELETE | `DELETE /cart/:itemId` |
| Persistence | Reload keeps items | Server cart, not localStorage |

**Not client-side cart:** Browser memory alone breaks the Week 3 demo when the customer switches devices or clears cache.
