# 19.3 — What you'll build

Blueprint for **order tracking UI** — no new backend routes except optional status PATCH if not added in Ch 18.

Read this once before the build cluster (19.4 onward). You are not writing logic yet — you are learning **which files and routes** complete Week 3 in the browser.

---

## What changes in your repo

| Layer | This chapter adds |
|---|---|
| **Client pages** | Checkout, confirmation, customer history, vendor inbox |
| **Client components** | Order card, status badge, line row |
| **API helpers** | `checkout()`, `getMyOrders()`, `getVendorOrders()`, `updateOrderStatus()` |
| **Server (maybe)** | `PATCH /vendor/orders/:id` if missing from Ch 18 |

No new MongoDB collections. No Redis. No worker. Pure **UI on top of existing APIs**.

## Routes

| Path | Page | Auth |
|---|---|---|
| `/checkout` | Checkout | Customer |
| `/orders/confirmation` | OrderConfirmation | Customer — pass state or refetch |
| `/orders` | MyOrders | Customer |
| `/vendor/orders` | VendorOrders | Vendor |

Wire checkout CTA from Cart page (Ch 17) to `/checkout`.

Add nav links: customer **Orders** in header; vendor **Orders** in `VendorLayout`.

---

## User flows you are wiring

```
Customer:  /cart → /checkout → [Place order] → /orders/confirmation → /orders
Vendor:    /vendor/orders → [Mark packed] → customer refreshes /orders
```

## Optional API addition

## Common beginner questions

**Q: Do I rebuild the cart page?**  
No. Ch 17 cart stays. You add `/checkout` and wire the existing CTA.

**Q: Which page is hardest?**  
Usually checkout submit + confirmation state passing. Sub-chapters 19.5–19.6 walk each step.

**Q: Can vendor and customer share one login session for demo?**  
Use separate browser profiles or incognito windows — demo script 19.11 requires isolation proof.
