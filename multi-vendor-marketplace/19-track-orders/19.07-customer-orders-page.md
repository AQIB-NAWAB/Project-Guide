# 19.7 — Customer orders page

**Why this step matters:** Confirmation is a one-time screen — order history is where customers return days later. This page proves snapshot prices stay correct when vendors change the catalogue.

**Where the project is now:** Confirmation page shows orders after checkout (19.6). No persistent **order history** route — refresh or return later has no dedicated home.

**After this step:** `/orders` lists customer history from `GET /orders/my` — split orders as separate cards with snapshot line prices.

History is where the **price snapshot trap** (Ch 18) shows in UI. Every line reads `orderItems` embedded fields — never live catalog prices.

## Step 1 — API contract

```
GET /api/orders/my
Authorization: Bearer <customer-token>
```

**Middleware:** `requireAuth`, `requireRole('customer')` — Ch 18.15.

| Field | Use in UI |
|---|---|
| `_id` | Card key |
| `status` | `OrderStatusBadge` |
| `total` | Formatted currency |
| `createdAt` | Date/time |
| `storeId` / populated store | Store name heading |
| `orderItems[]` | `nameAtPurchase`, `priceAtPurchase`, `unitAtPurchase`, `quantity` |

Response: array or `{ orders: [...] }` — match vendor list envelope. Sort `{ createdAt: -1 }` on server.

---

## Step 2 — Page load contract

1. On mount → `getMyOrders()`.
2. Loading → spinner (not empty state).
3. Zero orders → empty state + link to `/shop`.
4. With orders → one `OrderCard` per order, newest first.
5. `401` → redirect login; 5xx → message + retry.

---

## Step 3 — OrderCard contract

| Data | Display |
|---|---|
| Store name | Card title |
| `createdAt` | Human-readable date |
| `status` | Badge |
| `total` | Prominent total |
| `orderItems` | `OrderLineRow` list or expandable section |

**Split orders:** Two orders from one checkout → **two cards** — do not merge. Match confirmation formatting (19.6).

Add **Orders** nav link; confirmation "View all orders" → `/orders`. Customer role only.
