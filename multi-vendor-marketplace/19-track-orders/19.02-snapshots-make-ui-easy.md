# 19.2 — Snapshots make UI easy

Order history UI would be fragile if every row required a live product fetch. **Snapshots mean the list endpoint is the only read** — no N+1 populate for price, no surprise when vendor deletes a product.

Chapter 18 embedded `nameAtPurchase`, `priceAtPurchase`, and `unitAtPurchase` on each order line for exactly this reason. Chapter 19 is where that design pays off in React.

---

## Why snapshots matter for beginners

| Approach | What goes wrong |
|---|---|
| Show `product.price` from live lookup | Vendor changes milk from ₹65 → ₹70 — last week's receipt shows ₹70 |
| Fetch `/products/:id` per line | 10-line order = 10 extra requests; product deleted → row breaks |
| Recompute total from catalogue | Order total changes when vendor edits — accounting nightmare |

**Snapshots freeze the truth at purchase time.** The order document is a **receipt**, not a live view of the catalogue.

---

## Render contract — order line row

| Display | Source field |
|---|---|
| Product name | `nameAtPurchase` |
| Unit price | `priceAtPurchase` + `unitAtPurchase` |
| Quantity | `quantity` |
| Line total | `lineTotal` or `priceAtPurchase * quantity` |

**Forbidden for price display:** `product.price` from populate.

---

## Order header row

| Display | Source |
|---|---|
| Store name | populated `storeId.name` or `store.name` |
| Order total | `order.total` |
| Date | `createdAt` |
| Status | `order.status` |

---

## Deleted or unpublished product

`productId` link optional — "View product" may 404. Receipt line still renders from snapshots.

This is a feature, not a bug. Customers keep proof of what they bought even if the vendor delists the item.

---

## Customer vs vendor same data shape

Both UIs read same `orderItems` structure. Vendor view adds customer name from safe populate on `customerId`.

You build **one** `OrderLineRow` component — reuse on confirmation, customer history, and vendor inbox.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Snapshot** | A copy of product name, price, and unit **at the moment of purchase** — stored on the order, not fetched later |
| **Embedded line items** | Order lines live **inside** the order document — not separate Product documents |
| **Populate** | MongoDB join — fine for store **name** on the header; **not** for line prices |

---

## Think of it like a paper receipt

A paper receipt prints the price **that day**. If the shop raises prices next week, your old receipt does not change. `orderItems` with snapshot fields are that paper receipt — digital, but same rule.

---

## Common beginner questions

**Q: Can I populate product for a photo on history rows?**  
Optional stretch. Gate does not require it. If you do, photo is cosmetic — price still comes from snapshots.

**Q: What if `nameAtPurchase` is missing on an old test order?**  
Re-seed or re-checkout. Your UI should handle missing fields gracefully (show "Unknown item") but gate data should have snapshots from Ch 18.

**Q: Why not just store `productId` and look up current price?**  
That was the **price snapshot trap** in Chapter 18.2 — the whole reason embedded snapshots exist.

## ✅ Do / ❌ Don't

**✅ Do:** TypeScript/JSDoc optional fields matching API — `nameAtPurchase` required on lines.

**✅ Do:** Format unit in grocery style — `₹120/kg`.

**❌ Don't:** Fetch `/products/:id` per line for history page.

**❌ Don't:** Recompute order total from catalogue.
