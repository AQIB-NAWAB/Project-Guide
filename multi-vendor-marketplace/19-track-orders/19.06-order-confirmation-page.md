# 19.6 — Order confirmation page

**Why this step matters:** Customers need immediate proof that checkout worked — especially when one payment creates **multiple orders**. This page is the receipt moment mentors look for in the Week 3 demo.

**After this step:** Customer sees **split orders** immediately after checkout — receipt-style summary per store.

---

## Route

`/orders/confirmation` — protected customer route.

**Data source:**

| Pattern | Acceptable |
|---|---|
| Location state from checkout navigation | `{ orders: [...] }` |
| Refetch `GET /orders/my?limit=2` | If state lost on refresh |

Prefer state + fallback refetch for refresh survival.

---

## Display contract

For **each order** in checkout response:

- Store name
- Order id (short display ok)
- Line items from **snapshots** — name, unit price, quantity, line total
- Order total
- Status badge — `pending`
- Link to `/orders` — full history

Heading copy: e.g. "Thank you — your orders are placed" (plural when N > 1).

---

## Multi-order messaging

Explain briefly: *Each store fulfils separately* — one sentence, editorial tone.

---

## Empty state

If user lands without data and refetch empty → link to shop.

---

## ✅ Do / ❌ Don't

**✅ Do:** Use snapshot fields only on lines.

**✅ Do:** Show count — "2 orders from 2 stores".

**❌ Don't:** Single combined receipt hiding split.

**❌ Don't:** Require product populate for display.

---

## Verify

- [ ] Two orders displayed after two-store checkout
- [ ] Totals match API
- [ ] Refresh with refetch still shows recent orders
