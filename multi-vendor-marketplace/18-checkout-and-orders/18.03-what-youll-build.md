## The three endpoints in plain English

| Route | Who calls it | What it does |
|---|---|---|
| `POST /checkout` | Logged-in customer | Cart → N orders (one per store), stock down, cart cleared |
| `GET /orders/my` | Customer | "Show my past orders" — snapshot prices only |
| `GET /vendor/orders` | Vendor | "Show orders for **my** store only" |

Checkout is the **big bang** — everything else is read-only lists. Do not build list routes before checkout works; there is nothing to list yet.

**Success picture:** Cart with items from Store A and Store B → one POST → response with **two orders**, each with embedded snapshot lines, cart empty, stock reduced.

## POST /checkout — behaviour contract

**Input:** Authenticated customer. Body empty or `{ note }` optional — cart is source of truth.

**Process (spec — you implement):**

1. Load user's cart with products; reject empty cart → 400.
2. Validate every line: product exists, published, stock ≥ quantity.
3. Group lines by `storeId`.
4. For each group, build Order document:
   - `customerId` = auth user
   - `storeId` = group key
   - `status` = `pending` (or your enum default)
   - `orderItems[]` = embedded snapshots per line
   - `total` = sum of snapshot line totals
5. Decrement stock **atomically** per product — fail entire checkout if any decrement fails.
6. Insert all orders — prefer transaction if replica set available; document if single-node.
7. Delete all cart items for user.
8. Return `{ orders: [...] }` — array length = number of stores in cart.

**Idempotency:** Out of scope for MVP — double-click may double order; note in README. Chapter 18.6 mentions Stripe keys as future read.

## Order line snapshot shape

```javascript
// orderItems[] element — illustrative
{
  productId: ObjectId,      // reference optional for reorder
  nameAtPurchase: String,
  priceAtPurchase: Number,
  unitAtPurchase: String,
  quantity: Number,
  lineTotal: Number         // priceAtPurchase * quantity
}
```

**You implement** — field names must match Order schema from Ch 5.

---

## GET /orders/my response contract

Array of orders for `customerId === auth.userId`. Each order includes embedded `orderItems` — no live product price. Populate `storeId` name optional for display.

---

## GET /vendor/orders response contract

Vendor role only. Orders where `storeId` belongs to auth vendor's store. Populate `customerId` with safe fields (name or email) — not password hash.

Cross-vendor: Vendor B must not see Vendor A's orders — 403 or empty, enforced by store ownership lookup.
