# 18.2 — Price snapshot trap

Store order lines as `{ productId, quantity }` only. Six months later the vendor changes basmati rice from ₹120/kg to ₹140/kg. Your order history page reloads and shows **₹140** for a purchase that happened at **₹120**. Accounting, customer trust, and your viva all fail.

This sub-chapter explains the trap **before** you implement the fix in 18.11.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Live join** | Loading order line, then fetching **current** product price from catalogue |
| **Snapshot** | Copying name, price, unit onto the order line **at checkout** — frozen forever |
| **Immutable order** | After creation, the receipt does not change when vendor edits the product |
| **Historical truth** | "What did the customer pay that day?" — must come from order, not product |

**The trap in one sentence:** Joining live product data into old orders **rewrites history**.

---

## The bug in one sentence

**Joining live product data into historical orders rewrites the past.**

---

## How teams accidentally build this

| Step | What goes wrong |
|---|---|
| Order stores only `productId` | Seems normalized — DRY |
| Order detail API populates Product | Shows current `name`, `price`, `unit` |
| Vendor edits price | Every old order display changes |
| Customer disputes charge | You have no frozen receipt |

MongoDB does not prevent this. SQL foreign keys do not prevent this. **Design** prevents this.

---

## Grocery makes it worse

From Chapter 3 — units matter:

- ₹120 **per kg** vs ₹120 **per piece**
- Weight-based quantity (2.5 kg)
- Perishable products renamed seasonally

Snapshot must capture at minimum:

| Field | Purpose |
|---|---|
| `nameAtPurchase` | Product title at checkout time |
| `priceAtPurchase` | Unit price at checkout time |
| `unitAtPurchase` | `kg`, `litre`, `piece`, etc. |
| `quantity` | How many units bought |
| `productId` | Optional link for "buy again" — not for pricing display |

Line total = `priceAtPurchase × quantity` — computed from snapshot fields only on read.

---

## What snapshots are not

| Misconception | Truth |
|---|---|
| "Denormalization is always bad" | Orders are **events** — events are immutable |
| "I'll version products instead" | Still need point-in-time copy on the order |
| "Snapshots duplicate data" | Yes — intentionally — receipts always duplicate |
| "Cart already has price" | Cart is mutable; order is committed |

---

## UI consequence (Chapter 19)

Order history UI reads **only embedded order line fields** — no `populate('productId')` for price display. Populate for product image link is optional; price display is not.

Chapter 19 sub-chapter 19.2 expands this for frontend.

## Test you will run (18.16)

1. Checkout product at price P1.
2. Vendor PATCHes product to price P2.
3. `GET /orders/my` still shows P1 on that line.

If it shows P2, snapshot implementation is wrong.

---

## ✅ Do / ❌ Don't

**✅ Do:** Copy name, price, unit from product document **inside checkout transaction** at commit time.

**✅ Do:** Display order totals from embedded lines only.

**❌ Don't:** Use `populate` on product for historical price in order detail.

**❌ Don't:** Recompute order total from live catalogue on read.

## Mentor interview angle

*"How do you handle price changes after an order is placed?"*

Strong answer: snapshot at purchase; order lines self-contained; product link for reference only.

Weak answer: "We use the product table" — describes the trap.

## Common beginner questions

**Q: Isn't copying price "bad normalization"?**  
On **Product**, yes — one price field. On **Order**, the copy **is** the receipt. Different entity, different rules.

**Q: Can I populate product for the product image on order history?**  
Optional for image/link — **never** for price or unit display on fulfilled orders.

**Q: What if the product is deleted later?**  
Snapshot fields still show what was bought. `productId` link may 404 — order line remains valid.

**Q: Should cart lines snapshot price too?**  
Not for MVP — cart is mutable intent. **Checkout** is the commit point (18.11).
