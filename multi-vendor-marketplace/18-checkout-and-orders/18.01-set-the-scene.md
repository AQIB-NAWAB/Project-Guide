**Where this chapter goes:** One `POST /checkout` turns the cart into **one order per vendor** — with **price snapshots** on every line, **atomic stock decrement**, and cart cleared on success. List routes for customer and vendor follow.

## In plain English

| Term | Simple meaning |
|---|---|
| **Checkout** | The server action that turns cart lines into **orders** |
| **Split orders** | One customer click → **multiple order documents** — one per vendor/store |
| **Price snapshot** | Copy of name, price, unit **at purchase time** — receipt never changes when vendor edits catalogue later |
| **Order line** | One purchased item embedded inside an order — with snapshot fields, not live product join |
| **Atomic stock decrement** | Reduce stock in one safe DB operation — two people cannot both buy the last item |
| **Fail-all checkout** | If any line fails stock check, **no orders created** and cart stays intact |
| **Immutability** | After order is created, historical prices on the order do not update |

## Think of it like one register, many stall receipts

The customer stands at **one checkout counter** with one basket (cart). The cashier (your server) looks inside, sorts items by stall (`storeId`), and prints **separate receipts** — Green Valley gets one, Harbor Market gets another. Each receipt lists exactly what that stall must pack, with prices **frozen at that moment**.

The customer walks away with one confirmation ("I ordered"). Each vendor sees **only their receipt** in their dashboard later.

## Common beginner questions

**Q: Why is checkout harder than cart?**  
Cart is mutable intent ("maybe I'll buy this"). Order is a **committed event** — money, stock, legal receipt. Wrong design breaks trust, accounting, and vendor fulfilment.

**Q: One order or many?**  
**Many orders** (one per store in cart) for this marketplace. A single combined order would hide lines from Vendor B.

**Q: Why snapshot price — can't we just read the product?**  
If the vendor changes price tomorrow, old orders would show the **new** price. Customers and vendors would both be angry. Snapshots copy price **once** at checkout.

**Q: When is stock reduced?**  
At **successful checkout**, atomically — not when adding to cart.

**Q: What if checkout fails halfway?**  
**Fail-all:** no partial orders, cart unchanged, clear error. Never leave "order created but stock not reduced."

**Q: Do I need Stripe or PayPal?**  
Not in this course. Checkout creates **orders**; payment integration is out of scope.

**Q: Double-click "Place order" — two order sets?**  
Known MVP limitation. Chapter 18.6 covers **idempotency** as reading; production uses idempotency keys.

**Q: Can checkout run in the browser without the server?**  
**No.** Client cannot be trusted to set prices, group by store, or decrement stock. All of that is server-side in this chapter.

## Extra FAQ — checkout complexity for beginners

**Q: Walk me through checkout in plain steps.**  
1) Load customer's cart with products. 2) Reject if empty. 3) Validate every line (published, enough stock). 4) Group lines by `storeId`. 5) For each group, build an Order with **snapshot** lines. 6) Decrement stock for each product (conditional update). 7) Save all orders. 8) Delete cart items. 9) Return the created orders.

**Q: What's the difference between cart `storeId` and order `storeId`?**  
Cart line `storeId` = "which stall this item belongs to." Order `storeId` = "this **whole order** is fulfilled by this stall." One order document per store; many lines inside it share that order's `storeId`.

**Q: Embedded order lines vs separate OrderLine collection?**  
This course **embeds** snapshot lines inside the Order document — one read gives a full receipt. Separate collections do not remove the need to snapshot price.

**Q: MongoDB transactions — do I need them?**  
Ideal when your MongoDB deployment supports multi-document transactions (replica set). Local standalone may not — document your approach: transaction **or** careful ordering + compensating rollback in learning log.

**Q: What should `GET /orders/my` return for price display?**  
Only **embedded snapshot fields** on order lines — never live `product.price` from populate.

**Q: Vendor sees customer orders how?**  
`GET /vendor/orders` — orders where `storeId` matches **that vendor's store**, with ownership check from Chapter 8.

**Q: I'm overwhelmed — what sub-chapters matter most?**  
Read **18.2** (snapshot trap), **18.3** (blueprint), then build in order 18.8→18.16. Do not skip **18.16** snapshot test — it catches the most common bug.

## The hardest commerce problem in this course

Multi-vendor marketplaces fail demos in predictable ways:

| Failure | Symptom |
|---|---|
| Single combined order | Vendor B never sees their lines |
| Live product price on old orders | Receipt changes when vendor edits price |
| Partial stock decrement | Overselling under concurrency |
| Cart not cleared | Double checkout |

Chapter 18 addresses all four in the **orders module** — not in React.

## One checkout, many orders

```
Customer cart (3 lines, 2 stores)
        │
        POST /checkout
        │
        ├─► Order A (store 1) — embedded lines with snapshots
        └─► Order B (store 2) — embedded lines with snapshots
```

Customer experience: one button. Vendor experience: each sees **only their order**. Fulfilment is per store.
