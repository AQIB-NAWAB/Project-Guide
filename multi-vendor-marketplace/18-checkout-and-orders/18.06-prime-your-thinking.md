# 18.6 — Prime your thinking

Before scaffolding the orders module, read how production commerce systems avoid **double charge** and **duplicate orders**. You will not implement payment in this course — but checkout shares the same **at-most-once** instinct.

---

## Extra FAQ before the build cluster

**Q: I'm scared of checkout — where do I start?**  
Read 18.3 blueprint. Build in order: scaffold module (18.9) → checkout skeleton (18.10) → snapshots (18.11) → stock (18.12) → clear cart (18.13) → list routes (18.14–15) → snapshot test (18.16). One sub-chapter per sitting.

**Q: Do I need payment before orders?**  
No. Order = "customer committed to buy." Payment is a separate integration. Your gate is correct order documents + stock + empty cart.

**Q: What's idempotency in one sentence?**  
Sending the same checkout request twice should not create two order sets — production uses idempotency keys; MVP documents double-click risk.

**Q: Transactions vs no transactions?**  
Transactions bundle "decrement stock + insert orders + clear cart" into all-or-nothing. If your local MongoDB lacks them, document compensating steps in learning log — do not silently skip fail-all semantics.

## Idempotency — concept for your README

**Problem:** Customer double-clicks "Place order". Two `POST /checkout` → two order sets.

**Production fix:** Client sends `Idempotency-Key: uuid`; server stores key → response mapping; replay returns same response.

**Course MVP:** Document as known limitation. Optional stretch: accept `Idempotency-Key` header and reject duplicate within 24h.

## Connection to atomic stock (18.5)

Idempotency prevents duplicate orders from **retries**. Conditional `$inc` prevents oversell from **concurrency**. Both matter in production; Chapter 18 gate requires **stock atomicity** — idempotency is stretch.
