# 18.4 — Normalization payoff

Chapter 18.2 showed the trap. This sub-chapter explains **why the fix is embedded snapshots** — not a separate `order_line_prices` table obsession, and not live joins.

---

## In plain English — when copying data is correct

Normalization means **one source of truth** for data that changes — product price lives on Product. **Orders break that rule on purpose** because an order is not "current state" — it is a **record of a past event**, like a printed receipt you file in a drawer. Receipts duplicate the price from the shelf label **at that moment**. Your order document does the same with `priceAtPurchase`, `nameAtPurchase`, `unitAtPurchase`.

**Rule of thumb:** If the business would say "but that's what it cost **that day**," snapshot it on the order.

---

## Orders are events, not current state

| Entity type | Mutability | Storage pattern |
|---|---|---|
| Product | Changes often | Normalized document / row |
| Cart | Changes until checkout | Referenced lines |
| Order | **Immutable after commit** | Embedded snapshot lines |

Normalization optimizes for **current truth**. Orders record **past truth**.

---

## Embedded vs referenced order lines in MongoDB

**Embedded (mandated for this course):**

```
Order {
  customerId, storeId, status, total,
  orderItems: [ { nameAtPurchase, priceAtPurchase, ... } ]
}
```

| Pro | Con |
|---|---|
| Single read for receipt | Order doc grows with line count |
| Atomic order document | Duplicate strings across orders |
| Matches grocery receipt mental model | Must copy fields at write time |

**Referenced lines collection:**

```
Order { ... }
OrderLine { orderId, productId, quantity }  // still need snapshots!
```

Extra collection without removing snapshot requirement — more joins, same copy step.

---

## What you still reference

| Field | Why keep Id |
|---|---|
| `productId` on line | "View product" if still published; analytics |
| `storeId` on order | Vendor scoping, filters |
| `customerId` on order | Customer history |

Never use `productId` populate for **price** on fulfilled orders.

---

## Copy moment — single source of truth at checkout

During `POST /checkout`, product document is read **once**:

```
const product = await Product.findById(line.productId)
snapshot.nameAtPurchase = product.name
snapshot.priceAtPurchase = product.price
snapshot.unitAtPurchase = product.unit
```

After insert, vendor may change `product.price` — order unaffected.

---

## Total field on order

Store `total` on order header at checkout — sum of `lineTotal` values. Do not recompute from live products on `GET`.

Optional: store `currency` if you internationalize later — out of scope now.

## ✅ Do / ❌ Don't

**✅ Do:** Treat checkout as the **only** copy-into-order step.

**✅ Do:** Validate product exists at checkout — reject deleted products.

**❌ Don't:** Backfill snapshots on read — race with price edits.

**❌ Don't:** Store only price snapshot without unit — grocery receipts need unit.
