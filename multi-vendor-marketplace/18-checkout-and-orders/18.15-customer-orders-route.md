# 18.15 — Customer orders route

**Why this step matters:** Customers need **order history** without re-running checkout — `GET /orders/my` returns embedded snapshot lines only, training the read path Chapter 19 UI will use.

**Where the project is now:** Vendor list route works; customer needs history without re-running checkout.

**After this step:** `GET /orders/my` returns all orders for authenticated customer, split orders visible as separate entries.

---

## Route contract

```
GET /api/orders/my
Authorization: Bearer <customer-token>
```

**Middleware:** `requireAuth`, `requireRole('customer')`.

---

## listOrdersForCustomer(userId) — spec

```
Order.find({ customerId: userId }).sort({ createdAt: -1 })
```

Populate `storeId` with store **name** for display — optional but helps Chapter 19 UI.

**Never populate product price** — display from `orderItems` snapshots only.

---

## Split orders appearance

One checkout creating two orders → customer history shows **two rows** — same `createdAt` roughly, different `storeId` / store name. UI may group "checkout session" later — not required for gate.

---

## Response contract

Array or `{ orders: [...] }` — match vendor route envelope style for client helper consistency.

Each order includes:

- `_id`, `status`, `total`, `createdAt`
- `store` or populated `storeId`
- `orderItems` with snapshot fields

---

## Isolation test

Customer A orders → Customer B `GET /orders/my` does not include A's orders.

---

## curl verify

```bash
curl -s http://localhost:5000/api/orders/my \\
  -H "Authorization: Bearer <customer-token>"
```

After two-store checkout, expect **two orders** in array.
