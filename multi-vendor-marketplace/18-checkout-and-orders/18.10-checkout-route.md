# 18.10 — Checkout route

**Why this step matters:** This is the **spine** of checkout — load cart, group by `storeId`, create order shells. Snapshots and stock come next, but grouping must work first or multi-vendor fulfilment never happens.

**Where the project is now:** Orders module scaffolded; checkout handler empty.

**After this step:** `POST /checkout` loads cart, groups by `storeId`, creates **one Order document per group** — snapshots and stock in 18.11–18.12.

---

## checkout(userId) — phase 1 contract

Implement grouping and order shell first; add snapshots and stock next sub-chapters.

**Steps:**

1. Fetch all cart items for `userId` with product populated.
2. If empty → `400` `{ message: 'Cart is empty' }`.
3. Build `Map<storeId, lines[]>`.
4. For each store group, construct order payload:
   - `customerId`, `storeId`, `status: 'pending'`
   - `orderItems` — placeholder copies from cart lines (refine in 18.11)
   - `total` — sum of line totals from **current** product price (snapshot in 18.11)
5. `Order.insertMany(orders)` — or one-by-one inside transaction.
6. Return `{ orders: createdOrders }`.

**Defer to 18.12–18.13:** stock decrement, cart clear — or stub with TODO if building incrementally.

---

## Response contract

**Success 201:**

```json
{
  "orders": [
    { "_id": "...", "storeId": "...", "total": 305, "orderItems": [...] },
    { "_id": "...", "storeId": "...", "total": 40, "orderItems": [...] }
  ]
}
```

Array length equals distinct `storeId` count in cart.

## Validation before insert

| Check | Failure |
|---|---|
| Product missing | 400 |
| Product unpublished | 400 |
| quantity < 1 | 400 |
| storeId mismatch product.storeId | 400 — data integrity |

```bash
curl -s -X POST http://localhost:5000/api/checkout \\
  -H "Authorization: Bearer <customer-token>" \\
  -H "Content-Type: application/json" \\
  -d '{}'
```

**Expected:** JSON with **2 orders** if cart had two stores. Inspect in Compass — two Order documents, different `storeId`.
