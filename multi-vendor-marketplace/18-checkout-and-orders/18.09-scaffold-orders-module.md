# 18.9 — Scaffold orders module

**Why this step matters:** Orders deserve their own module — checkout logic will grow fast. Scaffolding routes + service + mount **before** implementation gives you a named home for the most complex handler in Week 3.

**Where the project is now:** Order Mongoose model may exist; no checkout or list routes mounted.

**After this step:** `orders.routes.js`, `orders.service.js`, and validation schema stub exist; routes mounted on app — handlers may return 501 until 18.10.

---

## Files to create

```
server/src/modules/orders/
  orders.routes.js
  orders.service.js
  orders.schema.js    ← optional Zod/Joi for checkout body
```

---

## routes.js contract

Mount paths (adjust prefix to match `/api` pattern):

| Method | Path | Middleware | Handler |
|---|---|---|---|
| POST | `/checkout` | requireAuth, requireRole('customer') | checkoutHandler |
| GET | `/orders/my` | requireAuth, requireRole('customer') | listMyOrdersHandler |
| GET | `/vendor/orders` | requireAuth, requireRole('vendor') | listVendorOrdersHandler |

Wire in `app.js`:

```javascript
// import ordersRouter from './modules/orders/orders.routes.js'
// app.use('/api', ordersRouter)  // or mount subpaths per your convention
```

**You implement** — match existing module mount style from cart and products.

---

## service.js exports — spec

| Function | Purpose |
|---|---|
| `checkout(userId)` | Full checkout pipeline |
| `listOrdersForCustomer(userId)` | Customer history |
| `listOrdersForVendor(userId)` | Resolve vendor store → orders |

Handlers stay thin — delegate to service.

## orders.schema.js

Checkout body may be empty `{}`. If you accept optional fields:

```javascript
// illustrative — note field only, no full schema
{ deliveryNote?: string }
```

Reject unknown customerId in body — cart owner is auth user only.
