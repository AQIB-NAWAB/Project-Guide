# 18.5 — Atomic stock decrement

Two customers checkout the last bag of rice at the same time. Without atomic decrement, both succeed and you oversell by one. Grocery stock is often fractional and low — races are real even in class demos.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Race condition** | Two requests read the same stock, both think they can buy, both succeed — oversell |
| **Atomic** | One indivisible database operation — "decrement only if stock ≥ quantity" |
| **Conditional update** | Update that **fails** if stock is no longer sufficient — `$gte` in filter |
| **Fail-all** | If any product in checkout fails stock, **abort entire checkout** — no half orders |

**Think of it like:** One physical bag of rice on the shelf. Two hands grab at once — only one can take it. The database must enforce **one winner**, not "both read 1 in stock and both leave happy."

---

## The race in timeline form

```
Stock = 1

Customer A read stock 1  ─┐
Customer B read stock 1  ─┤ both think OK
A writes stock 0         ─┤
B writes stock 0         ─┘  sold 2 units with stock 1
```

**Fix:** Decrement only if stock still sufficient at write time — single atomic operation.

---

## Mongoose pattern — conditional update

**Spec (you implement):**

```javascript
// Pseudocode — one product line
const result = await Product.findOneAndUpdate(
  { _id: productId, stockQty: { $gte: quantityNeeded } },
  { $inc: { stockQty: -quantityNeeded } },
  { new: true }
)
if (!result) throw insufficientStock
```

`$gte` in filter makes decrement **conditional**. If another request took stock first, `result` is null — checkout must **abort entirely**.

---

## Fail-all checkout semantics

| Partial success | Course rule |
|---|---|
| Order A created, Order B fails stock | **Reject** — do not leave partial orders |
| Stock decremented, order insert fails | Roll back — use transaction or manual compensating increment |

**Minimum gate:** If any line fails stock check, return **409 or 400** with clear message; **no orders created**; cart unchanged.

Transactions: `mongoose.startSession()` + `withTransaction` when MongoDB deployment supports it. Local standalone MongoDB may lack multi-doc transactions — document your approach in learning log.

---

## Order of operations

Safe sequence inside checkout:

1. Validate all lines (existence, stock) — read phase
2. Begin transaction if available
3. Conditional decrement **each** distinct product
4. Insert all orders
5. Clear cart
6. Commit

Decrement before insert prevents orders without stock reduction. Insert before clear cart prevents empty cart with no orders.

---

## Duplicate products in cart

Same `productId` twice in cart (two lines) vs merged quantity — normalize to **per-product total quantity** before decrement to avoid double `$inc`.

---

## Vendor restock is separate

Stock increment on vendor edit is Chapter 11/12 — not checkout. Checkout only **decrements**.

---

## ✅ Do / ❌ Don't

**✅ Do:** Return which product failed stock if possible — helps UI.

**✅ Do:** Test with two curl checkouts against stock 1 — one must fail.

**❌ Don't:** Read stock in JS, write later without condition — race remains.

**❌ Don't:** Create orders then decrement — oversell window exists.
