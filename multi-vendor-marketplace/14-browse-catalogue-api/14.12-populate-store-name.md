# 14.12 — Populate store name

**Why this step matters:** Shop cards need **store name**, not a raw ObjectId. This step fixes the N+1 trap from 14.2 — populate on the **paginated page** keeps query count flat when Chapter 15 hammers this endpoint.

**Where the project is now:** Paginated list returns products with `storeId` ObjectId only — not enough for shop cards.

**After this step:** Each item includes `store: { id, name }`; query count verified **not N+1**.

---

## Populate spec

Add to find query in `listPublicProducts`:

```javascript
.populate({
  path: 'storeId',
  select: 'name slug',
  model: 'Store'  // if needed
})
```

Apply **after** `.limit()` on the product query — only populate products on current page.

---

## Response mapping

Transform populated document to public DTO:

| Internal after populate | Public JSON |
|---|---|
| `storeId: { _id, name, slug }` | `store: { id, name }` |

Remove raw `storeId` string from response if nested `store` present — cleaner for Chapter 15.

**Serializer location:** service layer or `toPublicProduct()` helper — reuse list + detail.

---

## Broken ref handling

If `storeId` points to deleted store:

| Strategy | Pick one |
|---|---|
| Exclude product from list | Filter post-query or aggregation match |
| Return product with `store: null` | Client shows "Unknown store" |
| 404 on detail only | List excludes |

Document choice in README.

## N+1 verification — required for gate

**Enable debug in dev:**

```javascript
mongoose.set('debug', true)
```

**Run:**

```bash
curl -s "http://localhost:5000/api/products?limit=20" > /dev/null
```

**Count MongoDB operations in log:**

| Result | Pass? |
|---|---|
| ~2 queries (find products + find stores) | ✅ |
| 21+ queries | ❌ fix loop lookup |

Disable debug before commit or gate to env flag.

Record count in learning log: *"20 products, X queries"*.
