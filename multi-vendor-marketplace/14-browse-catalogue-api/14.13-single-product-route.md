# 14.13 — Single product route

**Why this step matters:** The product detail page (Chapter 15) needs one item by id — with the **same public DTO** as the list. Detail route also enforces "unpublished = 404" so customers cannot guess draft product ids.

**Where the project is now:** Public list with filters, pagination, populated store.

**After this step:** `GET /products/:id` returns one public product or 404 — same DTO shape as list item.

---

## Route contract

| Field | Value |
|---|---|
| Method | GET |
| Path | `/products/:id` |
| Auth | None |
| Params | `id` — MongoDB ObjectId |

---

## Visibility rules

Apply same base filter as list:

```javascript
{
  _id: id,
  isPublished: true,
  stockQty: { $gt: 0 }
}
```

| Case | Status |
|---|---|
| Valid published in-stock | 200 + product |
| Unpublished / draft | 404 |
| Zero stock | 404 |
| Invalid ObjectId format | 400 or 404 — document |
| Unknown id | 404 |

**Recommended:** 404 for unpublished — do not reveal draft exists.

---

## Response shape

Match list item — include optional extra fields for detail page:

| Field | List | Detail |
|---|---|---|
| name, price, unit, category, imageUrl | ✅ | ✅ |
| description | optional omit | ✅ if on schema |
| store nested object | ✅ | ✅ |
| stockQty | ✅ | ✅ |

Chapter 15 detail page consumes this endpoint.

---

## Handler spec

```javascript
// getPublicProductById(id)
// 1. Validate ObjectId
// 2. findOne with publicFilter + _id
// 3. populate storeId (same as list)
// 4. map toPublicProduct
// 5. null → 404
```

One query + populate — not separate store fetch.

## Route ordering

Ensure `GET /products/:id` does not capture `/products` literal paths like `/vendor/products` — mount paths correctly on router.

If using sub-router at `/products`, detail is `/:id` on that router; vendor routes stay at `/vendor/products`.
