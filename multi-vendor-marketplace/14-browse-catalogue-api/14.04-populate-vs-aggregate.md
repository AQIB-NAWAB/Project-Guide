# 14.4 — Populate vs aggregate

Mongoose gives two ways to attach store data to product results: **`.populate()`** on a find query and **`$lookup`** in an aggregation pipeline. For FreshMarket's public catalogue, populate is the default — aggregate when you need one round trip or complex shaping.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Reference field** | Product stores `storeId` — a pointer to the Store document, not the store name copied inline |
| **Populate** | "Follow the pointer and attach the store's name" — Mongoose batches this efficiently |
| **Aggregate / `$lookup`** | A pipeline query that joins collections inside MongoDB in one request |
| **DTO shape** | What JSON looks like to the client — e.g. nested `store: { id, name }` not raw `storeId` ObjectId |

**Why this sub-chapter exists:** Product cards need **"Sold by Green Valley"**. The product document only stores **which** store — not the name string. You must **join** store data for display without the N+1 loop from 14.2.

---

## Populate — course default

Use Mongoose `.populate()` on the `storeId` reference when listing products:

| Setting | Value |
|---|---|
| Path | `storeId` |
| Select | `name` (and `slug` if you use it in URLs) |
| Applied on | `Product.find(publicFilter)` before skip/limit |

| Pros | Cons |
|---|---|
| Readable; matches Mongoose refs | Often 2 queries (find + stores batch) |
| Easy to add later fields | Harder to filter on store fields in one query |
| Familiar in CRUD codebases | Less control over nested shape |

**When to use here:** `GET /products` list and detail with store **name** only — populate satisfies gate.

---

## Aggregate with `$lookup` — stretch only

Alternative: single aggregation pipeline with `$lookup` from `products` to `stores`, then `$unwind`, `$project`, skip/limit.

| Pros | Cons |
|---|---|
| Single round trip to MongoDB | Verbose pipeline; ObjectId typing errors common |
| Filter/sort on joined fields in one pass | Count + pagination need parallel pipeline |
| Performance at very large scale | Overkill if populate passes query-count gate |

**When to use:** Mentor stretch or measured slowness — **not required** if populate stays at ~2 queries.

---

## N+1 vs populate vs loop

| Pattern | Queries for 20 products |
|---|---|
| Loop `Store.findById` | 21 ❌ |
| `.populate('storeId')` | ~2 ✅ |
| `$lookup` aggregate | 1 ✅ |

**Not N+1:** populate's second query batches all store ids — not one query per product in the common case.

Read Mongoose populate docs on **lean()** + populate if you return plain objects — affects serializer.

---

## Response shaping

Populate returns:

```json
"storeId": { "_id": "...", "name": "Green Valley" }
```

Public API spec wants:

```json
"store": { "id": "...", "name": "Green Valley" }
```

**You implement** mapping in service or toJSON transform — consistent across list and detail.

---

## Detail route `GET /products/:id`

Same populate path as list item — one product, one store:

```javascript
Product.findOne({ _id, ...publicFilter })
  .populate('storeId', 'name slug')
```

Do not fetch store in separate handler call after product — same N+1 class of mistake for single item (only 2 queries but inconsistent pattern).

---

## Fields to select on Product

Public list — omit heavy or internal fields:

| Include | Exclude |
|---|---|
| name, price, unit, category, imageUrl, stockQty | vendor notes, cost |
| store.name | store.ownerId, store internal address if sensitive |

Address on store may be public for marketplace — course choice; document in README.

---

## ✅ Do / ❌ Don't

**✅ Do:** Use same `publicFilter` object for findOne detail and list.

**✅ Do:** Handle missing store (deleted store, broken ref) — exclude product or 404 — document behaviour.

**❌ Don't:** Populate full Store with vendor email — customer browse leak.

**❌ Don't:** Mix aggregate list with populate detail without matching DTO shapes.
