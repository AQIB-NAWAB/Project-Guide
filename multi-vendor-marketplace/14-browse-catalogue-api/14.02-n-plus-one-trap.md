# 14.2 — The N+1 trap

A catalogue endpoint that returns 50 products and **51 database queries** looks fine in development with 5 products. It fails under real traffic — latency spikes, connection pool exhaustion, MongoDB CPU climbs.

This sub-chapter explains **N+1** in the context of `GET /products` with store names. You will **not** ship the naive pattern — you will verify query count in 14.12 and 14.14.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Query** | One trip from your server to MongoDB to fetch data |
| **N+1** | 1 query for the list + **1 extra query per row** — the "+1" repeats N times |
| **Round trip** | Server asks MongoDB → waits → gets answer. Many round trips = slow page |
| **Populate** | Mongoose shortcut to fetch related docs (store names) without a manual loop |

**Beginner intuition:** If your shop page shows 20 products and each product triggers its own "what store is this?" question to the database, you asked **21 questions** when **2** would do.

---

## What N+1 looks like

**Naive handler pseudocode:**

```javascript
// ❌ Do not ship this pattern
const products = await Product.find({ isPublished: true })
for (const p of products) {
  p.store = await Store.findById(p.storeId)  // +1 query per product
}
return products
```

| Products returned | Queries |
|---|---|
| 20 | 1 + 20 = **21** |
| 100 | **101** |

Customer shop loads page 1 with limit 20 → **21 round trips** to MongoDB per request. Multiply by concurrent users.

---

## Why store name needs a join

Product card spec (Chapter 15): show **store name** under product title. Product document has `storeId` ref — not `storeName` string (denormalization trap from Chapter 5).

You must **resolve** store for display — either:

- **Populate** in one query (Mongoose)  
- **Aggregate** with `$lookup`  
- **Denormalize** `storeName` on product (write-time copy — not this chapter)

Course gate: **populate or aggregate** — fix N+1; verify count.

---

## Symptoms in production

| Symptom | Cause |
|---|---|
| p95 latency grows with catalogue size | Linear query count |
| MongoDB slow query log full of identical `findOne` on stores | Loop lookup |
| Works in dev, slow with seed | Seed too small to notice |

---

## The fixed pattern (concept)

**One product query + one populate:**

```javascript
// ✅ Pattern direction — you implement
Product.find(filter)
  .populate('storeId', 'name slug')
  .limit(limit)
  .skip(skip)
```

Mongoose issues **at most 2 queries** for populate on homogeneous refs (products + stores batch) — not N+1.

**Alternative:** `$lookup` in aggregation pipeline — one query total. Harder to read; valid for gate if query count is 1.

---

## How you will verify (14.14)

Enable Mongoose debug or log:

```javascript
mongoose.set('debug', true)  // dev only
```

Or count queries with middleware / test hook.

**Gate expectation:** Listing 20 products uses **O(1)** store queries relative to page size — typically 2 queries with populate, not 21.

Document actual count in learning log.

---

## Pagination interaction

N+1 hurts **per page**, not total catalogue size. Page size 20 with N+1 = 21 queries **every browse**. Pagination without fixing N+1 just caps damage — still unacceptable.

Fix populate **before** shipping UI — Chapter 15 will hammer this endpoint.

---

## ✅ Do / ❌ Don't

**✅ Do:** Populate in the same service method that handles list — not in a React loop calling API per product.

**✅ Do:** Select only needed store fields — `name`, maybe `slug` — not full store document with vendor PII.

**❌ Don't:** Fetch store per product in a `for` loop.

**❌ Don't:** Add `storeName` to every product to "fix" N+1 without understanding write amplification — optional later, not default.

**❌ Don't:** Assume populate is free — still 2 queries; acceptable vs N+1.
