# 14.5 — Pagination

A marketplace catalogue without pagination returns every product in one response — fine with 12 seed items, catastrophic with 12,000. Chapter 14 implements **offset pagination** with `page` and `limit` query params.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Page** | Which slice of results — page 1 = first batch, page 2 = next batch |
| **Limit** | How many items per page — default 20 in this course |
| **Skip** | How many rows MongoDB skips before returning — `(page - 1) × limit` |
| **Total / totalPages** | Metadata so the UI knows "47 products, 3 pages" without loading all 47 |

**Think of it like:** A grocery flyer with **page numbers** — you read page 1 of vegetables, flip to page 2, not one infinite scroll of every tomato in the country.

**Why beginners need this:** Chapter 15's shop will call `GET /products?page=1&limit=20` on load. Without pagination, the API and browser both choke as the catalogue grows.

---

## Offset pagination contract

| Param | Meaning |
|---|---|
| `page` | 1-based index |
| `limit` | Items per page |
| `skip` | `(page - 1) * limit` |

**Example:** `?page=3&limit=20` → skip 40, return items 41–60.

---

## Response metadata

Every list response includes:

| Field | Source |
|---|---|
| `items` | Current page documents |
| `page` | Echo requested page (after validation) |
| `limit` | Echo effective limit (after cap) |
| `total` | `countDocuments(filter)` |
| `totalPages` | `Math.ceil(total / limit)` |

Client (Chapter 15) uses `totalPages` for page controls or "load more" math.

---

## Defaults and caps

| Rule | Value |
|---|---|
| Default `page` | 1 |
| Default `limit` | 20 |
| Max `limit` | 100 |

If client sends `limit=500`, clamp to 100 — do not error unless you prefer 400; document choice.

If `page` exceeds `totalPages` → **200** with `items: []` — not an error.

---

## Count + find — same filter

```javascript
// Pseudocode — two operations, same match object
const filter = buildPublicFilter(query)
const [total, items] = await Promise.all([
  Product.countDocuments(filter),
  Product.find(filter).populate(...).skip(skip).limit(limit)
])
```

**Bug to avoid:** count without `isPublished` filter but find with it — wrong `totalPages`.

## Sort order

Spec a stable default sort — document in README:

| Option | Example |
|---|---|
| Newest first | `{ createdAt: -1 }` |
| Name A–Z | `{ name: 1 }` |
| Price low–high | `{ price: 1 }` |

Gate does not mandate sort — pick one and stay consistent. Chapter 15 UI may expose sort later.

---

## Performance note

Offset pagination slows on very deep pages (`page=5000`) — MongoDB still scans skipped docs. Acceptable for course scale. Chapter 14.6 discusses **cursor** pagination for high-traffic catalogues — not required for gate.

---

## Invalid params

| Input | Response |
|---|---|
| `page=0` | 400 |
| `page=abc` | 400 |
| `limit=-1` | 400 |
| `limit=0` | 400 or treat as default — document |

Use consistent validation — reuse schema from vendor routes if you have Zod/Joi.

---

## ✅ Do / ❌ Don't

**✅ Do:** Run count and find in parallel with `Promise.all` when independent.

**✅ Do:** Return `total: 0` and `items: []` for empty catalogue — valid marketplace state.

**❌ Don't:** Return all products when pagination params omitted — apply defaults.

**❌ Don't:** Use different filters for count vs find.
