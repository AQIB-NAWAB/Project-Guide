# 14.1 — Set the scene

**Where the project is now:** Week 2 vendor experience is complete — stores, products, photos. All product reads so far are **vendor-scoped** (`GET /vendor/products`). Customers have accounts (Chapters 6–7) but no way to browse the market.

**Where this chapter goes:** A **public catalogue API** — `GET /products` returns published, in-stock items from all vendors, with filters, pagination, and store name populated **without N+1 queries**.

This is a **build chapter**. Sub-chapters 14.8 onward add routes and query logic. The gate is **runnable proof**: curl returns paginated catalogue; query count stays flat when listing many products.

## In plain English

| Term | Simple meaning |
|---|---|
| **Public catalogue** | The shop window — products any visitor can browse without being a vendor |
| **Vendor-scoped route** | API that only returns **your** inventory when **you** are logged in as vendor |
| **N+1 query problem** | Loading a list, then running one extra database query **per row** — fast with 5 items, slow with 500 |
| **Populate** | Mongoose fetching related documents (e.g. store name) in a batched way instead of a loop |
| **Pagination** | Returning products in pages (e.g. 20 at a time) instead of the whole catalogue at once |
| **DTO** | The shape of JSON the API sends — public browse hides draft products and vendor secrets |

## Common beginner questions

**Q: Do I delete vendor routes or the vendor dashboard?**  
No. You **add** public read routes. Vendors keep `GET /vendor/products` for their own inventory.

**Q: Why API before the shop UI (Chapter 15)?**  
Same pairing as Chapters 6→7 and 11→12: **backend contract first**, React second. Building the grid before `GET /products` exists forces fake hardcoded products you throw away.

**Q: Can customers browse without logging in?**  
Yes for this chapter's gate — public `GET /products` needs no JWT. Cart and checkout (Chapters 16–18) require login.

**Q: What's the difference between `/vendor/products` and `/products`?**  
`/vendor/products` = "show **my** products" (vendor token). `/products` = "show **everyone's published, in-stock** products" (no token).

**Q: I only have products from one store in seed — is that OK?**  
Finish Week 2 seed with **two stores** before the gate. Multi-vendor browse is the whole point of a marketplace course.

**Q: What is N+1 and why should I care before I have users?**  
It is a pattern bug: 20 products → 21 database trips instead of 2. Dev feels fine; production does not. Chapter 14 teaches you to **measure** query count before shipping the shop UI.

## The gap between "vendor inventory" and "marketplace"

Right now your API can answer:

- What products does **this vendor** sell?
- Can Vendor A edit Vendor B's product?

It cannot answer:

- What can **any customer** browse without logging in (or with customer login)?
- How do we filter by category or store?
- How do we page through 200 tomatoes without loading all at once?

Those are **public read model** problems — different from vendor CRUD in Chapter 11.

## Public vs vendor routes — mental split

| Audience | Route pattern | Auth | Data scope |
|---|---|---|---|
| Vendor | `GET /vendor/products` | Required vendor JWT | Own store only |
| Customer/public | `GET /products` | Optional none for browse | All published, in-stock |
| Vendor write | `POST/PATCH/DELETE /products` | Vendor + ownership | Own only |

Never expose draft products or internal fields on public DTO.
