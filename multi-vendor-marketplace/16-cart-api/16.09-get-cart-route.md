# 16.9 — Get cart route

**Why this step matters:** GET /cart is what Chapter 17's page will render — populated product and store names avoid N+1 and give the UI everything for grouped display without extra fetches per line.

**Where the project is now:** POST /cart creates lines; no list endpoint.

**After this step:** `GET /cart` returns current user's items with populated product and store — ready for Chapter 17 UI grouping.

---

## GET /cart contract

| Field | Value |
|---|---|
| Method | GET |
| Path | `/cart` |
| Auth | Required |

**Query:** none — always current user's cart from JWT.

---

```javascript
// getCartForUser(userId)
```

Apply N+1 discipline — populate both refs in one find, not loops.

---

## Response shape

```json
{
  "items": [
    {
      "id": "cartLineId",
      "quantity": 2,
      "storeId": "674a...",
      "product": {
        "id": "...",
        "name": "Organic Tomatoes",
        "price": 3.5,
        "unit": "kg",
        "imageUrl": "https://...",
        "stockQty": 10
      },
      "store": {
        "id": "...",
        "name": "Green Valley Grocer"
      }
    }
  ]
}
```

Include raw `storeId` on line for checkout grouping in Ch 17 UI — even if `store` nested present.

**Subtotal:** optional `{ subtotal: 7.0 }` — sum of price * quantity; exclude if not implemented yet.

---

## Empty cart

**200** with `{ items: [] }` — not 404.

---

## Deleted product handling

If productId broken ref:

| Strategy | Pick |
|---|---|
| Omit line + log | cleanup job |
| Return line with product: null | UI shows remove |
| Auto-delete orphan line | aggressive |

Document choice — gate prefers no 500 on GET.
