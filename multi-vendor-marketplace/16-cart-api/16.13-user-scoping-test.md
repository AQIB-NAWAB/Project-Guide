# 16.13 — User scoping test

**Why this step matters:** Cart isolation is a **security gate** — User B must never read or mutate User A's itemIds. This explicit test prevents IDOR bugs that are embarrassing in demo and dangerous in production.

**Where the project is now:** Cart CRUD implemented; isolation may be untested explicitly.

**After this step:** Documented proof that **User A cannot read or mutate User B's cart** — IDOR/BOLA class bug prevented.

---

## Threat model

Cart items are **object-level resources**. Knowing another user's `itemId` (MongoDB ObjectId) must not grant access.

Attack pattern:

1. Customer A adds item — note `itemId` from response  
2. Customer B uses A's `itemId` in PATCH or DELETE  
3. **Must fail** — 404 or 403  

Same for GET /cart — only own lines returned.

---

## Test matrix

| # | Actor | Action | Expected |
|---|---|---|---|
| 1 | Customer A | POST two products (2 stores) | 201/200 |
| 2 | Customer A | GET /cart | 2 items, A's userId in Compass |
| 3 | Customer B | GET /cart | empty or B-only items |
| 4 | Customer B | PATCH A's itemId | 404 or 403 |
| 5 | Customer B | DELETE A's itemId | 404 or 403 |
| 6 | Customer A | GET /cart after B attack | unchanged 2 items |
| 7 | No token | GET /cart | 401 |

Record itemId from step 1 for steps 4–5.
