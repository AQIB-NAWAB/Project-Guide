# 16.12 — Stock check on add

**Why this step matters:** Adding 999 tomatoes when only 3 exist should fail **now**, not at checkout — early rejection is clearer UX and protects checkout from obviously invalid carts.

**Where the project is now:** Stock may be partially validated in POST/PATCH.

**After this step:** Centralized stock validation — reject add/update when `stockQty` insufficient or zero; shared helper used by POST and PATCH.

---

## Stock rules — canonical spec

| Operation | Rule |
|---|---|
| POST /cart | product.stockQty must be ≥ requested quantity; reject if stockQty === 0 |
| PATCH increase | new quantity ≤ product.stockQty |
| PATCH decrease | always OK if quantity ≥ 1 |
| GET /cart | no stock mutation — may include stockQty in product for UI warnings |

Stock decremented at **checkout** (Chapter 18) — not when adding to cart.

---

## Shared helper contract

Create in `cart.service.js`:

```javascript
// assertSufficientStock(product, requestedQuantity) → throws or returns error
// message: "Insufficient stock" / "Out of stock"
```

Call from:

- `addToCart` — after product load  
- `updateCartItem` — after product load  

Single message shape — `{ message: "Insufficient stock", available: 5 }` optional.

---

## Edge cases

| Case | Behaviour |
|---|---|
| Product OOS after in cart | GET still shows line; PATCH up may fail; checkout Ch 18 fails |
| Unpublished after add | GET may still show — checkout rejects; stretch: prune on GET |
| quantity exactly stock | allow |
| quantity stock + 1 | 400 |

---

## POST duplicate add with upsert

If upserting quantities:

```
existing 2 + add 3 = 5
stock 4 → reject entire operation or cap — document
```

**Recommended:** reject if sum > stock — clear error.
