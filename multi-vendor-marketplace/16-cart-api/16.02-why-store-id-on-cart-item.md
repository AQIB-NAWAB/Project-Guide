# 16.2 — Why storeId on cart item

Checkout in Chapter 18 creates **one order per vendor** from a single customer checkout click. That split is trivial if every cart line already knows which store it came from.

This sub-chapter explains why **`storeId` is stored on CartItem** at add time — not derived at checkout from product lookup alone.

---

## Multi-vendor cart — the idea in one paragraph

FreshMarket is **not** Amazon shipping from one warehouse. It is **many independent vendors** sharing one website. The customer carries **one cart** across the whole market. Each line remembers **which vendor** sells that item via `storeId`. At checkout, the server sorts lines into piles by `storeId` and prints **one order per pile**. Without `storeId` on each line, the server would have to re-investigate every product at checkout — slow, fragile, and easy to get wrong.

---

## In plain English

| Term | Simple meaning |
|---|---|
| **Denormalize** | Copy a fact (storeId) onto the cart line even though product already has it — on purpose |
| **Group by storeId** | Sort cart lines into buckets — all lines with same storeId belong together |
| **Validate at add** | Server checks `product.storeId === body.storeId` — client cannot lie about vendor |
| **Fulfillment** | Which vendor packs and delivers — determined by `storeId`, not customer address (MVP) |

---

## The checkout split problem

Customer cart contains:

- Milk from Store A  
- Eggs from Store A  
- Salmon from Store B  

**Desired:** 2 orders — A gets milk+eggs, B gets salmon.

**Without storeId on cart line:**

```javascript
// ❌ Fragile at checkout
for (const line of cart) {
  const product = await Product.findById(line.productId)
  const storeId = product.storeId  // extra query; product may have moved stores
}
```

| Risk | Consequence |
|---|---|
| Product changed store (rare admin bug) | Line groups wrong |
| Product deleted | Checkout breaks |
| N queries at checkout | Latency under load |
| Race: product unpublished | Checkout fails late |

**With storeId on line at add time:**

```javascript
// ✅ Group in memory
const byStore = groupBy(cartItems, 'storeId')
```

One pass; checkout service trusts line snapshot of vendor context.

---

## Add-to-cart contract preview

`POST /cart` body:

```json
{
  "productId": "...",
  "storeId": "...",
  "quantity": 2
}
```

Server **validates** `product.storeId === body.storeId` — reject mismatch (400) — prevents client tampering to wrong vendor bucket.

---

## Relationship to product ref

Cart line typically stores:

| Field | Purpose |
|---|---|
| `userId` | Scoping — whose cart |
| `productId` | What to buy |
| `storeId` | Who fulfills — **denormalized copy** |
| `quantity` | How many |

`storeId` duplicates product.storeId **intentionally** — same denormalization pattern as order snapshots, lighter weight than full price snapshot.

Chapter 18 adds price snapshot on **order** — not on cart line for MVP.

## UI implication (Chapter 17)

Product detail stub already has `product.store.id` — POST cart must send it. Shop grid could add-to-cart later with same field from list API `store.id`.

## Multi-vendor demo requirement

Gate expects cart API holding lines with **two different storeId values** for one user — proves split key works before checkout UI.

---

## ✅ Do / ❌ Don't

**✅ Do:** Validate storeId matches product at add time.

**✅ Do:** Index `{ userId: 1, productId: 1 }` for duplicate-line upsert logic if you merge quantities.

**❌ Don't:** Omit storeId and rely on checkout lookup only.

**❌ Don't:** Store store **name** on cart line — populate at read time or use product+store populate.

## Mentor interview angle

*"How does one checkout become multiple vendor orders?"*

Strong answer starts with cart lines grouped by storeId stored at add time; checkout creates one order per group; price snapshot on order not cart.
