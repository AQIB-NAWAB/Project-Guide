## In plain English

| Term | Simple meaning |
|---|---|
| **Cart** | The customer's saved list of "I want to buy these" before checkout |
| **Cart line / cart item** | One row: a product, how many, which store sells it |
| **Multi-vendor cart** | **One basket** with items from Shop A **and** Shop B — not two separate carts |
| **`storeId` on each line** | A copy of which vendor fulfills that item — checkout uses it to split orders later |
| **User scoping** | Every cart query filters by **logged-in user's id** — you never see someone else's basket |
| **Stock check on add** | Server reads current `stockQty` when adding — does not reserve stock until checkout (Ch 18) |
| **Persisted cart** | Saved in the database — survives refresh, different tab, tomorrow |

## Think of it like one shopping basket at a real market

At a farmers' market you carry **one basket**. You pick tomatoes from Stall A and bread from Stall B. You do **not** get a second basket per stall.

FreshMarket works the same:

```
Your one cart
├── Tomatoes   (Green Valley stall)
├── Milk       (Green Valley stall)
└── Salmon     (Harbor Market stall)
```

At checkout (Chapter 18), the system splits into **two orders** — one per stall — but the customer still clicked **one** "Place order" button. That split only works if each cart line already knows its **`storeId`**. Chapter 16 stores that key when the customer adds an item.

## Common beginner questions

**Q: Why not store the cart in React state or localStorage?**  
Browser-only carts vanish on refresh, fail across devices, and cannot be checked securely at checkout. Production carts live on the **server**, tied to the logged-in user.

**Q: Do I need two carts for two vendors?**  
**No** — that is the beginner mistake. One cart, many vendors. **`storeId` on each line** tells the server how to split later.

**Q: Why save `storeId` if the product already has `storeId`?**  
You could look it up at checkout — but that adds queries, breaks if a product moves stores, and makes grouping fragile. Copy `storeId` at **add time** and validate it matches the product.

**Q: When does stock actually decrease?**  
**Checkout (Chapter 18)**, not add-to-cart. Add only **checks** there is enough stock right now.

**Q: Can vendors use the cart?**  
Yes — anyone with a customer account. Same API. Vendors shopping as buyers use the same cart routes.

**Q: Guest cart without login?**  
Out of scope for MVP. Customer must log in before `POST /cart`. Chapter 16.4 explains merge-on-login as optional reading.

**Q: Why API-only this chapter — no cart page yet?**  
So you can test user isolation and multi-vendor adds with **curl** before debugging React. Chapter 17 is the UI pair.

## The gap between "browse" and "buy"

Browse answers *what exists*. Cart answers *what this customer intends to purchase* — across **multiple vendors** in one basket.

Without `storeId` on each line, checkout would re-query products to guess vendor grouping — fragile and slow. Chapter 16 embeds the split key at add time.
