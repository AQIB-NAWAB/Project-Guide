# 16.3 — What you'll build

By chapter end, authenticated customers have a **persisted cart** — add, list, update quantity, remove lines — with **storeId on each item**, **stock checks on add**, and **strict user scoping**.

Blueprint for 16.6–16.13. Specs only.

---

## Before you scaffold — what "cart API" means

You are **not** building a checkout yet. You are building **saved intent**:

| Action | HTTP | Plain English |
|---|---|---|
| Add item | POST /cart | "Put this in my basket" |
| View cart | GET /cart | "What's in my basket?" |
| Change qty | PATCH /cart/:itemId | "I want 3 not 2" |
| Remove | DELETE /cart/:itemId | "Remove this line" |

Every route requires **login** — the server knows **which user** owns the cart from the JWT, never from the request body.

**Gate story to aim for:** Customer adds tomato from Store A and fish from Store B → `GET /cart` shows **two lines, two storeIds**. Customer B's token cannot see or edit Customer A's lines.

## API contract summary

| Method | Path | Auth | Purpose |
|---|---|---|---|
| POST | `/cart` | Required | Add or increment line |
| GET | `/cart` | Required | List current user's items |
| PATCH | `/cart/:itemId` | Required | Update quantity |
| DELETE | `/cart/:itemId` | Required | Remove line |

All routes scope by **`req.user.userId`** (or your JWT payload field) — never trust userId from body.

---

## POST /cart body

```json
{
  "productId": "674b...",
  "storeId": "674a...",
  "quantity": 1
}
```

| Validation | Error |
|---|---|
| productId invalid | 400 |
| storeId invalid | 400 |
| quantity < 1 | 400 |
| product not found | 404 |
| product not published or OOS | 400 or 404 — document |
| product.storeId !== storeId | 400 "Store mismatch" |
| quantity > stockQty | 400 "Insufficient stock" |

**Success 201 or 200:** Return created/updated cart item or full cart — document shape; recommend full cart or item DTO consistent with GET.

## GET /cart response

```json
{
  "items": [
    {
      "id": "cartItemId",
      "quantity": 2,
      "product": {
        "id": "...",
        "name": "Organic Tomatoes",
        "price": 3.5,
        "unit": "kg",
        "imageUrl": "...",
        "stockQty": 10
      },
      "store": {
        "id": "...",
        "name": "Green Valley Grocer"
      },
      "storeId": "..."
    }
  ],
  "subtotal": 7.0
}
```

**Populate** productId and storeId — avoid N+1 (same discipline as Ch 14). Subtotal optional for gate — sum price * quantity if included.

## PATCH /cart/:itemId

Body:

```json
{ "quantity": 3 }
```

| Case | Status |
|---|---|
| Own item, valid qty ≤ stock | 200 |
| qty 0 or negative | 400 — use DELETE instead |
| qty > stock | 400 |
| itemId not found | 404 |
| item belongs to other user | 404 or 403 — **must not update** |

---

## DELETE /cart/:itemId

| Case | Status |
|---|---|
| Own item | 200 or 204 |
| Other user's item | 404 or 403 |
| Unknown id | 404 |

## User isolation — hard requirement

Every query includes `{ userId: currentUser }`:

```javascript
// find cart items WHERE userId = req.user.id
// update/delete WHERE { _id: itemId, userId: req.user.id }
```

**Gate test 16.13:** User A token cannot GET or PATCH User B's itemId.

---

## Stock check on add/update

Read **current** `product.stockQty` from DB at write time — not cached client value.

| Rule | Behaviour |
|---|---|
| stockQty 0 | reject add |
| requested qty > stock | reject |
| PATCH increases above stock | reject |

Decrement stock at **checkout** Ch 18 — not on add to cart.

## Verification story (16.13)

1. Customer A adds product from Store 1 and Store 2 — GET cart shows 2 lines, 2 storeIds  
2. PATCH quantity on one line  
3. DELETE one line  
4. Customer B GET cart — empty or B's items only; A's itemIds return 403/404 for B  
5. Add OOS product — 400

## ✅ Do / ❌ Don't

**✅ Do:** requireAuth on every cart route.

**✅ Do:** Validate storeId against product at POST.

**❌ Don't:** Accept userId in POST body.

**❌ Don't:** Snapshot price on cart line — Ch 18 order embed.
