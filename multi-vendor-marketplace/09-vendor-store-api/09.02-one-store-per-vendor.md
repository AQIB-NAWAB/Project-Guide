# 9.2 — One store per vendor

A vendor who runs **Green Valley Organics** should not also appear as **Green Valley Organics (2)** because they clicked "create store" twice. Multi-store per vendor is a real product pattern — but **not** for this course. FreshMarket assumes **one local shop per vendor account**.

---

## Why one store matters for grocery

Local grocery marketplaces sell **trust in a named shop**:

| Multi-store per vendor | One store per vendor |
|---|---|
| Customer sees duplicate names in search | One clear storefront per vendor |
| Product ownership ambiguous — which store? | Product → Store → Vendor chain stays simple |
| Checkout split logic harder | Cart groups by `storeId` — one store per vendor simplifies Week 3 |
| Onboarding UX branches ("pick a store") | Linear path: register → open shop → add products |

You are not building a franchise dashboard. You are building **"I run one neighbourhood shop on this platform."**

---

## Where enforcement belongs

Uniqueness can be attempted in three places — use **all that apply**:

| Layer | Mechanism | Strength |
|---|---|---|
| **Database** | Unique index on `vendorId` in stores collection | Hard guarantee — race-safe with unique constraint |
| **Service** | Query existing store before insert | Clear 409 Conflict message for API consumers |
| **Route** | `requireRole('vendor')` + service check | Correct HTTP semantics |

**✅ Do:** Unique compound or single-field index on `vendorId` in the Store schema (Chapter 5 may already have this — verify).

**❌ Don't:** Rely only on UI disabling the button — API must reject duplicate `POST /stores`.

---

## HTTP semantics for duplicate create

When a vendor who already owns a store calls `POST /stores` again:

| Status | Meaning | Body hint |
|---|---|---|
| **409 Conflict** | Business rule violation — store already exists | `{ "message": "Vendor already has a store" }` or similar |
| 400 Bad Request | Wrong if payload is valid but rule fails | Avoid — 409 is clearer |

**Not 403** — the vendor is allowed to create a store; they simply already have one.

**Not 500** — duplicate key from MongoDB should be caught and mapped to 409 in the service layer.

---

## Edge cases to spec now

| Scenario | Expected behaviour |
|---|---|
| New vendor, no store | `POST /stores` succeeds → 201 + store document |
| Vendor with store | `POST /stores` → 409 |
| Customer calls `POST /stores` | 403 from `requireRole` |
| Unauthenticated `POST /stores` | 401 from `requireAuth` |
| Vendor deletes store (if you add delete later) | Out of scope this chapter — no delete route |

This course does **not** implement store deletion in Chapter 9. Assume stores are permanent once created.

---

## Relationship to `GET /stores/me`

"One store per vendor" makes **`GET /stores/me`** natural:

- Returns **200** + single store document when vendor has a shop  
- Returns **404** when vendor has no store yet — Chapter 10 uses this to trigger onboarding redirect  

**Alternative:** Return `200` with `{ "store": null }` — pick one contract and document it in README. This course recommends **404 for no store** so the client can branch cleanly.

---

## Seed data implication

If your Chapter 5 seed created **two vendors with two stores**, you already have the happy path. For duplicate-create testing:

- Use a **fresh vendor account** registered in Chapter 6/7  
- Create store once → success  
- `POST /stores` again → 409  

Or use seed vendor and attempt second create — same result.

## Comparison to real platforms

| Platform pattern | FreshMarket choice |
|---|---|
| Etsy: one shop per account (simplified) | ✓ aligned |
| Amazon Seller: many storefronts | ✗ out of scope |
| Shopify: multiple locations per merchant | ✗ out of scope |

Defend in viva: *"One store keeps cart split, ownership checks, and onboarding linear for an MVP grocery marketplace."*

---

## ✅ Do / ❌ Don't

**✅ Do:** Enforce uniqueness in service **and** unique index on `vendorId`.

**✅ Do:** Return 409 with a human-readable message — UI will show it in Chapter 10.

**✅ Do:** Document the one-store rule in API README or `docs/AUTHORIZATION.md` store section.

**❌ Don't:** Allow `vendorId` to be sent in the request body — derive from JWT only.

**❌ Don't:** Create a second store "for testing" and leave orphan data — use 409 test then stop.
