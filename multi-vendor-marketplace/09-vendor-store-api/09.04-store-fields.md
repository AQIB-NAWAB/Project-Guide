# 9.4 — Store fields

The Store document is the vendor's **shop card** — what customers see when browsing by store and what vendors edit in settings. Chapter 9 implements a focused field set aligned with your Chapter 5 schema. This sub-chapter specs **what each field means**, not the Mongoose code.

---

## Core fields (this chapter)

| Field | Type (conceptual) | Purpose | Validation notes |
|---|---|---|---|
| `name` | string | Shop display name | Required on create; reasonable max length (e.g. 80 chars) |
| `description` | string | Editorial intro — "Family-run organic since 1998" | Optional; trim whitespace |
| `address` | string or embedded object | Where the shop is — pickup/delivery trust | If string: single line OK for MVP; if object: `{ line1, city, postcode }` per ERD |
| `logoUrl` | string (URL) | Avatar in UI | Optional; must be URL format if present; no binary in MongoDB |
| `vendorId` | ObjectId ref → User | Owner | Set server-side from JWT; immutable after create |
| `createdAt` / `updatedAt` | dates | Audit | Mongoose timestamps |

Match your **`docs/erd.md`** or draft — if you planned `slug`, `phone`, or `openingHours`, note them as **future** unless you add now consistently in POST/PATCH validation.

---

## Name vs slug (if you have both)

| | `name` | `slug` |
|---|---|---|
| **Shown to users** | "Maya's Market" | Used in URLs: `/shop/mayas-market` |
| **Uniqueness** | Can duplicate across regions (confusing but allowed) | Should be unique platform-wide |
| **Chapter 9** | Required | Optional — generate from name if you add slug |

If you skip `slug` in Chapter 9, routes use MongoDB `_id` in `PATCH /stores/:id` — fine for MVP. Chapter 14 public catalogue may add slug later.

**✅ Do:** If you add `slug`, generate server-side on create — never trust client uniqueness.

**❌ Don't:** Add `slug` without unique index — collisions break URLs.

---

## Address: string vs structured

**String (MVP):**

```
"12 High Street, Bristol BS1 4ST"
```

Pros: simple form in Chapter 10. Cons: hard to filter by city later.

**Structured (better long-term):**

```
{ line1, line2, city, postcode, country }
```

Pick what your ERD committed. Chapter 9 POST/PATCH must accept the same shape your model uses.

---

## `logoUrl` without upload yet

Chapter 9 stores a **URL string only** — vendor may paste a placeholder or leave empty. Chapter 13 adds Cloudinary upload for **product** images; store logo upload is optional stretch.

| Approach | Chapter 9 |
|---|---|
| Empty `logoUrl` | Show initials avatar in UI — Chapter 10 |
| Paste external URL | Validate URL format |
| Upload file | Not yet — no multer on store route |

## Fields that belong elsewhere

| Field | Why not on Store |
|---|---|
| `productCount` | Derived — count products query |
| `averageRating` | Reviews — out of scope |
| `commissionRate` | Platform admin — out of scope |
| `isVerified` | Admin flag — out of scope |
| Vendor email/phone | User/Profile document — single source of truth |

**❌ Don't:** Denormalize `vendorName` onto Store — duplication drifts when vendor renames profile (Chapter 5 warned this).

---

## Mutable vs immutable

| Field | Create | PATCH |
|---|---|---|
| `vendorId` | Set from JWT | **Immutable** — reject body attempts |
| `name` | Required | Allowed |
| `description` | Optional | Allowed |
| `address` | Per your rules | Allowed |
| `logoUrl` | Optional | Allowed |
| `_id` | Auto | Never in body |

---

## Validation layer (`stores.schema.js`)

Define **request shapes** separate from Mongoose schema:

- `createStoreSchema` — fields allowed on POST  
- `updateStoreSchema` — partial fields on PATCH; strip `vendorId`  

Return **400** with field-level errors for validation failures — consistent with auth module patterns from Chapter 6.

---

## Response shape to client

Return the **store document** JSON — exclude internal fields if any (`__v` is harmless). Include `_id` for PATCH from settings page in Chapter 10.

Do not leak other vendors' data in `GET /stores/me` — scoped query by `vendorId` from token only.

---

## Grocery-specific copy (editorial)

Encourage vendors to write descriptions that fit **local grocery**:

- What they specialize in (organic, halal, farm-direct)  
- Delivery radius or pickup hours — even as plain text in `description` until structured hours exist  

Product-level `unit` and `isPerishable` live on **Product** — not Store.

---

## ✅ Do / ❌ Don't

**✅ Do:** Align POST/PATCH validation with Mongoose schema — mismatches cause 500s.

**✅ Do:** Trim string fields on write — avoid `"  "` names.

**❌ Don't:** Store image binary or base64 in MongoDB — URL only.

**❌ Don't:** Let client set `vendorId` — IDOR vector.
