## Request/response contracts (spec level)

### `POST /stores`

**Body (example shape — you define exact validation):**

| Field | Required | Notes |
|---|---|---|
| `name` | yes | Display name — "Maya's Market" |
| `description` | optional | Short editorial blurb |
| `address` | yes or optional per your ERD | Pickup/delivery context |
| `logoUrl` | optional | String URL; upload in later chapter |

**Response 201:** Store document including `_id`, `vendorId`, timestamps.

**Must not accept:** `vendorId` in body — set from `req.user.id` (or your JWT claim name).

### `GET /stores/me`

**Response 200:** Store document owned by caller.

**Response 404:** No store for this vendor yet.

### `PATCH /stores/:id`

**Body:** Partial update — any subset of mutable store fields.

**Response 200:** Updated document.

**Response 403:** Store exists but `vendorId` ≠ caller (via `assertStoreOwner`).

**Response 404:** Store id not found.

## Middleware stack per route

| Route | Middleware order |
|---|---|
| All store routes | `requireAuth` |
| POST, GET me, PATCH | `requireRole('vendor')` |
| PATCH `/:id` | + ownership check on `storeId` param |

Reuse Chapter 8 helpers — do not duplicate ownership logic inline in every handler.

---

## Ownership chain (reminder)

```
JWT userId (vendor)
    └── Store.vendorId
            └── Product.storeId (Chapter 11)
```

Store creation sets `vendorId` once at insert. Products inherit `storeId` from the vendor's single store.

## Testing plan (9.12)

You will prove with curl:

1. Register/login vendor A → token A  
2. `POST /stores` with token A → 201  
3. `GET /stores/me` with token A → 200, same `_id`  
4. `POST /stores` again with token A → 409  
5. Register/login vendor B → token B  
6. `PATCH /stores/:idA` with token B → 403  
7. `PATCH /stores/:idA` with token A → 200  

Save redacted curl transcript or notes in learning log — not necessarily committed as a script.
