# 9.10 — Get my store route

## Why this step matters

Vendors should not hunt MongoDB ids in Compass. `GET /stores/me` resolves the store from the JWT — and returns **404** when no shop exists yet. That 404 is not an error; Chapter 10 turns it into the onboarding redirect signal.

---

**Where the project is now:** Vendor can create a store via POST.

**After this step:** `GET /api/stores/me` returns the authenticated vendor's store — **200** with document, or **404** when they have not opened a shop yet. Chapter 10 uses this contract for the onboarding gate.

---

## Why `/me` exists

| Without `/me` | With `/me` |
|---|---|
| Client must store `_id` after POST | Client asks "do I have a shop?" anytime |
| Onboarding state unclear after refresh | Single source of truth from API |
| Risk client sends wrong id | Server scopes by JWT only |

---

## Handler flow (spec)

```
GET /api/stores/me
  1. requireAuth
  2. requireRole('vendor')
  3. getStoreByVendorId(userId)
  4. if null → 404
  5. else → 200 + store JSON
```

No `:id` param — no IDOR surface on this route.

---

## Service: `getStoreByVendorId(vendorId)`

Single query:

- `Store.findOne({ vendorId })`  
- Return document or null — do not throw for missing  

Route maps null → 404.

---

## 404 contract (important for Ch 10)

When vendor has **no** store:

| Choice | Status | Body |
|---|---|---|
| **Recommended** | 404 | `{ "message": "Store not found" }` |
| Alternative | 200 | `{ "store": null }` |

**Pick recommended 404** unless you update Chapter 10 client handling accordingly. Document choice in README.

---

## Response 200

Full store document — same shape as POST 201 response. Client can populate settings form from this payload.

---

## Error matrix

| Condition | Status |
|---|---|
| No token | 401 |
| Customer role | 403 |
| Vendor, no store | 404 |
| Vendor, has store | 200 |

---

## curl verify

**Vendor with store** (after 9.9):

```bash
curl -s http://localhost:5000/api/stores/me \
  -H "Authorization: Bearer TOKEN"
```

Expect 200; `_id` matches POST response.

**Vendor without store** (fresh account, no POST):

Expect 404.

**Customer token:**

Expect 403.

---

## Route ordering reminder

`GET /me` must be registered **before** any `GET /:id` you add later. Express matches in registration order.

## ✅ Do / ❌ Don't

**✅ Do:** Query by `vendorId` from JWT only.

**✅ Do:** Return 404 for no store — enables clean onboarding branch.

**❌ Don't:** Return empty 200 with null without documenting — confuses fetch error handling.

**❌ Don't:** Expose other vendors' stores on this route — ever.
