# 9.12 — Test with curl

## Why this step matters

The store API must work without React before you build forms. curl proves create → fetch me → update → cross-vendor 403 in one session. If this step fails, Chapter 10 will hide the bugs behind loading spinners.

---

**Where the project is now:** POST, GET me, and PATCH implemented.

**After this step:** You have run the **full vendor store lifecycle** twice — happy path and isolation path — and recorded results for the learning log.

## Test script (spec — you execute)

### Phase 1 — Vendor A creates and reads store

| Step | Request | Expect |
|---|---|---|
| 1 | Login vendor A → `TOKEN_A` | 200 + token |
| 2 | `GET /stores/me` with A (if no store yet) | 404 |
| 3 | `POST /stores` with A — valid body | 201; note `STORE_ID_A` |
| 4 | `GET /stores/me` with A | 200; same `_id` |
| 5 | `POST /stores` again with A | 409 |

### Phase 2 — Vendor A updates store

| Step | Request | Expect |
|---|---|---|
| 6 | `PATCH /stores/STORE_ID_A` with A — change `name` | 200 |
| 7 | `GET /stores/me` with A | 200; new `name` |

### Phase 3 — Isolation

| Step | Request | Expect |
|---|---|---|
| 8 | Login vendor B → `TOKEN_B` | 200 |
| 9 | `PATCH /stores/STORE_ID_A` with B | **403** |
| 10 | `GET /stores/me` with B | 404 or 200 (if B has own seed store) |

### Phase 4 — Auth boundaries

| Step | Request | Expect |
|---|---|---|
| 11 | `POST /stores` no header | 401 |
| 12 | Customer token `POST /stores` | 403 |

## Common failures

| Symptom | Likely cause |
|---|---|
| POST 500 on duplicate | E11000 not mapped to 409 |
| GET me 200 with wrong store | Query not filtering by vendorId |
| PATCH 200 for vendor B | Ownership check missing or wrong id field |
| 404 on PATCH for owner | Wrong STORE_ID or invalid ObjectId |
| 401 always | Bearer prefix missing; token expired |
