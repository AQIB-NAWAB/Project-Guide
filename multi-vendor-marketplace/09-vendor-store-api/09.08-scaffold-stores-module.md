# 9.8 — Scaffold stores module

## Why this step matters

Scaffolding separates **structure** from **logic** — routes, service, and validation files exist and mount before you write create/fetch/update behavior. Same pattern as auth (Ch 6) and products (Ch 11). One folder, one responsibility per file.

---

**Where the project is now:** Store **model** exists; routes may be a Chapter 8 stub or empty.

**After this step:** `stores.routes.js`, `stores.service.js`, and `stores.schema.js` exist; router mounted on `app.js`; handlers are stubs or thin wrappers ready for 9.9–9.11.

---

## Target module layout

```
server/src/modules/stores/
  stores.model.js      ← already exists
  stores.routes.js     ← NEW or replace stub
  stores.service.js    ← NEW
  stores.schema.js     ← NEW
```

**You implement** — course specifies responsibilities, not full file contents.

## File responsibilities

| File | Owns |
|---|---|
| `stores.routes.js` | Express `Router`; HTTP verbs; middleware chain; call service |
| `stores.service.js` | `createStore`, `getStoreByVendorId`, `updateStore`; DB calls |
| `stores.schema.js` | Validate POST/PATCH bodies; export validators or schemas |
| `stores.model.js` | Mongoose schema — minimal changes unless index missing |

Confirm no duplicate mount from Chapter 8 stub — one mount only.

## Route skeleton (spec)

Register routes in this order:

| Order | Method | Path | Handler stub |
|---|---|---|---|
| 1 | POST | `/` | `createStoreHandler` |
| 2 | GET | `/me` | `getMyStoreHandler` |
| 3 | PATCH | `/:id` | `updateStoreHandler` |

**`/me` before `/:id`** — critical if you add `GET /:id` later.

Each route: `requireAuth` → `requireRole('vendor')` → validation → handler.

PATCH adds ownership middleware or in-handler `assertStoreOwner`.

---

## Service skeleton (spec)

Export async functions — no `req`/`res` in service:

| Function | Returns / throws |
|---|---|
| `createStore(vendorId, data)` | Store doc; throws conflict if store exists |
| `getStoreByVendorId(vendorId)` | Store doc or null |
| `updateStore(storeId, vendorId, data)` | Updated doc; throws forbidden / not found |

Map thrown errors to HTTP status in routes or shared error middleware.

---

## Schema module (spec)

| Export | Validates |
|---|---|
| `validateCreateStore(body)` | POST fields; strip unknown keys |
| `validateUpdateStore(body)` | PATCH partial; reject `vendorId` |

Invalid → 400 with `{ errors: { field: message } }` shape consistent with auth module.

## ✅ Do / ❌ Don't

**✅ Do:** Keep handlers thin — three to eight lines calling service.

**✅ Do:** Export router as default from `stores.routes.js`.

**❌ Don't:** Put Mongoose calls directly in route handlers long-term.

**❌ Don't:** Mount stores at `/stores` without `/api` if rest of app uses `/api` — client breaks in Ch 10.
