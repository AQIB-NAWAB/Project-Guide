# 9.1 — Set the scene

**Where the project is now:** Chapters 6–8 gave you identity and isolation — vendors register, log in, and your API knows who they are. `requireAuth` returns 401 without a valid token. `requireRole('vendor')` returns 403 for customers. `assertStoreOwner` blocks cross-vendor store edits. But a verified vendor still has **no shop** in the database — only a user record with `role: vendor`.

**Where this chapter goes:** Vendors create and manage **their store** through the API — one store per vendor, owned by the authenticated user, protected by the isolation middleware from Chapter 8.

This is a **build chapter**. Every sub-chapter from 9.7 onward adds routes and service logic to your repo. The gate is **runnable proof**: a vendor can create a store, fetch it, update it, and a second vendor cannot PATCH the first vendor's store.

---

## Bridge from last chapter

Chapter 8 installed **server-side permission rules** — `requireRole('vendor')`, `assertStoreOwner`, and a stub `PATCH /stores/:id` that returns **403** when Vendor B attacks Vendor A's store. You proved isolation with curl; you committed `docs/AUTHORIZATION.md`. But a new vendor still has **no store document** — only a user row with `role: vendor`. Chapter 9 is the first **vendor feature** that uses those guards for real: create shop, fetch mine, update mine. Authorization before features — not the other way around.

## In plain English

| Term | Simple meaning |
|---|---|
| **Store** | A vendor's shop on the marketplace — name, address, description |
| **One store per vendor** | Each seller gets exactly one shop; duplicates rejected at create |
| **`POST /stores`** | API call that opens a new shop (once per vendor) |
| **`GET /stores/me`** | Fetch **my** shop without guessing MongoDB ids |
| **`vendorId`** | Field linking a store document to the User who owns it |
| **Service layer** | Business logic separate from HTTP routes — finds store by vendor |
| **404 on GET /stores/me** | No shop yet — Chapter 10 turns this into an onboarding redirect |

## Common beginner questions

**Q: Why API before the "Open your shop" UI in Chapter 10?**  
Same pattern as auth: Chapter 6 built the API, Chapter 7 built the screens. curl proves the contract before React forms hide bugs behind spinners.

**Q: Can a vendor have two stores?**  
Not in this course. One vendor, one store — enforced at `POST /stores`. Multiple shops per owner is a different product decision.

**Q: Why `GET /stores/me` instead of `GET /stores/:id`?**  
Vendors should not need to copy their MongoDB `_id` from Compass. "Me" resolves the store from the JWT — simpler and safer.

**Q: Do customers use these routes?**  
Not for creating stores. Customers browse products in Week 3. These routes are **vendor-only**, guarded by `requireRole('vendor')`.

**Q: What if I skipped Chapter 8?**  
Stop. Store routes without ownership checks are an IDOR vulnerability on day one. Finish the 403 test first.

## The gap between "logged in vendor" and "open shop"

Right now your marketplace can answer:

- Who is this user? (JWT + role)
- Can Vendor B edit Vendor A's stub store? (403 — Chapter 8)

It cannot answer:

- How does a new vendor **open their first shop**?
- What happens if they try to create a **second** store?
- How does the vendor fetch **their own** store without knowing the MongoDB `_id`?

Those are **store lifecycle** problems — the first real vendor-facing feature after auth.

---

## Why this chapter sits after authorization

Chapter 8 was deliberate: you built **role checks and ownership helpers** before writing store CRUD. Without that foundation, `PATCH /stores/:id` becomes an IDOR vulnerability on day one.

Chapter 9 assumes:

- **`docs/AUTHORIZATION.md`** committed from Chapter 8
- **`requireRole`** and **`assertStoreOwner`** exist in `shared/` or `auth/`
- **Store model** exists from Chapter 5 with `vendorId` (or equivalent owner ref)
- You tested **Vendor B → 403 on Vendor A's store** with a stub route

If Chapter 8 gate is incomplete, finish the IDOR test first — store routes inherit the same checks.

## What "vendor store API" means in product terms

A store is the vendor's **public face** on the marketplace:

| Concern | API responsibility |
|---|---|
| Shop exists | `POST /stores` — vendor creates once |
| Shop identity | Name, description, address, logo URL |
| One vendor, one shop | Enforce at create — reject duplicate |
| Vendor sees own shop | `GET /stores/me` — no ID guessing |
| Vendor updates own shop | `PATCH /stores/:id` — ownership check |

Customers will browse products **by store** in Week 3. Vendors cannot list products until a store exists — Chapter 10 enforces that in the UI; Chapter 9 builds the API.
