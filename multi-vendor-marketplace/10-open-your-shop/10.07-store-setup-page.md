# 10.7 — Store setup page

## Why this step matters

This is the first form a new vendor sees after registration — their shop's birth certificate. It wires `POST /stores` to human input: name, description, address. Errors must be readable; success must navigate somewhere sensible. Week 2 demo starts here in the browser.

---

**Where the project is now:** `OpenShop.jsx` stub inside `VendorLayout`; `api/stores.js` exports `createStore` signature.

**After this step:** Storeless vendor submits **Open your shop** form → `POST /api/stores` → success navigates to vendor home; errors show inline or in banner.

---

## Page purpose

`OpenShop` is **first-time** shop creation — not settings. Single responsibility: collect fields and POST once.

---

## UI contract

| Element | Spec |
|---|---|
| Route | `/vendor/open-shop` |
| Title | "Open your shop" |
| Helper | One shop per vendor on [ProductName] |
| CTA | "Create shop" |

### Form fields

| Field | Type | Client validation |
|---|---|---|
| `name` | text | required; trimmed non-empty |
| `description` | textarea | optional |
| `address` | text | required if API requires |
| `logoUrl` | url text | optional; basic URL pattern |

Mirror Ch 9 server validation — client improves UX; server is authoritative.

### UX states

| State | Display |
|---|---|
| idle | Form ready |
| submitting | Disabled CTA + "Creating…" |
| success | Brief toast optional → redirect |
| error | Banner or per-field; values retained |

---

## API contract — `createStore(data)`

```
POST /api/stores
Authorization: Bearer <vendor-token>
Body: { name, description?, address, logoUrl? }
```

| Status | Behaviour |
|---|---|
| 201 | Refetch GET me or use body; navigate `/vendor` |
| 400 | Map `errors` to fields |
| 409 | "You already have a shop" + link to settings |
| 401 | Redirect login |

**Must not send:** `vendorId`, `_id`.

---

## Submit flow (spec)

```
onSubmit:
  1. Client validate → field errors
  2. setSubmitting true
  3. createStore(formData)
  4. 201 → refetch(); navigate('/vendor')
  5. 409 → conflict message
  6. 400 → map errors; focus first invalid
  7. finally setSubmitting false
```

---

## Who sees this page

Storeless vendor → form. Vendor with store → redirect settings (stretch: GET me on mount). Customer → role guard blocks. Use `<label htmlFor>`, `aria-invalid`, focus first invalid on 400.

---

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| 201 but gate still redirects | Context not refetched | Call `refetch()` before navigate |
| 409 on first submit | Double-click or store exists | Disable button; GET me on mount |
| Form clears on 400 | State reset in catch | Keep values on error |
| CORS in console | Server origin | Fix CORS config |

---

## ✅ Do / ❌ Don't

**✅ Do:** Disable double-submit; handle 409; map server errors to fields.

**❌ Don't:** Navigate away on 400; send `vendorId`; store data only in React state.
