# 10.8 — Store settings page

## Why this step matters

Vendors change shop details after opening — seasonal description, corrected address. Settings pre-fills from `GET /stores/me` and saves via `PATCH`. This proves your client handles both read and update flows, not just first-time create.

---

**Where the project is now:** Vendor can create shop via OpenShop.

**After this step:** Vendor with a store can open **Settings**, see current fields pre-filled from `GET /stores/me`, save changes via `PATCH /stores/:id`, and see success confirmation.

---

## Page purpose

`StoreSettings` is the **ongoing edit** screen — same fields as open-shop, different verb and load path.

---

## Load flow (mount)

```
onMount:
  1. setLoading true
  2. getMyStore() → GET /stores/me
  3. 200 → populate form state from response; save storeId
  4. 404 → redirect /vendor/open-shop
  5. 401 → login
  6. setLoading false
```

Store `_id` from response for PATCH URL — never hardcode.

---

## Save flow

```
onSubmit:
  1. Client validate
  2. updateStore(storeId, changedFields) → PATCH
  3. 200 → show success; update local form from response
  4. 400 → field errors
  5. 403 → "You cannot edit this shop" — should not happen for own store
```

Optional: PATCH only **dirty** fields — full body also fine for MVP.

---

## Form parity with OpenShop

Reuse field components or shared `StoreFormFields` — reduces drift between POST and PATCH.

| Field | Pre-filled | Editable |
|---|---|---|
| name | ✓ | ✓ |
| description | ✓ | ✓ |
| address | ✓ | ✓ |
| logoUrl | ✓ | ✓ |

Read-only display of `createdAt` optional — stretch.

---

## UX states

| State | Display |
|---|---|
| loading | Skeleton or spinner |
| loaded | Form with values |
| saving | Disabled save button |
| saved | "Changes saved" message — auto-dismiss OK |

---

## Navigation

Link from layout nav: **Settings** → `/vendor/settings`.

Breadcrumb optional: Vendor → Settings.

---

## Error handling

| Status | Action |
|---|---|
| 404 on load | Redirect open-shop — store deleted? unlikely |
| PATCH 404 | Bad id — refetch GET me |
| Network | Retry message |

## ✅ Do / ❌ Don't

**✅ Do:** Refetch or use PATCH response to sync form — avoid stale UI.

**✅ Do:** Gate this route behind store existence — 404 load → redirect.

**❌ Don't:** Allow PATCH without storeId from GET me.

**❌ Don't:** Show settings form empty while loading — use spinner.
