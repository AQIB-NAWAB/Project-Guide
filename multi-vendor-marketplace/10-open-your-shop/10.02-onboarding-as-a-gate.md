# 10.2 — Onboarding as a gate

A vendor without a store is a **signed-up user**, not a **marketplace participant**. Letting them reach "Add product" or "Orders" before they open a shop creates empty screens, API 403/404 errors, and support confusion. Chapter 10 treats **having a store** as a **hard gate**.

---

## What the gate protects

| Route (examples) | Without gate | With gate |
|---|---|---|
| `/vendor/products` | Form posts fail — no storeId | Blocked until shop exists |
| `/vendor/dashboard` | Misleading empty state | Open-shop or dashboard |
| `/vendor/orders` | Later chapter — same rule | Redirect if storeless |

**Rule:** Any vendor **management** route except open-shop itself requires `GET /stores/me` → 200.

---

## Gate implementation options

| Approach | Pros | Cons |
|---|---|---|
| **Layout loader** | One check for all child routes | Must handle loading state |
| **Per-route guard** | Explicit | Duplicated fetch |
| **Context + cache** | Fetch once per session | Stale after create — must invalidate |

**Recommended:** `VendorLayout` runs store check on mount; cache store in state; refetch after POST create.

---

## API contract driving the gate

From Chapter 9:

```
GET /api/stores/me
  200 → vendor has store → allow child routes
  404 → no store → redirect /vendor/open-shop
  401 → logout / login
  403 → should not happen for vendor role — bug if seen
```

Client must **not** infer store existence from localStorage alone — always trust API on layout entry.

---

## Open-shop route exceptions

| Path | Store required? |
|---|---|
| `/vendor/open-shop` | **No** — this is the gate exit |
| `/login`, `/register` | No |
| `/vendor/settings` | **Yes** — only after store exists |

Avoid redirect loop: open-shop page must **not** redirect to itself when 404.

---

## After successful POST /stores

| Step | Action |
|---|---|
| 1 | POST returns 201 |
| 2 | Update client store state (or refetch GET me) |
| 3 | Navigate to `/vendor` or `/vendor/dashboard` |
| 4 | Gate passes — 200 on GET me |

Handle **409** if user double-submits — show message + redirect to settings (store already exists).

---

## Customer vs vendor routing

Gate applies to **vendor role only**. Customers never see open-shop — different layout and routes from Chapter 7 role redirect.

| Role | After login |
|---|---|
| customer | Shop/browse routes |
| vendor | Vendor layout + gate |

---

## UX copy (editorial)

Open-shop is not an error page — it is **welcome**:

- Headline: "Open your shop on [YourProduct]"  
- Subtext: local grocery positioning — one shop, your neighbourhood  
- Primary CTA: Create shop  

Settings page headline: "Shop details" — same fields, PATCH instead of POST.

---

## Edge cases

| Scenario | Behaviour |
|---|---|
| Seed vendor with store | Gate passes immediately |
| Vendor logs out mid-form | Draft lost — OK for MVP |
| Token expires on open-shop | 401 → login → return to open-shop if still 404 |
| PATCH settings with stale id | Use `_id` from GET me response |

---

## ✅ Do / ❌ Don't

**✅ Do:** Centralize gate in `VendorLayout` — one source of truth.

**✅ Do:** Show loading spinner while GET me in flight — avoid flash redirect.

**❌ Don't:** Let vendor reach product routes with 404 store — even "empty state" with broken API calls.

**❌ Don't:** Create store on login automatically — vendor must submit form (name matters).

---

## Security note

Gate is **UX enforcement**. Server still enforces vendor role and one-store rule — client gate does not replace API auth.
