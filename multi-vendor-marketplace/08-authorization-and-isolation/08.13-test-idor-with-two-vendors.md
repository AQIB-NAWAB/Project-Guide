# 8.13 — Test IDOR with two vendors

## Why this step matters

Security features without a negative test are wishful thinking. Proving Vendor B gets **403** on Vendor A's store is the Chapter 8 gate — and the template you'll repeat for products in 11.13. Document the curl commands; you'll rerun them when refactoring.

---

**Where the project is now:** PATCH /stores/:id with full middleware; happy path owner update works.

**After this step:** Documented proof that **Vendor B cannot PATCH Vendor A's store** — 403. This test becomes regression template for Ch 11 products.

---

## Prerequisites

From Compass or seed notes:

| Variable | Source |
|---|---|
| STORE_A_ID | Vendor A's store _id |
| STORE_B_ID | Vendor B's store _id |
| TOKEN_A | Login vendor-a@test.local |
| TOKEN_B | Login vendor-b@test.local |

---

## IDOR attack test (critical gate)

```bash
# Vendor B attempts to rename Vendor A's store
curl -s -o /dev/null -w "%{http_code}" -X PATCH \
  http://localhost:5000/api/stores/STORE_A_ID \
  -H "Authorization: Bearer $TOKEN_B" \
  -H "Content-Type: application/json" \
  -d '{"name":"Compromised By Vendor B"}'
```

**Expected: 403**

**Compass check:** Store A name **unchanged**.

---

## Control tests

| Test | Token | Store id | Expected |
|---|---|---|---|
| Owner update | A | STORE_A_ID | 200 |
| IDOR | B | STORE_A_ID | **403** |
| IDOR reverse | A | STORE_B_ID | **403** |
| No auth | — | STORE_A_ID | 401 |
| Customer | customer token | STORE_A_ID | 403 |

Run all five — document in learning log.

---

## Response body check

403 JSON example:

```json
{ "message": "Forbidden" }
```

Must **not** be 401 — B is authenticated.

Must **not** be 200 — IDOR succeeded — **blocker bug**.

## Troubleshooting

| Got 200 on IDOR | Ownership check missing or compares wrong field |
| Got 401 | Token B invalid — re-login |
| Got 404 | Wrong store id |
| Got 500 | assertStoreOwner uncaught — fix error handler |
