# 8.1 — Set the scene

**Where the project is now:** Users authenticate via JWT — `requireAuth` attaches `req.user.id` and `req.user.role`. Any authenticated vendor could still PATCH any store id if you wrote a naive route. **Logged in ≠ allowed.**

**Where this chapter goes:** **Authorization** — role checks (RBAC), object-level **ownership** checks, IDOR prevention, authorization matrix in `docs/AUTHORIZATION.md`, and proof that Vendor B gets **403** on Vendor A's store.

Concept-heavy with applied middleware — not essay-only.

## Bridge from last chapter

Chapter 7 gave vendors and customers **login screens**, token storage in memory, and protected routes in React. A vendor can reach a stub page in the browser — but the **server** still trusts any authenticated user with any store id if you write a naive route. Chapter 8 moves the guardrails to the **API**: who may call which endpoint, and who owns which store document. Frontend auth proved identity; this chapter proves **permission**.

## In plain English

| Term | Simple meaning |
|---|---|
| **Authentication (authn)** | Proving who you are — login, JWT, `requireAuth` |
| **Authorization (authz)** | Proving what you're allowed to do — role and ownership checks |
| **RBAC** | Role-Based Access Control — vendors vs customers vs admin |
| **Ownership** | This store belongs to Vendor A — only A may edit it |
| **IDOR** | Insecure Direct Object Reference — changing an id in the URL to touch someone else's data |
| **403 Forbidden** | Server knows who you are but refuses the action |
| **Middleware** | Code that runs before your route handler — can block the request |

---

## Think of it like a building pass

Chapter 6–7 gave everyone a **badge** (JWT) that says their name and role. That gets you through the front door. Authorization is the **room key**: a vendor badge does not open another vendor's stockroom. You check the badge *and* the room assignment on every sensitive door — not once at login and never again.

## Common beginner questions

**Q: We already have `requireAuth` — isn't that enough?**  
No. `requireAuth` only answers "who is this?" A logged-in Vendor B could still `PATCH /stores/vendor-a-id` unless you check ownership. Authentication and authorization are separate layers.

**Q: Why 403 instead of 404 when Vendor B hits Vendor A's store?**  
Both hide data from attackers. This course uses **403** with a clear message while learning — you can switch to generic 404 in production. The important part is **blocking**, not the exact status code.

**Q: Do I need authorization on GET requests too?**  
For **vendor-private** reads (my store, my products), yes. For **public** catalogue reads in Week 3, unauthenticated GET is intentional — different rules, documented in your matrix.

**Q: Is this only for vendors?**  
Vendors are the highest-risk actors in a marketplace (many owners, shared platform). Customers need scoping too — cart and orders in later chapters — but store/product isolation starts here.

**Q: Can I skip the matrix doc and just code middleware?**  
Don't. `docs/AUTHORIZATION.md` is your contract for every later chapter. Without it, each route invents its own rules and IDOR bugs slip in.

## The gap

Authentication answers: *Who are you?*

Authorization answers: *May you perform this action on this object?*

Multi-vendor marketplaces fail when only the first question is asked.
