# 8.6 — Prime your thinking

Before writing middleware, read OWASP guidance on object-level authorization — same concept as IDOR/BOLA. This is the last **learn** step before the build cluster. The goal is not to memorize OWASP jargon — it is to internalize one habit: **every route that touches an object by id must verify the caller owns it or has the right role**.

---

## Why read before you code

Marketplace bugs from missing authorization are not theoretical. A single `PATCH /stores/:id` without an ownership check lets Vendor B rename Vendor A's shop. OWASP calls this **BOLA** (Broken Object Level Authorization) — the API security industry's name for IDOR. Reading their cheat sheet now saves you from shipping a pattern you'll have to rip out in Chapter 11 when products get the same treatment.

---

## In plain English — status codes for this chapter

| Code | Meaning | When you use it |
|---|---|---|
| **401 Unauthorized** | "I don't know who you are" | Missing or invalid JWT — `requireAuth` |
| **403 Forbidden** | "I know you, but you can't do this" | Wrong role or wrong owner — `requireRole`, `assertStoreOwner` |
| **404 Not Found** | "That id doesn't exist (or I'm not telling you)" | Optional in production to hide existence — course may use 403 for clarity |

Beginners often return 401 for everything. That confuses client retry logic — the browser may try to refresh a token that is already valid. Use **403** when identity is known but permission is denied.

## Key takeaways to apply

1. Use unpredictable ids (ObjectId OK) — **but never rely on obscurity**  
2. Verify ownership on every mutating request  
3. Use generic error messages in production — course may use explicit 403 for learning  
4. Automated tests with two users — your 8.13 curl script

## Middleware behaviour — you implement

`requireRole(...roles)` must behave as follows (no copy-paste handler — write your own):

| Condition | HTTP | Response |
|---|---|---|
| No `req.user` (auth middleware did not run or failed) | `401` | `{ message: 'Unauthorized' }` or your app standard |
| `req.user.role` not in allowed roles | `403` | `{ message: 'Forbidden' }` or your app standard |
| Role allowed | call `next()` | — |

**Mount order:** `requireAuth` → `requireRole('vendor')` → route handler. Role middleware assumes identity is already on `req.user`.

**Hint:** Read role from JWT payload you set in Chapter 6 — same field name everywhere (`role`, not `userRole` in one file and `roles` in another).

## Error class optional

`ForbiddenError` / `UnauthorizedError` — map to 403/401 in central error handler if you have one from Ch 2.

## Think of it like a bouncer with a guest list

`requireAuth` checks you have a wristband. `requireRole` checks the wristband color matches the room (vendor vs customer). `assertStoreOwner` checks your name is on the lease for **this specific stall**. All three can apply on one route. Skipping any layer is how marketplaces leak data.
