# 8.12 — Apply to store routes

## Why this step matters

Middleware in isolation is untested theory. A minimal `PATCH /stores/:id` proves your stack runs in the correct order — `requireAuth` → `requireRole` → `assertStoreOwner` → handler. Chapter 9 extends this file; Chapter 8 proves the guards work.

---

**Where the project is now:** assertStoreOwner and requireRole exist; no store HTTP routes mounted.

**After this step:** Minimal `stores.routes.js` with **PATCH /api/stores/:id** — full middleware stack, stub handler updates allowed fields. Router mounted on app.

## PATCH /api/stores/:id

**Middleware stack:**

```
requireAuth
requireRole('vendor')
assertStoreOwnerFromParams   // or inline assertStoreOwner
updateStoreHandler
```

**Request body (minimal):**

| Field | Type |
|---|---|
| name | string optional |
| description | string optional |
| address | string optional |

Reject userId or storeId in body — ignore if sent.

**Success 200:**

```json
{
  "store": {
    "id": "...",
    "name": "Updated Name",
    "slug": "...",
    ...
  }
}
```

No sensitive userId exposed in response optional — document.

---

## Service layer

```
updateStore(store, payload) → saved store
```

Only mutates allowed fields — whitelist keys.

Handler calls assertStoreOwner before service or relies on middleware — **once**, not twice.

## Owner success test

```bash
# Vendor A token, Store A id
curl -s -X PATCH http://localhost:5000/api/stores/STORE_A_ID \
  -H "Authorization: Bearer $TOKEN_A" \
  -H "Content-Type: application/json" \
  -d '{"name":"Vendor A Updated Market"}'
```

Expect **200** — Compass shows new name.
