# 8.4 — Authentication vs authorization

Two words, two responsibilities. Conflating them causes either open APIs or 401 loops where 403 is correct.

---

## Authentication (authn)

**Proves identity** — who is calling.

| Mechanism | FreshMarket |
|---|---|
| JWT access token | requireAuth |
| Result | req.user = { id, role } |
| Failure | **401 Unauthorized** |

*"I am Vendor A"*

---

## Authorization (authz)

**Proves permission** — may this identity perform this action on this resource.

| Layer | Check | Example |
|---|---|---|
| Function-level (RBAC) | role | requireRole('vendor') |
| Object-level (ownership) | resource owner | assertStoreOwner |
| Failure | **403 Forbidden** |

*"Vendor A may edit Store A, not Store B"*

---

## 401 vs 403 — decision tree

```
Missing Bearer?
  → 401

Invalid/expired JWT?
  → 401

Valid JWT, customer hits vendor-only route?
  → 403

Valid JWT, vendor hits another vendor's store?
  → 403

Valid JWT, vendor hits own store, invalid body?
  → 400
```

Client refresh loop: **401 only** — not 403.

---

## Function-level vs object-level

**Function-level:** "Only vendors may call POST /products" — role check once at route.

**Object-level:** "This product belongs to caller's store" — load resource, compare ids.

Both required for PATCH /products/:id — Chapter 11.

---

## Anonymous vs authenticated public reads

GET /products public catalogue — no authn (Ch 14).

GET /cart — authn required, authz scopes to userId (Ch 16).

Matrix documents each route.

---

## ✅ Do / ❌ Don't

**✅ Do:** Return 403 for wrong role on authenticated request.

**✅ Do:** Document 401 vs 403 in AUTHORIZATION.md.

**❌ Don't:** Use 401 to mean "not allowed" — confuses client refresh.
