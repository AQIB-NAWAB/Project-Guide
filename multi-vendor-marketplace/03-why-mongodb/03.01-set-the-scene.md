## In plain English

| Term | Meaning |
|---|---|
| **Database** | Where the app stores data permanently — survives server restarts |
| **Document** | One JSON-like record in MongoDB — e.g. one product, one order |
| **Collection** | A named group of documents — like a table, but without fixed columns for every row |
| **Schema / model** | The rules for what fields a document must have — enforced in code (Chapter 5) |
| **Reference** | Storing another document's id and looking it up when needed — like a foreign key |
| **Embed** | Copying sub-objects inside a parent document — e.g. order line items inside an order |
| **ERD** | Entity-relationship diagram — a picture of collections and how they connect |
| **ODM (Mongoose)** | A library that defines schemas and talks to MongoDB from Node |

## Common beginner questions

**Q: Why not install MongoDB in Chapter 2?**  
A: Because you would connect before knowing **what shape** your data needs. Grocery products need units and perishability; orders need price snapshots. Designing first prevents copy-paste schemas that break at checkout.

**Q: Is this chapter just reading — does my repo change?**  
A: You **commit** `docs/erd-draft.md` and actor narratives in sub-chapters 3.8–3.9. That is real project progress — design artifacts live beside code in professional teams.

**Q: MongoDB was chosen for MERN — do I need to agree?**  
A: For **this repo**, yes — chapter paths assume Mongoose and documents. You must also articulate **when SQL would compete** (reporting, strict FK enforcement). Interviewers want trade-offs, not fanboyism.

**Q: What's the difference between `erd-draft.md` and code in Chapter 5?**  
A: Draft is your **plan**; Mongoose models are the **enforceable plan**. They should match. If they diverge, fix the draft or the model — do not leave both telling different stories.

**Q: I don't know SQL or MongoDB deeply. Can I still do this chapter?**  
A: Yes. Sub-chapters 3.4–3.6 explain both styles using **the same marketplace entities** side by side. You need curiosity, not prior DBA experience.

## Why every marketplace needs a deliberate database choice

Your app will store:

- **Users** with roles (vendor vs customer)  
- **Stores** owned by vendors  
- **Products** with grocery-specific fields (unit, weight, perishability, stock)  
- **Cart items** tied to customers and products  
- **Orders** split per vendor with **snapshotted prices**  

You could store that in PostgreSQL, MySQL, MongoDB, or something else. The choice affects:

- How you model **ownership** (vendor → store → product)  
- How you model **order lines** (embedded snapshots vs join tables)  
- How you enforce **multi-vendor checkout** (one cart → many orders)  
- How you query the **public catalogue** (filters, pagination, search, cache)

Picking a database because "MERN starts with M" is how tutorials teach. Picking a database because **this product's read/write patterns fit** is how engineers ship.
