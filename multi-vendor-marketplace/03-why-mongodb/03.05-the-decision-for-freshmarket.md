# 3.5 — The decision for FreshMarket

This sub-chapter is the **mandate** stated with pros, cons, and consequences. After reading it, you should be able to say: *"We use MongoDB with Mongoose for this marketplace because …, and we accept … as the trade-off."*

## Why this matters for absolute beginners

A mandate without reasons feels arbitrary. This page gives you **sentences for the viva**: document shape matches JSON APIs, order lines embed naturally, grocery attributes evolve without constant migrations — and you accept fewer database-enforced foreign keys and harder ad-hoc BI. When a interviewer asks "why not Postgres?", you answer from **this project's weights**, not from MongoDB marketing.

## Common beginner questions

**Q: Is MongoDB "mandated" because MERN starts with M?**  
A: No — read the pros table: embedded orders, catalogue + JSON, Mongoose teaching path. MERN coherence is one factor, not the whole argument.

**Q: What cons must I honestly accept?**  
A: App-layer enforcement of refs, reporting takes more work, multi-document checkout needs explicit design — say these aloud in 3.10 learning log.

**Q: Atlas or local — does that change the mandate?**  
A: No — only the connection string. Modelling decisions are the same.

**Q: Can I change my mind at Chapter 10?**  
A: Practically no — twenty chapters of examples assume MongoDB. Decide here; implement in 4–5.

## Pros for this specific project

**1. Document shape matches the API**

Express sends JSON. React consumes JSON. MongoDB stores BSON documents that map cleanly — less impedance mismatch than ORM mapping every row to a nested cart response.

**2. Embedded order lines**

An order is a natural aggregate: header (customer, store, status, timestamps) + array of lines with snapshot fields. One read fetches the full order for customer history — no join table for lines unless you want one.

**3. Grocery product evolution**

Core fields fixed (`unit`, `stockQty`, …). Optional nested `attributes: { organic: true }` on some products without ALTER TABLE migrations — discipline required, flexibility available.

**4. Mongoose ecosystem for teaching**

`populate()`, schema enums, middleware for hash-on-save, indexes declared in schema — all appear in later chapters with official docs and measurable exercises (N+1, text index).

**5. MERN portfolio coherence**

Employers scanning a MERN project expect MongoDB. Delivering that stack completely is a portfolio goal of the course.

---

## Cons you accept (say these in the viva)

**1. Fewer database-enforced constraints**

A bug can insert a product with invalid `storeId` if service layer skips validation. **Mitigation:** Mongoose refs, service checks, tests (Chapter 8 isolation tests). You own correctness — not the DB FK.

**2. Reporting queries**

"Revenue by category last month across all vendors" is often more SQL-fluent than aggregation for beginners. **Mitigation:** out of scope for core course; bar-raiser analytics optional.

**3. Transaction mental model**

Multi-document checkout must be designed explicitly. **Mitigation:** Chapter 18 spec; read MongoDB transactions if using Atlas.

**4. Schema drift without discipline**

Schemaless database doesn't mean schema-less team — documents without Mongoose schema become chaos. **Mitigation:** every collection gets a `*.model.js` in Chapter 5.

| Criterion | Weight for this course | MongoDB | PostgreSQL |
|---|---|---|---|

## Mongoose's role — not optional

Chapter 5 files:

```
server/src/modules/products/products.model.js
```

Pattern repeats per module. Mongoose provides:

```javascript
// Illustrative shape only — YOU implement in Ch 5
// product schema: storeId ref 'Store', unit enum [...], stockQty Number
```

Raw driver inserts are fine for scripts but **not** the course convention for models.
