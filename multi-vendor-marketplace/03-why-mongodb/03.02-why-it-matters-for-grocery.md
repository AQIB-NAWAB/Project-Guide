# 3.2 — Why it matters for grocery

A generic marketplace stores `title`, `price`, and `imageUrl`. A **grocery** marketplace stores **how much** of **what** — kilograms of rice, litres of milk, six eggs, a bunch of coriander marked **perishable**. Those fields are not cosmetic. They change validation, display, checkout copy, and the stories you tell in a viva about domain-driven design.

This sub-chapter ties the **FreshMarket domain** to **database modelling** before you choose SQL vs documents.

## Why this matters for absolute beginners

"Marketplace" tutorials often use generic products — a book, a T-shirt, one price. Real grocery breaks that template: price is **per unit**, stock may be decimal, and a receipt without "per kg" is meaningless. If your ERD only has `name` and `price`, you are building a different app than this course — and checkout in Chapter 18 will not make sense. Domain fields are how you prove the product is **FreshMarket**, not a renamed generic shop.

## Think of it like…

**Price tags at a real greengrocer.** Bananas are per bunch; rice is per kilogram; milk is per litre. The tag always says **price per unit**, not just a number. Your Product document is that tag — `unit`, `stockQty`, and `isPerishable` tell the story the customer expects at the shelf.

## Common beginner questions

**Q: Can I add grocery fields later in Chapter 5?**  
A: You could — but deciding them **now** in the ERD prevents a generic schema you must retrofit. Put units and categories on your draft Product shape today.

**Q: Does perishability change API logic in Week 1?**  
A: Not yet — it enables UI badges and portfolio narrative later. Still model it; seed data should mix perishable and non-perishable items.

**Q: Why mention SQL in a MongoDB chapter?**  
A: Honest engineers know trade-offs. One paragraph on when relational reporting wins is interview gold — not betrayal of the stack.

## Attributes that generic schemas miss

| Field | Example | Why it exists |
|---|---|---|
| `unit` | `kg`, `g`, `litre`, `piece` | Price is "per unit" — ₹120/kg ≠ ₹120/piece |
| `weightOrQuantity` | `0.5`, `1`, `2` | Default pack or increment — loose produce |
| `stockQty` | `45.5` | Stock is not always integer items |
| `isPerishable` | `true` / `false` | Drives fulfilment thinking; bar-raiser features |
| `category` | `Dairy`, `Grains`, `Produce` | Browse filters in Chapter 14 |

If your product document is only `{ name, price }`, you built a different product — not the course scope.

---

## Flexible attributes without schema explosions

Grocery inventory varies:

- Rice: brand, origin, weight bag size  
- Cheese: refrigerated, expiry-minded  
- Herbs: perishable, sold in bunches  

**Relational approach:** `products` table + `product_attributes` EAV table or many nullable columns — workable, verbose migrations for each new attribute class.

**Document approach:** core fields on every product (`unit`, `stockQty`, …) plus optional nested object for category-specific extras when you need them — e.g. `{ organic: true }` for produce without ALTER TABLE.

Neither is magic. Documents reduce friction when **shape varies by category** and you accept **application-level validation** instead of database-enforced column types on every edge field.

---

## Multi-vendor ownership still dominates

Grocery fields don't replace the harder rule: **every product belongs to one store belonging to one vendor**. Whether SQL or MongoDB, you model:

```
Vendor (User) → Store → Product
```

Customer cart items reference **product + store** so checkout can split by vendor (Chapter 18). Perishability doesn't relax isolation — Vendor A still never edits Vendor B's `stockQty`.

---

## Orders and grocery units — snapshot includes unit

When a customer buys **2 kg** of rice at **₹120/kg**, the order line snapshot must record:

- `nameAtPurchase` — "Basmati Rice"  
- `priceAtPurchase` — 120 (per unit at purchase time)  
- `unitAtPurchase` — `kg`  
- quantity — 2  

Without `unitAtPurchase`, history shows "₹120" with no idea if that was per kg or per bag. Grocery receipts are **unit-aware** by nature.

You'll embed these on order lines in MongoDB (Chapter 5 sketch, Chapter 18 implementation) — the decision is about **correct receipts**, not NoSQL fashion.

---

## Catalogue reads — hot path

Customers browse more than vendors write. A typical session:

- Many **reads** — list products, filter category, paginate  
- Fewer **writes** — vendor updates stock or price  

Both SQL and MongoDB handle this with indexes, pagination, and (Chapter 20) caching. The grocery domain adds **filter dimensions** (`category`, `storeId`, `isPerishable` for future features) — your index strategy in Chapter 20 must match fields you define now in the ERD draft.

---

## When grocery would push you toward SQL

Be honest — MongoDB is not universal:

| Scenario | Relational appeal |
|---|---|
| Strict inventory ledger with double-entry | Transactions across ledger tables |
| Complex reporting across vendors for tax | SQL analytics, JOINs, materialized views |
| Heavy constraints on unit enums at DB level | CHECK constraints on `unit` column |
| Existing team standard | Skill fit beats greenfield theory |

This course still uses MongoDB — but your learning log must name **one case** where SQL would compete. Interviewers respect trade-offs, not fanboyism.

## Domain vocabulary for your ERD

Use these names consistently in `docs/erd-draft.md`:

| Concept | Collection (MongoDB) | Notes |
|---|---|---|
| Login identity | `users` | role: vendor \| customer |
| Profile fields | `profiles` | linked to user |
| Shop | `stores` | one per vendor |
| Listing | `products` | belongs to store |
| Cart row | `cartitems` | customer + product + store + qty |
| Purchase | `orders` | one per vendor per checkout; embeds lines |

Exact naming (`cartitems` vs `cart_items`) is yours — stay consistent in Chapter 5.

## ✅ Do / ❌ Don't (design phase)

**✅ Do:** Put grocery fields on the Product shape in your ERD — not "we'll add unit later."

**✅ Do:** Note on Order embedded lines that snapshot includes **unit**, not just price.

**❌ Don't:** Model products as strings on the cart without `productId` — checkout needs references before snapshot.

**❌ Don't:** Skip `storeId` on cart items — multi-vendor split depends on it (Chapter 16).

---

## Key ideas

- Grocery needs **unit, decimal stock, perishability** — not generic SKUs only  
- **Ownership chain** unchanged: User → Store → Product  
- **Order snapshots** must capture unit and price per unit  
- Documents ease **optional attributes**; SQL eases some **reporting** — know both
