# 3.3 — What you'll decide

By the end of this chapter you will have made **three decisions** that Chapter 4–5 implement without reopening fundamentals:

1. **Database family:** MongoDB (document store) for this marketplace  
2. **ODM:** Mongoose for schemas, validation, and references in Node  
3. **Modelling strategy:** References for ownership chain; embedded arrays for order line snapshots; cart as referenced documents  

This sub-chapter states the decisions plainly and lists **evidence** you'll produce in 3.8–3.9.

## Why this matters for absolute beginners

Beginners often treat database choice as "install whatever the tutorial uses." Here you **document why** — MongoDB for variable product attributes and embedded order lines, Mongoose for schema discipline, references vs embeds for cart vs checkout. Chapter 4 will not re-debate this; it connects. Chapter 5 will not ask "SQL or Mongo?" — it implements. Your job in this chapter is to leave a **paper trail** (`erd-draft.md`) so future you does not wonder why order lines embed snapshot fields.

## Common beginner questions

**Q: Can I use PostgreSQL if I know it better?**  
A: Not in this repo without mentor agreement — every later chapter assumes Mongoose paths and document examples.

**Q: What is an ODM and why not raw MongoDB driver?**  
A: ODM (Mongoose) adds schemas, validation, and `populate()` — taught explicitly when fixing slow catalogue queries. Raw driver works in industry; this course standardizes on Mongoose.

**Q: What's the one deliverable that gates Chapter 4?**  
A: `docs/erd-draft.md` plus learning log proving you can explain embed vs reference — not a running database yet.

**Q: Why decide embed/reference before writing code?**  
A: Wrong choice at checkout is expensive — order history showing live prices instead of paid prices is a data bug, not a UI bug.

## Decision 1 — MongoDB for FreshMarket (mandated)

**The course requires MongoDB.** Not because MERN is a meme — because:

- Product documents carry **variable grocery attributes** without migration churn for every new optional field  
- Order documents naturally **embed line items** with snapshot fields at checkout  
- The mentee path matches **Mongoose populate** patterns taught in catalogue and order chapters  
- One document store aligns with **JSON APIs** end-to-end (React ↔ Express ↔ MongoDB)

You may personally prefer PostgreSQL for marketplaces — many production marketplaces use it well. **For this repo, use MongoDB** so chapter paths, query examples, and isolation patterns align.

---

## Decision 2 — Mongoose as the ODM

Raw MongoDB driver works. Mongoose adds:

- **Schemas** with types and enums (`unit: { type: String, enum: ['kg','g','litre','piece'] }`)  
- **References** with `populate()` — taught explicitly when fixing N+1 (Chapter 14)  
- **Middleware** hooks — useful for password hashing before save (Chapter 6)  
- **Validation** at application boundary — pairs with request validation later  

Alternatives like Prisma (with MongoDB provider) or native driver are valid in industry. **This course uses Mongoose** — Chapter 5 filenames assume `*.model.js` Mongoose schemas.

## Decision 3 — Embed vs reference (preview)

| Data | Strategy | Reason |
|---|---|---|
| User ↔ Profile | **Reference** (two collections) | Identity vs profile split — Chapter 6 |
| Store → User | **Reference** | Store belongs to vendor |
| Product → Store | **Reference** | Product belongs to store; store name not copied onto every product |
| CartItem → Product, Store, User | **Reference** | Live cart tracks current product; still has storeId for split |
| Order → Customer, Store | **Reference** | Order belongs to customer and vendor store |
| Order line items | **Embed** in Order | Snapshot fields frozen at purchase — Chapter 18 |
| Refresh tokens (later) | **Reference or embed in User** | Chapter 6 specifies — hashed token storage |

Full depth in **3.6**. Your ERD must show embed for order lines explicitly.

## When you would pick SQL instead (required honesty)

Write at least one paragraph in your learning log on **when relational wins**:

**Strong relational cases:**

- **ACID checkout across multiple tables** with serializable transactions as the primary correctness tool — MongoDB multi-document transactions exist but relational teams often default to SQL for financial adjacency  
- **Ad-hoc analytics** — revenue by category across vendors with complex JOINs and window functions  
- **Strict foreign keys enforced by database** — every ORM can leak; Postgres FK constraints are hard guardrails  
- **Team expertise** — shipping beats theoretical fit  

**This project's checkout** still needs atomic stock decrement and multi-order creation — you'll implement carefully in Chapter 18 with Mongoose; SQL would use `BEGIN TRANSACTION`. Both can be correct; know what you traded.
