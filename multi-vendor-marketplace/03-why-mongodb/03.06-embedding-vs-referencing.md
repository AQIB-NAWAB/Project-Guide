# 3.6 — Embedding vs referencing

MongoDB gives you two ways to relate data: **reference** another document by `_id`, or **embed** subdocuments inside a parent document. Wrong choice causes pain at checkout or bloated documents. This sub-chapter is the decision framework for FreshMarket — apply it in **3.8** on your ERD.

## Why this matters for absolute beginners

Embed vs reference is the **most common MongoDB beginner mistake** after "put everything in one document." Cart items should **point at** live products; order lines should **copy** prices at purchase. Mixing those up gives customers wrong receipts or bloated user documents. The four-question framework below is your checklist before you draw arrows on the ERD.

## Think of it like…

**A shopping list vs a filed receipt.** The list (cart) says "2 kg rice from Stall A" and links to today's price board. The receipt (order) prints the exact price you paid — even if rice costs more next week. Reference the board while shopping; embed the numbers on the receipt.

## Common beginner questions

**Q: Why put `storeId` on CartItem if Product already has `storeId`?**  
A: Checkout groups by vendor in one pass without loading every product document — small, intentional duplication for a hot path.

**Q: Can order lines be a separate collection instead of embedded?**  
A: Yes in industry — this course embeds for simpler order history reads and teaching clarity. Your ERD must show snapshot **field names** either way.

**Q: Will embedded order lines hit MongoDB's size limit?**  
A: Not for normal grocery orders — 16 MB is enormous for line items. Embedding thousands of products **inside a store document** is the real anti-pattern.

**Q: Profile — embed in User or separate collection?**  
A: Separate — course choice. Login stays lean; profile loads when needed.

---

## Definitions in plain language

**Reference:** Document A holds `storeId: ObjectId("...")` pointing to document B in another collection. Loading A doesn't automatically load B — you `populate()` or second query.

**Embed:** Document A holds an array or sub-object copied inline:

```json
{
  "_id": "...",
  "status": "pending",
  "orderItems": [
    {
      "productId": "...",
      "nameAtPurchase": "Basmati Rice",
      "priceAtPurchase": 120,
      "unitAtPurchase": "kg",
      "quantity": 2
    }
  ]
}
```

No separate `order_items` collection required for that shape.

---

## Decision framework

Ask four questions:

1. **Does this data belong to one parent and rarely change independently?** → Embed candidate  
2. **Does this data have its own lifecycle and many owners?** → Reference  
3. **Must historical records stay frozen when source changes?** → Embed snapshot on the historical record  
4. **Will unbounded growth blow document size?** → Reference or paginate embeds  

MongoDB document limit: **16 MB** — far above a grocery order line list for normal orders; embedding order lines is safe.

---

## FreshMarket mapping

| Entity | Strategy | Why |
|---|---|---|
| Profile → User | **Reference** | Profile updated independently; login doesn't load bio every time if split |
| Store → User (vendor) | **Reference** | Store is entity vendors manage |
| Product → Store | **Reference** | Many products per store; store name not duplicated |
| CartItem → User, Product, Store | **Reference** | Cart tracks **live** product; price read at checkout for snapshot |
| Order → Customer, Store | **Reference** | Order belongs to customer and vendor store |
| Order line items | **Embed** in Order | Snapshot at purchase; no sync with live product |
| Product image URL | **Field on product** | String URL to Cloudinary — not binary embed |

---

## Cart: reference live product

CartItem holds:

- `userId`  
- `productId` → current product document  
- `storeId` → denormalized for split convenience **or** derived from product — course requires explicit `storeId` on cart item for checkout grouping without extra lookups  
- `quantity`  

**Why not embed full product in cart?** Vendor changes price; cart usually shows **current** price until checkout. At checkout you **copy** snapshot onto order lines, not onto cart permanently.

**Why storeId on cart item?** Checkout groups by vendor without loading every product to find store at checkout time — small denormalization for **read path** with clear ownership of store at add-to-cart time.

---

## Order: embed snapshot lines

At checkout, for each cart line you **read** current product fields, then **write** embedded objects on the order:

| Field | Source at checkout |
|---|---|
| `nameAtPurchase` | product.name now |
| `priceAtPurchase` | product.price now |
| `unitAtPurchase` | product.unit now |
| `quantity` | cart quantity |
| `productId` | optional reference for analytics — keep id but display snapshot fields |

After checkout, vendor changes product price — embedded fields **unchanged**. Customer history reads embedded array only — no populate of live product for prices.

---

## Anti-pattern: only productId on embedded line

```json
"orderItems": [{ "productId": "...", "quantity": 2 }]
```

**Failure:** display history uses live product price — wrong receipt. Chapter 18 rejects this design. Your ERD must show snapshot field names.

---

## Anti-pattern: embed entire product catalog in store

```json
"stores": { "products": [ /* thousands */ ] }
```

**Failure:** 16MB risk; loading store loads entire catalog; concurrent product edits conflict. **Products live in own collection** with `storeId` ref.

---

## Profile: embed vs reference

**Embed profile in user document:**

- Pros: one read on login  
- Cons: user document bloat; profile fields loaded when unnecessary  

**Reference (course choice):**

- `users` lean; `profiles` linked by `userId`  
- Login auth checks user; profile loaded when rendering account settings  

Chapter 6 implements split — ERD shows two collections.

```mermaid
  USER ||--o{ CARTITEM : has
  PRODUCT ||--o{ CARTITEM : in
  STORE ||--o{ CARTITEM : grouped
  USER ||--o{ ORDER : places
  STORE ||--o{ ORDER : receives
  ORDER ||--|{ ORDERITEM : embeds
```

`ORDERITEM` embedded — not a separate collection in this design.

## Populate vs embed read cost

| Read pattern | Pattern |
|---|---|
| Catalogue page 20 products + store names | Products query + populate storeId (batched) |
| Single order for customer history | Single order document; read embedded lines |
| Vendor order list | Orders query + populate customerId for name |

Chapter 14 optimizes catalogue. Chapter 19 relies on snapshots for history simplicity.

---

## ✅ Do / ❌ Don't for ERD

**✅ Do:** Draw embedded box inside `orders` for line items with snapshot fields listed.

**✅ Do:** Draw refs as arrows with `ObjectId` label.

**❌ Don't:** Embed cart lines inside user document — cart is many items, changes often, separate collection cleaner.

**❌ Don't:** Reference-only order lines without snapshot fields.

---

## Key ideas

- **Reference** for entities with own lifecycle (product, store, cart line)  
- **Embed** for order line **snapshots**  
- **Cart** references live product; **checkout** freezes embed  
- **16MB limit** — not a concern for order lines; is for catalog-in-store anti-pattern
