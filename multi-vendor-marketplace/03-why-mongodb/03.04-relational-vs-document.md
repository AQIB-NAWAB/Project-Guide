# 3.4 — Relational vs document

You need both models in your head — not to pick both, but to explain **why this course picks documents** for FreshMarket while acknowledging relational strengths. This sub-chapter walks the **same marketplace entities** through SQL tables and MongoDB collections so the comparison is concrete, not abstract.

## Why this matters for absolute beginners

"MongoDB vs SQL" debates online are often tribal — logos and memes, not your app. Walking **the same vendor, store, product, cart, and order** through both shapes makes the trade-off tangible: foreign keys vs ObjectId refs, `order_items` table vs embedded array, migrations vs flexible documents. You are not choosing a religion; you are choosing a **fit for this grocery marketplace and this course's path**.

## Think of it like…

**Two filing systems for the same shop.** Relational: every fact on a separate card cross-referenced by ID numbers — strict, great for accountants. Document: an order folder contains the full receipt stapled inside — great for "show me what the customer paid that day." FreshMarket needs both styles in different places — references for live catalogue, embedded snapshots for history.

## Common beginner questions

**Q: Are MongoDB documents schemaless chaos?**  
A: The **database** allows flexible documents; **your team** does not. Chapter 5 enforces Mongoose schemas — discipline is on you, not the engine.

**Q: Does MongoDB mean no joins?**  
A: You still combine data — via `populate()` or aggregation instead of SQL JOIN. N+1 query traps exist in both worlds (Chapter 14).

**Q: If SQL is better for reports, why MongoDB here?**  
A: Teaching path, JSON API fit, embedded order aggregates, and grocery attribute flexibility — weighted in 3.5's scorecard. You must still name one SQL win case in your learning log.

**Q: What's the denormalisation trap in one sentence?**  
A: Copying store names onto every product feels fast; vendor rename makes every copy wrong — except **order snapshots**, where stale prices are correct on purpose.

## The same marketplace in relational form

Sketch mentally (or on paper) these tables:

```
users (id, email, password_hash, role, ...)
profiles (id, user_id FK, display_name, avatar_url, ...)
stores (id, vendor_user_id FK, name, address, ...)
products (id, store_id FK, name, price_cents, unit, stock_qty, ...)
cart_items (id, customer_user_id FK, product_id FK, quantity, ...)
orders (id, customer_user_id FK, store_id FK, status, total, ...)
order_items (id, order_id FK, product_id FK, quantity,
             price_at_purchase, name_at_purchase, unit_at_purchase, ...)
```

**Foreign keys** enforce: a product's `store_id` must exist; an order_item's `order_id` must exist.

**Checkout query** joins `order_items` → `orders` → `stores` → `users` for vendor dashboard. **Catalogue query** joins `products` → `stores` for shop name on card.

This is proven, normal, and used by many marketplaces at scale.

## The same marketplace in document form

Collections (simplified):

```
users { _id, email, passwordHash, role, ... }
profiles { _id, userId ref users, displayName, ... }
stores { _id, userId ref users, name, ... }
products { _id, storeId ref stores, name, price, unit, stockQty, ... }
cartitems { _id, userId ref users, productId ref products, storeId ref stores, quantity }
orders {
  _id,
  customerId ref users,
  storeId ref stores,
  status,
  orderItems: [
    { productId, nameAtPurchase, priceAtPurchase, unitAtPurchase, quantity }
  ]
}
```

**References** (`ObjectId` + `ref`) replace FKs — **enforcement is application code** (and optional Mongoose validators), not Postgres rejecting bad inserts.

**Order lines embedded** — no separate `order_items` collection required; snapshot fields live inside the order document.

## Side-by-side comparison

| Concern | Relational | Document (MongoDB) |
|---|---|---|
| **Ownership chain** | FK: product.store_id → stores.id | Ref: product.storeId → stores._id |
| **Schema changes** | Migrations (ALTER TABLE) | Flexible documents; discipline still required |
| **Order snapshot** | order_items columns duplicated at insert | Embedded subdocuments in order |
| **Cart** | cart_items table with FKs | cartitems collection with refs |
| **Catalogue list + store name** | JOIN products + stores | populate('storeId') or $lookup |
| **Multi-vendor checkout** | Transaction: insert N orders + lines | Transaction or careful ordering; multiple order docs |
| **Constraints** | DB-enforced FK, CHECK, UNIQUE | App + Mongoose schema; unique indexes possible |
| **Ad-hoc reporting** | SQL excels | Aggregation pipeline; often more code |
| **JSON API fit** | ORM maps rows → objects | Documents map naturally to JSON |

Neither column "wins" universally — **fit to team and access patterns** decides in industry. **Fit to this course's chapter path** decides here.

## The denormalisation trap (relational and document)

**Wrong move:** copy `vendorName` onto every `products` row/document because JOIN feels slow.

**Cost:** vendor renames shop → thousands of stale copies.

**Right move:** store name lives on `stores`; catalogue query joins or populates once per page (batched — Chapter 14).

**Exception — order snapshot:** copy `nameAtPurchase`, `priceAtPurchase`, `unitAtPurchase` onto order lines **on purpose** — divergence is correct because history must not update when product changes. Same logic in SQL columns or embedded MongoDB subdocs.

You'll teach this again at checkout (Chapter 18). Your ERD draft should show snapshot fields on embedded lines, **not** only `productId`.

## Transactions — both can checkout

Multi-vendor checkout must:

- Create multiple `orders`  
- Decrement stock on multiple `products`  
- Clear `cartitems`  
- Fail entirely if any stock insufficient  

**PostgreSQL:** `BEGIN … COMMIT` familiar pattern.

**MongoDB:** multi-document transactions supported on replica sets (Atlas qualifies). Chapter 18 may use transaction API or ordered steps with checks — either way, **correctness is the requirement**, not which syntax.

Don't choose MongoDB believing you avoid transaction thinking.

---

## N+1 — both worlds

**Relational ORM trap:** load 20 products; loop 20 queries for store names.

**Mongoose trap:** same — `await Store.findById` per product.

**Fix:** JOIN in SQL; `populate()` or single aggregation in MongoDB. Chapter 14 measures query count — database family doesn't save you from lazy loading.

## Read-heavy catalogue

| Technique | SQL | MongoDB |
|---|---|---|
| Pagination | LIMIT/OFFSET or keyset | limit/skip or cursor |
| Filter category | WHERE category = | find({ category }) |
| Text search | FULLTEXT index | text index (Chapter 20) |
| Cache | Redis in front of either | Redis in front of either |

Chapter 20 caches catalogue reads — **cache sits above** SQL or MongoDB equally.

---

## Honest "pick Postgres" scenarios for this app

- You already run analytics on SQL and want one database  
- Compliance requires row-level audit triggers in DB  
- You want `CHECK (unit IN ('kg','g',...))` enforced even if app bugs  

Honest "pick MongoDB" scenarios for this app:

- Document shape matches JSON APIs and embedded order lines  
- Product attribute evolution without migration for optional nested fields  
- Course path and mentee timebox align with Mongoose
