# 22.2 — Secrets in Git forever

Before deploy, re-run Chapter 4 discipline at production scale. **`git log -- .env` must be empty.** Production credentials pasted into README or learning log disqualify the gate.

Deploy amplifies every secret leak — a key in Git history can be scraped by bots within minutes of a public push. This sub-chapter is the **audit** before you touch a host env panel.

---

## Why this matters for first-time deployers

| Mistake | Real consequence |
|---|---|
| Commit `server/.env` "just once" | Key lives in Git history forever unless rotated |
| Paste `MONGODB_URI` in README | Anyone with repo access owns your database |
| Put `JWT_SECRET` in client `VITE_*` | Bundled into JavaScript — visible in browser DevTools |
| Screenshot host panel with values | Portfolio post leaks production creds |

**Environment variables on the host** are the production version of `.env` — same keys, different values, never committed.

---

## Production secret inventory

| Variable | Host | Never in |
|---|---|---|
| `MONGODB_URI` | API server env | Git, client bundle |
| `JWT_SECRET` / refresh pepper | API server env | Git |
| `CLOUDINARY_*` | API server env | Git |
| `REDIS_URL` | API server env **and worker** | Git |
| `VITE_API_URL` | **client build** env | Committed `.env` with prod URL ok in CI secret — not passwords |

---

## Pre-deploy verification

```bash
git log --all --full-history -- "**/.env"
git check-ignore -v server/.env client/.env
```

Rotate any secret ever committed — new Atlas password, new JWT secret, invalidate old tokens.

## `.env.example` completeness

Gate 22.8 requires **every** production variable documented with placeholder — teammates reproduce deploy without guessing.

`.env.example` is **committed**. `.env` is **not**. The example file is a map, not a vault.

## Host env panels

Render, Railway, Vercel, VPS `/etc/environment` or PM2 ecosystem — **same keys** as local, **different values**.

## In plain English

| Term | Simple meaning |
|---|---|
| **Secret** | Password, API key, connection string — anything that grants access if leaked |
| **`.env.example`** | Template showing **which** variables exist — placeholders only |
| **Rotate** | Change a leaked secret to a new value — old one stops working |

---

## Common beginner questions

**Q: I accidentally committed `.env` once. Am I fine if I delete it now?**  
No. Git history still has it. Rotate every secret that was in the file and consider `git filter-repo` only if mentor advises — rotation is usually enough.

**Q: Can I commit production `VITE_API_URL`?**  
The **URL** is usually public anyway (visible in browser Network tab). Never commit API **passwords** or MongoDB URIs in client env.

**Q: Worker needs the same secrets as API?**  
Yes — especially `REDIS_URL` and `MONGODB_URI` if processor loads orders/users. Chapter 22.8 expands the full list.

---

## ✅ Do / ❌ Don't

**✅ Do:** Use separate MongoDB user for production.

**✅ Do:** Restrict Atlas IP to deploy host or `0.0.0.0/0` with strong password — document tradeoff in learning log.

**❌ Don't:** Commit production `.env` "just once".

**❌ Don't:** Put `MONGODB_URI` in client `VITE_*`.
