## Bridge from last chapter

Chapters 2–21 built a complete app on **your laptop**:

| Works on `localhost` | Does not work for mentors / employers |
|---|---|
| `http://localhost:5173` shop | They cannot open your browser |
| `server/.env` with secrets | Files on your disk are not "deployed" |
| `npm run dev` + `npm run worker` in terminals | Processes die when you close the laptop |
| MongoDB Atlas dev cluster | May block unknown IPs |

Chapter 22 is the **last build chapter** — same app, different **home**. You copy the API to a host, build the client with a production API URL, run the worker as a second process, and prove smoke tests pass on **HTTPS**.

Nothing new to learn about grocery or JWT here. This is **operations** — the skill that turns "I built a course project" into "here is my live demo link."

## In plain English

| Term | Simple meaning |
|---|---|
| **Deploy** | Put your app on a computer that stays on 24/7 so anyone on the internet can use it |
| **Host / hosting** | The company or server that runs your app — Render, Railway, Vercel, a VPS, etc. |
| **Environment variables (env vars)** | Secret settings the host injects at runtime — database URL, JWT secret — **not** baked into Git |
| **Production** | The live version real users (or mentors) hit — as opposed to `localhost` development |
| **HTTPS** | Encrypted web — padlock in browser; required for secure cookies and professional demos |
| **Build** | Compile the React app into static files (`client/dist/`) with the production API URL frozen in |
| **PaaS (Platform as a Service)** | Host manages the server for you — Render, Railway, Vercel — faster for beginners |
| **VPS (Virtual Private Server)** | You rent a Linux machine and configure everything — more control, more setup |
| **Smoke test** | Quick check that critical paths work on production — health, login, checkout |
| **Process manager** | Keeps API and worker running after reboot — PM2, systemd, or PaaS auto-restart |

## Think of it like moving into a shopfront

Chapters 2–21 were **building in your garage** — tools everywhere, `.env` on the workbench, you are the only one who knows the address.

Deploy is **opening the shopfront on Main Street**:

- **Signage** — HTTPS URLs mentors can click  
- **Utilities** — production MongoDB, Redis, Cloudinary wired through the host's env panel, not a file in Git  
- **Staff schedule** — API and worker processes that restart when the power blips  
- **Grand opening test** — smoke script: walk in as customer, buy from two vendors, confirm vendor sees the order  

The product inside is the same. The **address and utilities** change.

## Localhost is not shipped

| Local truth | Production requirement |
|---|---|
| `http://localhost:5000` | `https://api.yourdomain.com` |
| `VITE_API_URL` in client `.env` | Build-time production API URL |
| MongoDB local/Atlas dev cluster | Production database user + IP allowlist |
| Redis optional local | Production Redis URL (cache + BullMQ queue) or fallback documented |
| Worker `npm run worker` in second terminal | Second process on host (PM2, Render worker service, etc.) |
| `.env` on your laptop | Host environment panel — never Git |

## Common beginner questions

**Q: What is "deploy" in one sentence?**  
Copy your server and built client to machines that stay online, configure secrets on the host, and get HTTPS URLs that work without your laptop.

**Q: Where do secrets go in production?**  
In the **host's environment variable panel** — Render "Environment", Railway "Variables", VPS `/etc/environment` or PM2 ecosystem file. Same **names** as `server/.env.example`, different **values**. Never commit production `.env` to Git.

**Q: What is `VITE_API_URL` and why does it matter at build time?**  
Vite **bakes** `import.meta.env.VITE_API_URL` into the JavaScript bundle when you run `npm run build`. Changing the API URL later requires **rebuild and redeploy** the client — it is not read at runtime from a file.

**Q: Do I deploy only the API?**  
No. Gate requires **three pieces**: HTTPS API, HTTPS client, **worker process**. Checkout notifications stop if only the API is deployed (Ch 21 jobs enqueue but nobody processes them).

**Q: Which host should I pick?**  
Beginners: API on Render/Railway + client on Vercel — fastest HTTPS. Comfortable with Linux: one VPS with PM2 + Caddy — stronger viva story. Pick one path in 22.4 and document it.

**Q: Will deploy break my local setup?**  
No. Keep `server/.env` and `client/.env` for localhost. Production uses **separate** env values on the host — often a different MongoDB database and a new `JWT_SECRET`.

**Q: What if smoke test fails on production but worked locally?**  
Usually: wrong `VITE_API_URL` in client build, CORS still pointing at `localhost`, missing env var on worker, or Atlas IP not allowing the host. Chapter 22.13 walks the fix matrix.
