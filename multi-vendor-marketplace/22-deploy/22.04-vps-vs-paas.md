# 22.4 — VPS vs PaaS

Pick **one deploy path** for this course. Document in README — not every option.

This decision affects how you set env vars, run the worker, and renew TLS. There is no universally "best" choice — only **best for your comfort level**.

---

## Options comparison

| Approach | Examples | Pros | Cons |
|---|---|---|---|
| PaaS API | Render, Railway, Fly.io | Fast TLS, env UI, git deploy | Cost, cold start on free tier |
| PaaS client | Vercel, Netlify | Easy static deploy, auto HTTPS | CORS must match production client URL |
| VPS | DigitalOcean, Hetzner, EC2 | Full control, strong viva story | You manage TLS, PM2, firewall |
| All-in-one VPS | Nginx + API + static + worker | One bill, one machine | More setup steps |

---

## Course recommendation

**Beginners:** API on Render/Railway + client on Vercel + **worker as second Render/Railway service** — fastest HTTPS, least Linux admin.

**Comfortable with Linux:** Single VPS + Caddy/Nginx + Let's Encrypt + PM2 for API and worker — best "I understand the full stack" viva story.

**Mandate:** HTTPS on both client and API. HTTP-only gate fails.

---

## Worker deployment per path

| Path | How worker runs |
|---|---|
| Render / Railway | Second service, same repo, start command `npm run worker` |
| VPS + PM2 | `pm2 start src/worker.js --name worker` + `pm2 save` |
| Fly.io | Second process in `fly.toml` or separate app |

**Never** assume checkout notifications work without verifying worker service in host dashboard.

---

## CORS production

Update `app.js` CORS `origin` to production client URL — not `localhost:5173` only.

Typical env key: `CLIENT_ORIGIN=https://app.yourdomain.com`

Browser login fails with cryptic CORS errors if this still points at local dev.

---

## MongoDB Atlas

Allow deploy host IP or use `0.0.0.0/0` with strong password for PaaS (IPs change). Document security tradeoff in learning log.

Worker and API **both** need `MONGODB_URI` if processor loads User/Order documents.

## In plain English

| Term | Simple meaning |
|---|---|
| **PaaS** | Platform runs the server for you — you push code, they handle machine |
| **VPS** | You rent a virtual Linux computer — you install Node, nginx, PM2 yourself |
| **Static host** | Serves pre-built HTML/JS/CSS files — no Node server for React in production |

---

## Think of it like housing

**PaaS** = furnished apartment — utilities included, less control, faster move-in.

**VPS** = empty flat — you bring furniture (nginx, PM2, certs), but you own the layout.

Both are valid homes for FreshMarket. Pick one and document it — do not half-deploy on three platforms.

## Common beginner questions

**Q: Can I deploy client and API on the same Vercel project?**  
Vercel is for **static client** + serverless functions — long-running Express + worker need Render/Railway/VPS. Typical split: Vercel client, Render API+worker.

**Q: Do I need a custom domain?**  
No for gate — `*.onrender.com` and `*.vercel.app` HTTPS URLs pass. Custom domain is portfolio polish.

**Q: What is cold start?**  
Free PaaS tiers sleep when idle — first request after sleep may take 30–60s. Retry smoke test once if health is slow.

## ✅ Do / ❌ Don't

**✅ Do:** Match `VITE_API_URL` to deployed API including `https://`.

**✅ Do:** Deploy worker as explicit second service — not an afterthought.

**❌ Don't:** Deploy API without setting all env vars on host.

**❌ Don't:** Leave CORS on localhost only.
