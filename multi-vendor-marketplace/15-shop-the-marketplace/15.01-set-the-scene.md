# 15.1 — Set the scene

**Where the project is now:** Chapter 14 delivered a public catalogue API — paginated products with store names and photos. Customers can authenticate (Chapters 6–7) but have **no storefront** in the browser.

**Where this chapter goes:** The **customer shop** — editorial product grid, filters, pagination, and product detail page. Data comes from `GET /products`; no catalogue logic duplicated in React.

This is a **build chapter**. Sub-chapters 15.4 onward add pages and components. Gate: customer browses products from **two vendors** with working filters in the UI.

## Bridge from last chapter

Chapter 14 gave you a **public catalogue API** — `GET /products` with filters, pagination, store names, and photos. You proved it with curl: paginated JSON from multiple vendors, query count not exploding (N+1 fixed).

What Chapter 14 did **not** give you: anything a normal person would call a **shop**. Humans do not browse groceries with terminal commands.

What you carry into Chapter 15:

| From Chapter 14 | What the UI needs it for |
|---|---|
| `GET /products?page=&limit=&category=&storeId=` | Shop grid + filters + pagination controls |
| `GET /products/:id` | Product detail page before add-to-cart |
| `store.name` on each item | "Sold by Green Valley" on every card |
| `imageUrl` | Product photos on cards and detail |
| Stable public DTO | React displays what API returns — no duplicate filter logic |

If Chapter 14 gate is incomplete — especially N+1 verification and two-store seed — **finish that first**. Chapter 15 will hammer `GET /products` on every filter change; a slow or wrong API becomes a slow or wrong shop.

## In plain English

| Term | Simple meaning |
|---|---|
| **Storefront / shop page** | The customer-facing product grid — `/shop` in the browser |
| **Product card** | One tile: photo, name, price, store name, link to detail |
| **Filter bar** | Controls that narrow the catalogue (category, store) and refetch from API |
| **Pagination UI** | Prev/Next or page numbers that change the `page` query param |

| **Add-to-cart stub** | Button on detail page that exists visually but does not POST yet — wired in Chapter 17 |

## Common beginner questions

**Q: Why not build the shop UI in the same chapter as the API?**  
Splitting keeps each gate testable. API proven with curl; UI proven in browser. When the grid is empty, you know whether to debug React or MongoDB.

**Q: Do I need to be logged in to browse?**  
Not for the Chapter 15 gate — public shop is fine. Login matters when you add to cart (Chapter 17).

**Q: Should I filter products in React after one big fetch?**  
No. The API paginates. Send `category` and `page` as query params and **refetch** — do not download the whole catalogue into browser memory.

**Q: The add-to-cart button does nothing — is that a bug?**  
Not yet. Chapter 15 spec includes a **stub** (toast or console log). Chapter 17 wires `POST /cart`.

**Q: Can I use a UI component library?**  
Yes if you want — the course specifies **behaviour** (card fields, filter refetch), not which CSS framework.

**Q: My images don't show — where do I look first?**  
Check `imageUrl` in the API response (Chapter 13), then the `<img src=` in your card component, then browser network tab for broken URLs.

## The gap between "API catalogue" and "marketplace window"

Your API can answer catalogue queries via curl. Humans need:

- Visual grid with photos and price per unit  
- Filters that feel instant  
- Product detail before add-to-cart  
- Navigation that says "shop" not "developer console"

Chapter 15 is the **frontend pair** to Chapter 14 — same pairing pattern as 07, 10, 12.

## Editorial design intent

FreshMarket uses a **clean editorial style** — name the product yourself, but the UI pattern is:

- Generous whitespace  
- Product card: photo, name, price/unit, store name  
- Filters as horizontal bar — not heavy e-commerce chrome  
- Pagination or load-more — not infinite scroll required for gate

Match vendor dashboard typography where sensible — one design system.
