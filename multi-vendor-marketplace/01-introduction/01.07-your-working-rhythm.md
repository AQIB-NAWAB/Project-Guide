# 1.7 — Your working rhythm

Four weeks is long enough for good habits to matter and long enough for bad ones to wreck you. This page is the last read before the Chapter 1 checklist. None of it is novel — every item is what working developers actually do. The course **returns to these habits in passing** as chapters get heavier, because a rule stated once on day one is a rule nobody follows on day eighteen.

Internalize them now. You'll hear them again when you deploy secrets, when you debug auth, when checkout fails at midnight.

---

## Commit every day

Your Git history is a journal of the build. Commit at the end of every work session — not "when it's perfect," but when a **sub-chapter step verifies**.

**Good commit moments:**

- Health route returns 200  
- Seed script runs clean  
- Register endpoint creates a hashed user  
- Vendor dashboard lists products  

**Bad commit moments:**

- "wip" with broken server and half a middleware  
- End of week with one commit containing 4,000 lines you can't explain  

**Commit message shape:** Short imperative — `add vendor product PATCH with ownership check`, not `updates`.

**Why it matters for the viva:** You'll be asked what changed between auth and isolation. `git log` and `git diff` between tags you name are legitimate answers — if the history exists.

---

## Secrets never enter Git — check before every push

This course uses `.env` files for database URLs, JWT secrets, Cloudinary keys, and Redis URLs. **One secret in `git log` is permanent** — rotation doesn't erase history; you'd rewrite history or burn the key.

**Before every push, five seconds:**

```bash
git status          # .env must not appear as staged or untracked you're about to force-add
git check-ignore -v server/.env client/.env   # confirm ignore rules hit
```

Before deploy (Chapter 22), run:

```bash
git log --all --full-history -- "**/.env"
```

Empty output = good. Anything else = stop and fix before continuing.

> **Warning:** Pasting a connection string into a Discord message, a screenshot, or a learning-log file counts as a leak too. Secrets live in `.env` on your machine and in the host's env panel — nowhere else.

## Read the error before the tutorial

When something breaks — and it will, often — your order of operations:

1. **Read the full error** — stack trace bottom to top; the first line is rarely the cause  
2. **Identify the layer** — MongoDB connection? JWT verify? CORS? React render?  
3. **Read the doc** for that layer's error code or message  
4. **Reduce** — reproduce with curl without the UI; one query in Compass without the route  
5. **Ask** with *"I expected X, got Y, I tried Z"*  

Skipping step 1 and opening ChatGPT with "my app doesn't work" produces generic answers. Step 1 alone fixes a surprising share of typos, wrong env var names, and forgot-to-import.

## One sub-chapter at a time — finish the gate

The course is designed for **finishable units**. A sub-chapter should take one sitting — thirty minutes to two hours — and end with verification.

**Do not:**

- Read three chapters ahead "to get context" while nothing in the repo works  
- Merge sub-chapters in your head ("I'll scaffold and implement auth in one go")  
- Tick checklist boxes you can't demo  

**Do:**

- Complete the verify step at the bottom of each build sub-chapter before the next file  
- Tick the chapter checklist only when every box is observable  
- Write learning-log answers before opening the next chapter folder  

---

## Keep the repo runnable

After every chapter gate, someone (you, a mentor, future you) should be able to:

- Start server and client with documented commands  
- Hit health or a feature endpoint without fixing broken imports first  

If a chapter leaves the server in a non-booting state "until the next chapter completes it," you've gone off-script. The build steps are sequenced so each step **adds** without **breaking boot** — except brief moments explicitly told to expect failure (e.g. stop MongoDB to test error handling).

When in doubt: `npm run dev` in both folders before you call a chapter done.

## Learning log — write in your own words

At the end of Chapter 1 (below), you write in `learning-log/01-introduction.md`. Every chapter after adds a file. Rules:

- **Complete sentences** — not bullet keywords  
- **Reference your project** — "In FreshMarket, vendor isolation works by…" not "Isolation is when…"  
- **Commit the file** with the chapter — it's evidence  

If ChatGPT wrote your answer and you can't expand it verbally in the viva, it failed its purpose.

---

## When to use AI — and when not to

**Good uses:**

- Explain an error message you read fully first  
- Clarify a concept from the chapter after you tried the reads  
- Generate test curl commands or sample JSON payloads  

**Risky uses:**

- Paste the chapter spec: "implement register handler for me"  
- Accept code you can't walk through line by line  
- Skip the learning log because AI already "explained" it  

The course allows AI. The viva does not allow **not understanding what you shipped**. If AI wrote it, you still own it — read every line before commit.

---

## Physical rhythm — four weeks is a marathon

**~60 hours/week** is full-time. If you're part-time, scale weeks, not gates — still clear every checklist, still hit every demo target before the next phase.

Practical habits:

- **Same workspace** when possible — context switching costs  
- **Two terminal tabs** — server and client — labeled  
- **Compass or Atlas** open during database chapters  
- **Notebook or digital doc** for ERD iterations before they land in `docs/erd.md`  

Take breaks. Sleep. The checkout bug that baffled you at 11pm is often obvious at 9am. The course doesn't reward heroic all-nighters; it rewards **correct, explainable increments**.

## Chapter 1 — Introduction checklist (gate)

Do **not** open Chapter 2 until every box is ticked and learning-log questions are written.

### Understanding

- [ ] I can explain in one paragraph what FreshMarket (or my named product) **is** and who uses it  
- [ ] I can name the **two actors** and one thing each must never do  
- [ ] I understand the **checklist gate** — no next chapter until every Done item is demoable  
- [ ] I know the course gives **specs, not solution code** — I write the handlers  
- [ ] I can state **three things out of scope** (e.g. payments, admin, reviews)  
- [ ] I know the **Week 1 demo target** — health in browser + seed in Compass  

### Setup

- [ ] Node 18+ and Git installed and verified  
- [ ] I have read the prerequisites table and addressed any **must have** gaps  
- [ ] I will create `learning-log/` in my project repo at the start of Chapter 2  

### Learning log — write now

Create your project repo if you haven't, or a temporary `learning-log/01-introduction.md` you will move into the repo in Chapter 2. Answer **in your own words**:

1. **Why is a multi-vendor marketplace a harder learning project than a single-user todo app?** Name at least two concrete technical problems the todo app doesn't have.

2. **What is the checklist gate, and what will you do if you can't tick a box?** (One paragraph — not "tick it anyway.")

3. **Describe the customer checkout flow in five steps** — from browse to vendor seeing an order. Use your product name.

4. **What will you demo at the end of Week 2?** One sentence.

5. **What lives in `.env`, and what must never live in Git?** List at least four key names you'll set later (not values).

Commit the learning log when your repo exists. Until then, keep the file locally ready to commit in Chapter 2.8.
