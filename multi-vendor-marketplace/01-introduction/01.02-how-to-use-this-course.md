# 1.2 — How to use this course

This is not a tutorial you copy-paste from start to finish. It's a **guided build**: each chapter teaches an idea in depth, compares the approaches a professional would weigh, and then walks you through implementing it on **your own project** — one folder, one file, one verification step at a time. You write the logic. The course tells you what it must do, what to avoid, and how to know it's working.

Read this section carefully. Everything else in the course assumes you understand how progress works here.

---

## If you're newer to full-stack — how to read this course

You do **not** need to already know MongoDB, JWT, Redis, or deployment. You **do** need patience to read the **Learn it** sections before the **Build it** steps — skipping straight to "create this file" is how tutorials feel easy and projects feel impossible.

**Read in this order within each chapter:**

1. **Set the scene** — where your repo stands; don't build yet  
2. **Why it matters / What you'll build** — understand the goal in plain language  
3. **Learn it** — read every paragraph; use the "In plain English" boxes when jargon appears  
4. **Prime your thinking** — skim mandatory links; return when you're stuck on a build step  
5. **Build it** — one sub-chapter at a time; verify before the next  
6. **Checklist** — honest ticks only; learning log in your own words  

**When you're stuck:**

| Symptom | What to do |
|---|---|
| "I don't understand the words" | Re-read **In plain English** in that sub-chapter; check Appendix in your learning log |
| "Build step failed" | Read **Troubleshooting** at bottom of that sub-chapter; compare your tree to "Where the project is now" |
| "Chapter feels like a jump" | Read the **Bridge from last chapter** section at the top of set-the-scene |
| "I want to skip ahead" | Don't — later chapters assume files from earlier gates exist |

**Two folders, two jobs:** `server/` is your API (Node + database). `client/` is what runs in the browser (React). They talk over HTTP. You'll hear this repeated until it's automatic — repetition is intentional for beginners.

## The shape: course → chapters → sub-chapters

The course is organized in three levels on disk. **Week groupings** (Week 1–4) are labels in the outline — not folders in the repo.

### Level 1 — The course

One project: **your multi-vendor grocery marketplace.** One repository with `server/` and `client/`. One learning log. One deployed URL at the end.

Course content lives in numbered chapter folders at the course root:

```
multi-vendor-marketplace/
  01-introduction/
  02-project-skeleton/
  03-why-mongodb/
  ...
```

### Level 2 — Chapters (week groupings)

A **chapter** is one **feature area** — a milestone you feel yourself reach:

- *Project skeleton*
- *Authentication*
- *Login & registration*
- *Checkout & orders*

A chapter is **not** "Backend week" or "Module 4." It's a named piece of the product that works when you're done.

This course has **21 build chapters** (Chapters 2–22) plus this introduction and a closing chapter. Each chapter is a **folder** containing ordered sub-chapter files.

Chapters are grouped into **four weeks** at full-time pace (~60 hours/week). Each week ends with a **demo target** — something runnable you can show:

| Week | Name | Demo target |
|---|---|---|
| Week 1 | Foundations | Both apps run; MongoDB seeded; health check visible in browser |
| Week 2 | Vendor experience | Vendor opens shop, lists products with photos |
| Week 3 | Buyer experience | Customer browses, carts, checks out, sees split orders |
| Week 4 | Polish & ship | Cache, search, async jobs after checkout, live HTTPS URL |

Week labels are **checkpoints**, not directory names. If Week 2 ends and no vendor can add a product in the UI, Week 3 is not where you fix that.

### Level 3 — Sub-chapters

Inside every chapter, you work through **sub-chapters** in order — small, finishable steps. A typical feature chapter has **12–20 sub-chapters** in this course (concept chapters run shorter); the guide's ideal arc is 20–25 steps following:

```
Open & motivate   →  what this feature is; why it matters; what you'll build
Learn it          →  options compared; decision mandated; concept taught in depth; reads
Build it          →  recap where repo stands → scaffold folders/files → implement → test
Wrap              →  recap; optional stretch; checklist gate
```

You're never handed "go build authentication" as one page. You're handed: create the auth module folder → create the routes file → implement register → implement login → issue a token → test with curl — each its own sub-chapter.

## How a chapter reads — the rhythm you'll repeat

Every chapter follows the same rhythm until you internalize it:

**1. Set the scene.** Where the project stands and what this chapter adds. You'll hear this as continuity: *"You now have working login; the token isn't checked on any route yet."*

**2. Learn it.** The concept, the weak approach and why it disappoints, the right approach and why it's worth the effort — taught in prose, not "go read this instead." Links appear **after** the explanation, to go deeper.

**3. Read.** Mandatory reads appear where the next build step genuinely depends on them — password hashing before you hash passwords, JWT refresh before you wire the client interceptor.

**4. Build it.** The largest block in every feature chapter. Concrete steps:

- Name the **exact folder and file** to create
- Say what the file **must do** — the contract, not the implementation
- Give **✅ do / ❌ don't** at the tricky points
- End each step with **how to verify** — the curl command, the screen state, the Compass screenshot

**5. Checklist.** Observable outcomes. Every box ticked = permission to open the next chapter.

Concept chapters (like "Why MongoDB?") run shorter on *Build it* but still include **applied work on your repo** — sketch the ERD, audit your schema, run a query against seeded data. They never end as read-only essays.

## The checklist is a gate — the most important rule

Every chapter ends in a **checklist** — a Definition of Done for build chapters, or Key takeaways for pure concept chapters.

**The checklist is a gate. You do not move to the next chapter until every box is ticked.**

That means:

- Each item must be something you can **see or demo** — not "I understand auth" but "a protected route returns 401 without a token"
- "Done" can't be faked by skimming — if you can't show it, you're not done
- Skipping the gate means the next chapter builds on something that isn't really there

Phrase it to yourself as: **"All boxes ticked? Open the next chapter. If not, that's today's work."**

This is what keeps a self-paced course honest. There's no instructor at the door — the checklist is the door.

## Keep a learning log (mandatory)

Create a folder in your repo from day one:

```
your-repo/
  learning-log/
    01-introduction.md
    02-project-skeleton.md
    ...
```

At the end of each chapter, you'll answer questions — about decisions, trade-offs, and what would break if you changed something. **Write your answers in the matching file and commit them.**

This is not optional decoration. The learning log is:

- **Evidence** you cleared the gate with understanding, not checkbox theatre
- **Rehearsal** for the final viva — you'll already have explained auth, isolation, and snapshots in your own words once
- **What a mentor or reviewer reads** when checking your progress

For this introduction chapter, you'll write answers in `learning-log/01-introduction.md` before moving to Chapter 2. If you can't answer in your own words, you haven't finished the chapter — go back.

## How you're judged

You're evaluated on **understanding**, not just a running demo.

**During the course:** checklists + learning log keep you honest chapter by chapter.

**At the end:** a **viva** — you walk through how your marketplace works and what would break it. Expect questions like:

- Why are identity and profile separate collections?
- What happens if checkout stores only `productId` on the order line?
- How does your API stop Vendor B from editing Vendor A's product?
- Why is the access token short-lived but the refresh token stored hashed?

**AI tools:** You may use Cursor, Copilot, or similar to help you build. The course doesn't ban AI. The viva checks that **you** can explain what you committed. If you can't explain your own code, that's the signal — not whether AI helped you write it.

## Specs, not solutions — what the course will and won't give you

**You will get:**

- API **contracts** — method, path, request/response shape
- **Folder trees** — where each file lives
- **Commands** to run — seed script, curl tests, deploy steps
- **Config sketches** — `.env` key names, `tsconfig` options annotated
- **Diagrams** — ERD, architecture, request flow
- **Measured symptoms** — query logs showing N+1, stale cache behaviour

**You will not get:**

- The route handler's body
- The auth middleware implementation
- The cart-splitting function
- Copy-paste React components for the whole dashboard

The test: could you build the feature knowing **what** to create and **what** it must do, without ever being shown the answer? If yes, the course did its job.

## Mandatory reads, topics to read, and hints

Three kinds of reading appear in the course:

**Mandatory reads** — you must read these before the next chapter; the build assumes them. Short list, strong sources, one-line reason each.

**Topics to read** — recommended depth for the curious; not blocking.

**Interesting to read** — true stories from real teams; never something you implement; search or real articles only.

**Hints** — nudges when a step is genuinely tricky. They reframe the question or name the concept to reach for. They never contain solution code. Open them only when stuck.

## When you're stuck — in order

Don't skip steps. This order solves most problems:

1. **Read the error message** — the full stack trace, not the first line
2. **Read the official docs** for the tool throwing the error
3. **Explain the problem out loud** — what you expected vs what happened
4. **Try a smaller test** — isolate one route, one query, one component
5. **Take a short break** — return with fresh eyes
6. **Ask for help** — lead with: *"I expected X, I got Y, I tried Z"*

That framing alone filters half of all "stuck" moments into typos, wrong env vars, or middleware order.
